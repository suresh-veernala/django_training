1. What is Django?

Django is a free, open-source web framework written in Python. It was released in 2005 by the Django Software Foundation and is named after jazz guitarist Django Reinhardt.

One-sentence definition
Django is a "batteries included" Python web framework that handles the hard parts of building a web backend — routing, database access, authentication, validation — so you can focus on your application logic.

The official motto is "The web framework for perfectionists with deadlines." It is used by Instagram, Pinterest, Disqus, Mozilla, and NASA, among others.

2. Why Django?

ProblemDjango's Answer

Compared to Flask (micro-framework), Django ships with everything wired up. Compared to FastAPI, Django has a richer ecosystem and a mature ORM — though FastAPI has the edge on raw async performance.

3. Key Features

ORM

Map Python classes to database tables. Query with Python, not SQL.

URL Routing

Clean, readable URL patterns declared in urls.py.

Migrations

Auto-generated schema diffs keep your DB in sync with your models.

Authentication

Users, passwords, sessions, and permissions out of the box.

Admin UI

A full CRUD dashboard auto-generated from your models.

Security

CSRF, XSS, and SQL-injection protection enabled by default.

Templating

Server-side HTML via Django Template Language (not our focus here).

DRF (add-on)

Django REST Framework adds serializers, ViewSets, and API browsing.

4. MVT Architecture

Django follows MVT — Model, View, Template. It is similar to MVC (Model-View-Controller) from other frameworks, but with different naming:

MVT LayerResponsibilityMVC Equivalent
Model Data shape & database access (ORM) Model
View Business logic — receives a request, returns a response Controller
Template HTML presentation layer (in REST APIs: replaced by JSON serializers) View
REST API note
In backend/API work, Views return JSON (via DRF Serializers), not HTML templates. The "T" in MVT is mostly relevant for server-side rendered frontends.

Step-by-step walkthrough

  1. HTTP Request — A client (browser, Postman, mobile app) sends a request, e.g. GET /api/books/.
  2. urls.py — Django inspects the URL against patterns defined in urls.py and dispatches to the matching view function or class.
  3. View — The view receives a request object. It decides what to do: validate input, call services, or query the database.
  4. Business Logic — Complex logic lives in separate service functions (not strictly enforced by Django, but a strong convention).
  5. Model / ORM — The view or service calls Book.objects.all() or similar. Django's ORM translates this to SQL and executes it.
  6. Database — The database returns rows. The ORM converts them back to Python objects.
  7. JSON Response — A DRF Serializer converts Python objects to JSON. The view returns a Response, which Django turns into an HTTP response.

6. Django ORM — A First Look

The ORM (Object-Relational Mapper) is one of Django's most powerful features. You define a Python class (a Model), and Django manages the SQL table for you.

Defining a Model

# models.py
from django.db import models

class Book(models.Model):
    title  = models.CharField(max_length=200)
    author = models.CharField(max_length=100)
    pages  = models.IntegerField()

    def __str__(self):
        return self.title

This creates a books_book table with columns id, title, author, pages. Django handles the primary key automatically.

How ORM queries look

# Django ORM query style (example)
Book.objects.filter(author='Tolkien')

This is enough for Module 1: ORM queries are Python expressions on model objects. We go deeper in Module 3.

Key insight
You never write SQL directly. The ORM generates it for you, keeps your code database-agnostic, and prevents SQL injection by design.

7. REST APIs in Django

Django ships with views that return HTML templates. For REST APIs, we add Django REST Framework (DRF) — the de-facto standard.

The three building blocks

Building blockFileJob
Serializer serializers.py Converts Model ↔ JSON. Also validates incoming data.
View / ViewSet views.py Handles the request, calls the serializer, returns a Response.
Router / URLs urls.py Maps URL patterns to views.

A minimal API endpoint

# serializers.py
from rest_framework import serializers
from .models import Book

class BookSerializer(serializers.ModelSerializer):
    class Meta:
        model  = Book
        fields = ['id', 'title', 'author', 'pages']
# views.py
from rest_framework.decorators import api_view
from rest_framework.response import Response
from .models import Book
from .serializers import BookSerializer

@api_view(['GET'])
def book_list(request):
    books = Book.objects.all()
    serializer = BookSerializer(books, many=True)
    return Response(serializer.data)
# urls.py
from django.urls import path
from . import views

urlpatterns = [
    path('api/books/', views.book_list),
]

A GET /api/books/ now returns a JSON array of all books. That is the full loop.

Key Takeaways

Remember these three things
  • MVT: Model handles data, View handles logic, Template (Serializer in REST) handles output.
  • Views return JSON in backend apps — not HTML. The "Template" is replaced by a DRF Serializer.
  • ORM abstracts SQL — write Python, get safe, portable database queries.
