What Is DSU?

Django Swagger Utils (DSU) is your org's internal library that sits on top of Django and makes building production-grade APIs faster. With DSU you can:

Scaffold instantly

One command generates the full Clean Architecture folder structure — interactors, storages, presenters, all pre-wired.

Build from OAS

Write your OpenAPI spec → run build → DSU generates serializers, URLs, and view stubs automatically.

Mock before implementing

Generated views return dummy data by default so the frontend can develop in parallel without waiting for you.

Clean request handling

DSU validates, parses, and injects request data — no manual json.loads or request.POST.get ever.

Where it comes from
DSU is the django-swagger-utils package in your requirements.txt. It is maintained internally. Everything in this lesson is specific to your org's stack.

Multiple Settings Files

A plain Django project has one settings.py. In your org's projects, there are several — each for a different purpose:

FilePurpose
local.pyLocal development — SQLite3, debug mode, relaxed auth
base_swagger_utils.pyAll DSU configuration — the file DSU reads
alpha.py / beta.pyStaging environment settings
base_pg_db.pyPostgres database connection settings

Tell Django which file to use by setting an environment variable in your terminal:

export DJANGO_SETTINGS_MODULE=<project_name>.settings.local
Common mistake
If you get "Could not import settings" errors, check this environment variable first. It must point to a valid dotted module path.

Creating and Registering an App

Step 1 — Scaffold the app

One command creates the entire Clean Architecture boilerplate:

python manage.py create_cleanapp <app_name> -p <project_name>

This generates every folder from the Backend Code Guide — interactors/, storages/, presenters/, api_specs/, constants/, and all the rest — with __init__.py files pre-created.

Step 2 — Register the app in two places

Unlike plain Django (where you only add to INSTALLED_APPS), DSU requires you to register in base_swagger_utils.py in two spots:

# base_swagger_utils.py

# 1. Add to APPS list
APPS = [
    "movies",
    "actors",
]

# 2. Add to SWAGGER_UTILS["APPS"]
SWAGGER_UTILS = {
    ...
    "APPS": {
        "movies": {},
        "actors": {},
    },
    "HOST": os.environ.get('APIGATEWAY_ENDPOINT', '127.0.0.1:8000'),
}
Most common setup mistake
Forgetting to add the app to SWAGGER_UTILS["APPS"] is the #1 cause of "endpoint not found" after running build. Always update both spots.

Building Views from the OpenAPI Spec

Once your OpenAPI spec is written, copy it to <app_name>/api_specs/api_spec.json, then run:

# Build one specific app
python manage.py build -a <app_name>

# Build all apps installed in the virtualenv
python manage.py build -at

DSU reads api_spec.json and generates a folder inside views/ for every operationId in the spec:

movies/views/
├── get_movie_details/one folder per operationId
│ ├── __init__.pyENV_MOCK / ENV_IMPL toggle lives here
│ ├── api_wrapper.py← YOU write your logic here
│ ├── get_movie_details.pyauto-generated — do not edit
│ ├── request_response_mocks.pysample request/response shapes
│ └── validator_class.pypre-request validations
├── create_movie/
└── delete_movie/

What each generated file does

api_wrapper.py

The only file you edit. Write your business logic here: read request data, call the interactor, return the response.

get_movie_details.py

Auto-generated by DSU. Parses the request and calls api_wrapper. Never edit this file — it will be overwritten on the next build.

request_response_mocks.py

Sample request body and expected response shapes. Useful for seeing what DSU understood from your spec. Read-only reference.

validator_class.py

Pre-execution validations (e.g. check if a resource exists before the main logic runs). Runs before api_wrapper.

__init__.py

Contains API_ENVIRONMENT — the MOCK/IMPL toggle (see below).

Control Flow — How a Request Travels

urls.py
→
view file
(get_movie_details.py)
→
validator_class.py
→
api_wrapper.py
  1. urls.py — the incoming URL is matched to a generated URL pattern and routed to the view file.
  2. View file (get_movie_details.py) — does basic parsing and forwards to api_wrapper.
  3. ValidatorClass — custom pre-execution checks run first.
  4. api_wrapper — receives fully validated, parsed data as kwargs. This is where your code lives.

MOCK vs IMPL — The API Environment Toggle

By default, build sets every new view to ENV_MOCK. This means the view returns dummy data — your api_wrapper is never called. This is intentional: the frontend can start building against the API shape immediately, while you work on the real implementation.

When you are ready to wire in real logic, open views/<operation_id>/__init__.py and change the one line:

# views/get_movie_details/__init__.py

API_ENVIRONMENT = "ENV_IMPL"  # was "ENV_MOCK"
First debugging step
If you've written logic in api_wrapper.py but it's never executing, check __init__.py. If it still says ENV_MOCK, DSU is returning dummy data and your wrapper is never called.

Mock views are valuable during active development. A newbie developer might accidentally leave a view in MOCK mode — checking this is always the first step when an endpoint "doesn't seem to be running your code."

Handling the Request in api_wrapper

When DSU calls your api_wrapper, all parsed and validated request data arrives as kwargs. The keys you'll use most:

KeyWhat it containsUse this?
kwargs['request_data'] Request body as a Python OrderedDict — parsed from JSON Yes — use this for the request body
kwargs['path_params'] URL path parameters (e.g. {"movie_id": "42"}) Yes — use this for path params
kwargs['request_query_params'] Query string parameters (e.g. ?include_cast=true) Yes — use this for query params
kwargs['user'] The authenticated Django User object When you need the user's ID or auth info
kwargs['user_dto'] The user as a UserDTO(user_id=...) Preferred when passing to interactors
kwargs['request_object'] The full serialized request object (Python class) Rarely — prefer request_data dict

A typical api_wrapper for creating a movie looks like this:

# views/create_movie/api_wrapper.py

from django_swagger_utils.drf_server.utils.decorator.interface_decorator \
    import validate_decorator
from .validator_class import ValidatorClass


@validate_decorator(validator_class=ValidatorClass)
def api_wrapper(*args, **kwargs):
    user         = kwargs['user']
    request_data = kwargs['request_data']

    title  = request_data['title']
    rating = request_data['rating']

    from movies.storages.movie_storage import MovieStorage
    from movies.presenters.movie_presenter import JsonPresenter
    from movies.interactors.create_movie_interactor import CreateMovieInteractor

    storage    = MovieStorage()
    presenter  = JsonPresenter()
    interactor = CreateMovieInteractor(storage=storage, presenter=presenter)

    return interactor.create_movie(
        user_id=user.id,
        title=title,
        rating=rating
    )
No raw request object
DSU has already validated and deserialized the request before calling api_wrapper. You never need json.loads(request.body) or request.POST.get(). Everything arrives clean as Python dicts.

Sending Responses

There are two ways to send a response. DSU supports both but the Presenter pattern (using HTTPResponseMixin) is the org standard.

Option A — Raw HttpResponse (acceptable for very simple cases)

from django.http import HttpResponse
from raven.utils import json

@validate_decorator(validator_class=ValidatorClass)
def api_wrapper(*args, **kwargs):
    ...
    data     = json.dumps({'movie_id': movie.id})
    response = HttpResponse(data, status=201)
    return response

Option B — Presenter with HTTPResponseMixin (org standard)

The Presenter builds the response. The api_wrapper just calls the interactor and returns whatever the presenter returns. See the next section for how to write a Presenter.

Writing Presenters — HTTPResponseMixin

Presenters inherit from both your interface and HTTPResponseMixin. The mixin gives you pre-built response builders for every HTTP status. Error responses follow a three-field contract: response, http_status_code, res_status.

# movies/presenters/movie_presenter.py

from django_swagger_utils.utils.http_response_mixin import HTTPResponseMixin
from movies.interactors.presenter_interfaces.presenter_interface import \
    MoviePresenterInterface


INVALID_MOVIE_CONTENT = (
    "Please send valid movie data",   # [0] human-readable message
    "INVALID_MOVIE_CONTENT"           # [1] machine-readable enum
)


class JsonPresenter(MoviePresenterInterface, HTTPResponseMixin):

    def get_response_for_create_movie(self, movie_id: int):
        response = {"movie_id": movie_id}
        return self.prepare_201_created_response(response)

    def get_response_for_invalid_content(self):
        failure_data = {
            "response":        INVALID_MOVIE_CONTENT[0],
            "http_status_code": 400,
            "res_status":      INVALID_MOVIE_CONTENT[1],
        }
        return self.prepare_400_bad_request_response(failure_data)

The error response the client receives looks like:

{
    "response":         "Please send valid movie data",
    "http_status_code": 400,
    "res_status":       "INVALID_MOVIE_CONTENT"
}

HTTPResponseMixin helpers

MethodStatusUse for
prepare_200_success_response(data)200Successful GET / PUT
prepare_201_created_response(data)201Successful POST (resource created)
prepare_400_bad_request_response(data)400Validation failures
prepare_404_not_found_response(data)404Resource doesn't exist
Error response convention
Every error response must have all three fields: response (human message), http_status_code (integer), res_status (SCREAMING_SNAKE_CASE enum). These must also appear in your api_spec.json — see the Backend Code Guide.

Models in Multiple Files

When an app has many models, keeping them all in one models.py leads to circular import errors and poor readability. DSU apps use a models/ folder instead:

movies/models/
├── __init__.pyre-exports all models for external access
├── movie.pyMovie model
├── actor.pyActor model
└── rating.pyRating model

The __init__.py re-exports everything so other files can import from movies.models as usual:

# movies/models/__init__.py

from .movie  import Movie
from .actor  import Actor
from .rating import Rating

__all__ = ["Movie", "Actor", "Rating"]
Why this works
Any file that does from movies.models import Movie still works exactly as before — it resolves via __init__.py. The split is invisible to the rest of the codebase.

What to Do Next

  1. Find a real view folder in any production app. Open __init__.py and confirm whether it is MOCK or IMPL. Open api_wrapper.py and trace how it reads from kwargs and calls an interactor.
  2. Find a real Presenter. Locate a class inheriting from HTTPResponseMixin. Find both a success method and an error method. Check the error method uses all three fields: response, http_status_code, res_status.
  3. Check the settings in base_swagger_utils.py. Find both the APPS list and the SWAGGER_UTILS["APPS"] dict. Confirm every installed app appears in both.
