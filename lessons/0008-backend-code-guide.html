Why this matters
Code review feedback at your org is almost always about one of three things: wrong file placement, wrong naming, or wrong import style. This lesson maps all three. Read it before writing your first PR.

Part 1 — The Complete App Structure

In Lesson 7 you saw the four core folders. Production apps have more. Here is the full boilerplate:

my_app/
├── app.pyDjango AppConfig
├── admin.pyDjango admin registrations
├── api_specs/OpenAPI spec (api_spec.json)
├── app_interfaces/ServiceInterface — what this app exposes to other apps
├── adapters/Calls TO other apps — DTOs + ServiceAdapter
│ ├── dtos.pyadapter-specific DTOs
│ └── service_adapter.pysingle ServiceAdapter class wiring all adapters
├── constants/config, enums, exception_messages
├── exceptions/custom_exceptions.py
├── entities/pure data classes (no DB, no HTTP)
├── interactors/business logic + interfaces
│ ├── post_interactor.pyone interactor per entity (or per use-case)
│ ├── mixins/validation_mixin.py — only if used in 3+ interactors
│ ├── presenter_interfaces/ABC + DTOs for presenters
│ └── storage_interfaces/ABC + DTOs for storages
├── models/Django ORM models
├── storages/concrete ORM implementations
├── presenters/concrete response builders
├── populate/data seeding utilities (wraps interactors)
├── signals/Django signals
├── utils/helpers used across 3+ files within this app
└── tests/mirrors app structure
├── common_fixtures/reusable mock objects for adapters, storages
├── factories/models.py · storage_dtos.py · presenter_dtos.py
├── interactors/one file per public method
├── storages/
├── presenters/
└── views/snapshot tests

New folders vs Lesson 7

FolderPurpose (one sentence)
adapters/All code that calls other apps. DTOs are converted here so foreign types never enter your interactors.
app_interfaces/The ServiceInterface class — what other apps can call on this app. The public API between Django apps.
constants/Three files only: config.py (settings/thresholds), enums.py (all enums), exception_messages.py (error strings).
exceptions/All custom exception classes for this app, in custom_exceptions.py.
populate/Data seeding scripts that wrap interactors. Handle sheet parsing, field validation, and DTO construction here.
tests/factories/Factory Boy model factories + DTO factories used in tests.
tests/common_fixtures/Reusable mock objects (e.g. auth_service_adapter_mocks.py) shared across multiple test files.

Part 2 — Rules Per Module

Interactors Key rules

  • Pure CRUD (no logic) for a single entity → one interactor class with multiple methods is fine.
  • When interactor A calls interactor B, wrap the call in a try/except inside A to handle B's exceptions.
  • Group arguments into a DTO when a method takes more than 3 arguments.
  • Pass sort/filter parameters to storage as structured DTOs (SortBy, FilterBy dataclasses — see below).
  • DTOs used only within an interactor are defined at the top of that interactor file.
  • Never create a new self.some_variable just to pass state between private methods — pass it as an argument.
  • Never import or use Django models directly inside an interactor — that belongs in Storage.

Sort and filter DTOs look like this — put them in interactors/storage_interfaces/dtos.py:

import enum
from dataclasses import dataclass

class SortByEnum(enum.Enum):
    TITLE  = "TITLE"
    RATING = "RATING"

class OrderByEnum(enum.Enum):
    ASC  = "ASC"
    DESC = "DESC"

@dataclass
class SortBy:
    sort_by: SortByEnum
    order:   OrderByEnum

class FilterByEnum(enum.Enum):
    GENRE  = "GENRE"
    DECADE = "DECADE"

@dataclass
class FilterBy:
    filter_by: FilterByEnum
    value:     str

Presenters Key rules

  • DTO → dict conversion logic lives exclusively in Presenters — never in the Interactor.
  • One presenter class (interface + implementation) per API endpoint.
  • The return type of every presenter interface method must be HttpResponse.
  • Exception messages for an API are defined as constants at the top of the presenter implementation class.
  • If a storage DTO can be reused in the presenter without modification, reuse it — no need to duplicate.
  • Never put nested-structure building logic in the Interactor — nested dicts belong in the Presenter.

Storages Key rules

  • When the storage interface grows large, split it into multiple interface classes after discussing with your RP.
  • When splitting interfaces, split the corresponding dtos.py files into matching structure too.
  • Never put business logic or conditional branching into a Storage method — only ORM operations.

Adapters Key rules (inter-app communication)

An Adapter is what you write when your app needs to call a different Django app. It lives in adapters/.

  • DTO conversion and exception handling for the external call both live in the adapter file.
  • Never pass a DTO from App B directly into App A's interactors — always convert it in App A's adapter first.
  • Never let exceptions from App B bubble up into App A — catch and re-raise as App A's own exceptions in the adapter.
Wrong — using other app's DTO directly
# In app A's interactor — WRONG
from auth_app.interfaces.dtos import UserDTO

def do_something(self, user_dto: UserDTO):
    # directly using another app's DTO
    ...
Correct — adapter converts first
# In app A's adapter — CORRECT
from auth_app.interfaces.service_interface import \
    AuthServiceInterface

class AuthServiceAdapter:
    def get_user(self, user_id: int) -> LocalUserDTO:
        external_dto = AuthServiceInterface().get_user(user_id)
        return LocalUserDTO(
            id=external_dto.id,
            name=external_dto.full_name
        )

Tests Key rules

  • Use class-based tests — group by endpoint or interactor method.
  • For an interactor with multiple public methods, write a separate test file per method.
  • Build test data with Factory Boy — never create model objects with raw Model.objects.create() in tests.
  • Don't repeat the class name in every test method name (e.g. test_create_post_creates_post inside TestCreatePost).
  • Don't define fixtures that are reused across multiple test classes in a local class — move them to common_fixtures/.

Part 3 — Python Conventions

Naming

What you're creatingNaming ruleExample
Model instancesuffix _objectpost_object = PostFactory()
DTO instancesuffix _dtopost_dto = PostDTO(...)
Factory classsuffix Factoryclass PostFactory
DTO classsuffix DTOclass MovieDetailsDTO
Exception classdescriptive, ends Exceptionclass InvalidMovieIdException
Enum valuesSCREAMING_SNAKE_CASEDRAMA = "DRAMA"

Imports

The rule is: import as late as possible, as absolute as possible.

  • Local imports (inside a method) are the default — they reduce circular dependency risk and make refactoring easier.
  • Move to a global import only when the same thing is imported in multiple methods of the same file.
  • Standard library packages (os, uuid, json, datetime) are always global, one per line.
  • Use absolute paths: from myapp.interactors.storage_interfaces.storage_interface import StorageInterface
  • Never use relative imports: from .storage_interfaces.storage_interface import StorageInterface ← forbidden.
  • Never import classes directly from third-party packages — import the module, then use module.ClassName.
Wrong
# relative import
from .storages.storage_interface import SI

# importing class directly from third-party
from django.db.models import QuerySet

# global import of something used once
from movies.storages.movie_storage import MovieStorage
class GetMovieInteractor:
    def execute(self):
        ...   # MovieStorage used only here
Correct
# absolute import
from movies.storages.storage_interface import SI

# import to module, access class via module
from django.db import models
qs: models.QuerySet = ...

# local import (used in one method only)
class GetMovieInteractor:
    def execute(self):
        from movies.storages.movie_storage \
            import MovieStorage
        ...

Typing

  • All method signatures must have type annotations on every argument and the return type — including private methods.
  • Instance variables declared in __init__ must be typed: self.storage: MovieStorageInterface
  • Don't annotate local variables inside methods — it adds noise without benefit.
class GetMovieDetailsInteractor:
    def __init__(
        self,
        storage: MovieStorageInterface,    # ← typed
        presenter: MoviePresenterInterface # ← typed
    ) -> None:
        self.storage: MovieStorageInterface   = storage    # ← typed
        self.presenter: MoviePresenterInterface = presenter # ← typed

    def get_movie_details(self, movie_id: int) -> HttpResponse:  # ← return typed
        movie_dto = self.storage.get_movie_by_id(movie_id=movie_id)
        # ↑ no type annotation for local variable — correct
        return self.presenter.get_response_for_movie_details(movie_dto=movie_dto)

Raising Errors

When an exception needs to carry data (e.g. the invalid ID), stringify it and include it in the message key:

# constants/exception_messages.py
INVALID_MOVIE_ID = "Invalid movie_id: {movie_id}"

# interactor
from movies.constants.exception_messages import INVALID_MOVIE_ID
from movies.exceptions.custom_exceptions import InvalidMovieIdException

raise InvalidMovieIdException(
    message=INVALID_MOVIE_ID.format(movie_id=movie_id)
)

Enums

  • All enums live in constants/enums.py. Group related enums together in the file.
  • When enums.py grows too large, split into multiple files in constants/ — group by entity.

API Spec — error responses

Every possible error your API can raise must appear in the OpenAPI spec as a named response with a res_status enum field:

{
  "responses": {
    "UpdateFeedbackFailureResponse": {
      "description": "Update Feedback Failure Response",
      "schema": {
        "type": "object",
        "properties": {
          "response":        { "type": "string" },
          "http_status_code":{ "type": "integer" },
          "res_status": {
            "type": "string",
            "enum": [
              "INVALID_FEEDBACK_ID",
              "DUPLICATE_ITEM_IDS",
              "PERMISSION_DENIED"
            ]
          }
        },
        "required": ["response", "http_status_code", "res_status"]
      }
    }
  }
}
Code review checkpoint
Missing error responses in the spec is a common PR rejection reason. Every exception your interactor can raise must map to a named response entry in api_spec.json.

Part 4 — When to Split & Refactor

These are the signals your org uses to decide when a file has grown too big:

File / classAction when it grows
Interactor with multiple public methodsSplit each public method into its own class (one file per use-case)
Storage interface classSplit into multiple interface classes — discuss with RP first
Mixin class > 200 linesApply SRP + OCP — split into focused mixin classes
custom_exceptions.pySplit by entity into separate modules inside exceptions/
constants/enums.pySplit by entity into multiple files inside constants/
The general rule
"If a file/class is growing, split it into a module (folder)." This is the single refactoring principle behind all the specifics above.

What to Do Next

  1. Open a production app and verify it matches the folder structure above. Find one folder or file that is named differently and understand why.
  2. Find an interactor. Check: does it have type annotations on all methods? Are its imports local or global? Does it use DTOs for sort/filter?
  3. Find an adapter in adapters/. Read how it converts an external DTO and re-raises exceptions. This is the pattern you'll copy.
  4. Continue to Lesson 9 — DSU where you'll see how all of this is scaffolded with DSU.
