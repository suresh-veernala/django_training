1. Creating a Model

A model is a Python class that subclasses models.Model. Each class becomes a database table; each attribute becomes a column.

# movies/models.py
from django.db import models

class Movie(models.Model):
    title        = models.CharField(max_length=200)
    release_year = models.IntegerField()
    rating       = models.FloatField(default=0.0)
    created_at   = models.DateTimeField(auto_now_add=True)

    def __str__(self):
        return self.title

Most fields accept null=True (allow NULL in DB), blank=True (allow empty in forms), unique=True, and default=….

Primary keys

Automatic by default
You did not define an id field above — Django adds an auto-incrementing integer primary key called id automatically. Override it by setting primary_key=True on any field.

2. Migrations

Migrations turn model changes into database changes. Two commands, always in this order:

  1. python manage.py makemigrations — Django compares your models to the last migration and writes a new migration file describing the diff. No DB change yet.
  2. python manage.py migrate — Django runs the unapplied migration files against the database, creating/altering tables.
Workflow rule of thumb
Change a model → makemigrations → migrate. Commit the generated migration files to git — they are part of your code. See the Migrations docs.

3. Relationships

Real data is connected. Django models three kinds of relationships:

ForeignKey — One-to-Many

"A movie has one director; a director has many movies." The ForeignKey goes on the "many" side:

class Director(models.Model):
    name = models.CharField(max_length=120)

class Movie(models.Model):
    title    = models.CharField(max_length=200)
    director = models.ForeignKey(
        Director,
        on_delete=models.CASCADE,   # delete movies if director deleted
        related_name='movies',      # director.movies.all()
    )

Access both directions:

movie.director              # the Director object (forward)
director.movies.all()       # all that director's movies (reverse)
on_delete is required
Common choices: CASCADE (delete children too), PROTECT (block deletion), SET_NULL (set FK to NULL — needs null=True).

ManyToManyField — Many-to-Many

"A movie has many actors; an actor appears in many movies." Django creates a hidden join table for you:

class Actor(models.Model):
    name = models.CharField(max_length=120)
    age  = models.IntegerField()

class Movie(models.Model):
    title  = models.CharField(max_length=200)
    actors = models.ManyToManyField(Actor, related_name='movies')
movie.actors.add(actor)     # link
movie.actors.all()          # actors in this movie
actor.movies.all()          # movies for this actor (reverse)

OneToOneField — One-to-One

"Each movie has exactly one detail page." Like a ForeignKey with unique=True:

class MovieDetail(models.Model):
    movie    = models.OneToOneField(Movie, on_delete=models.CASCADE)
    synopsis = models.TextField()
    budget   = models.BigIntegerField(null=True)
RelationshipFieldExample
One-to-ManyForeignKeyDirector → Movies
Many-to-ManyManyToManyFieldMovies ↔ Actors
One-to-OneOneToOneFieldMovie → MovieDetail

4. Core ORM Operations

These are the eight operations the training plan asks you to demonstrate. Run them in python manage.py shell. The SQL each generates is shown in comments.

Create

# One step
Movie.objects.create(title='Inception', release_year=2010, rating=8.8)
# SQL: INSERT INTO movies_movie (title, release_year, rating) VALUES ('Inception', 2010, 8.8)

# Two steps (build then save)
m = Movie(title='Tenet', release_year=2020)
m.save()
# SQL: INSERT INTO movies_movie (title, release_year, rating) VALUES ('Tenet', 2020, 0.0)

Read — get / all / filter

Movie.objects.all()
# SQL: SELECT * FROM movies_movie

Movie.objects.get(id=1)
# SQL: SELECT * FROM movies_movie WHERE id = 1 LIMIT 2   ← fetches 2 to detect duplicates

Movie.objects.filter(release_year=2010)
# SQL: SELECT * FROM movies_movie WHERE release_year = 2010

Movie.objects.filter(rating__gte=8.0)
# SQL: SELECT * FROM movies_movie WHERE rating >= 8.0
get() vs filter()
get() returns a single object and raises DoesNotExist / MultipleObjectsReturned. filter() always returns a QuerySet (possibly empty) — safe for "zero or more" results.

Update

# Bulk update — efficient, one SQL UPDATE
Movie.objects.filter(release_year=2010).update(rating=9.0)
# SQL: UPDATE movies_movie SET rating = 9.0 WHERE release_year = 2010

# Single object
m = Movie.objects.get(id=1)
m.rating = 9.5
m.save()
# SQL: UPDATE movies_movie SET rating = 9.5, ... WHERE id = 1

Delete

Movie.objects.filter(rating__lt=2.0).delete()
# SQL: DELETE FROM movies_movie WHERE rating < 2.0

m = Movie.objects.get(id=99)
m.delete()
# SQL: DELETE FROM movies_movie WHERE id = 99

order_by / count / exists

Movie.objects.order_by('-rating')
# SQL: SELECT * FROM movies_movie ORDER BY rating DESC

Movie.objects.count()
# SQL: SELECT COUNT(*) FROM movies_movie

Movie.objects.filter(release_year=2010).exists()
# SQL: SELECT 1 FROM movies_movie WHERE release_year = 2010 LIMIT 1

5. Advanced Querying

The patterns below are used constantly in real codebases. Each one shows the ORM call, the SQL it generates, and the reason it exists.

5.1 QuerySets are Lazy

qs = Movie.objects.filter(rating__gte=8)   # NO database hit yet — just builds a query object
list(qs)                                    # NOW it executes: SELECT * FROM movies_movie WHERE rating >= 8
qs[0]                                       # also executes (LIMIT 1)
qs.exists()                                 # SELECT 1 ... LIMIT 1  (cheapest existence check)

Laziness lets you chain and refine a query before paying the DB cost. Nothing hits the database until you iterate, slice, or call an evaluation method.

5.2 Field Lookups (field__lookup)

The double-underscore syntax translates directly to SQL WHERE clauses. You can also traverse relationships with it.

# Text search
Movie.objects.filter(title__icontains='dark')
# SQL: WHERE title LIKE '%dark%'   (case-insensitive)

Movie.objects.filter(title__startswith='The')
# SQL: WHERE title LIKE 'The%'

# Numeric comparisons
Movie.objects.filter(release_year__gte=2000)         # WHERE release_year >= 2000
Movie.objects.filter(release_year__range=(2000, 2010)) # WHERE release_year BETWEEN 2000 AND 2010
Movie.objects.filter(rating__in=[7.5, 8.0, 8.5])    # WHERE rating IN (7.5, 8.0, 8.5)

# NULL checks
Movie.objects.filter(budget__isnull=True)            # WHERE budget IS NULL

# FK traversal — Django does the JOIN automatically
Movie.objects.filter(director__name='Nolan')
# SQL: SELECT movies_movie.* FROM movies_movie
#      INNER JOIN movies_director ON movies_director.id = movies_movie.director_id
#      WHERE movies_director.name = 'Nolan'

5.3 Q Objects — OR, NOT, Complex Logic

Normal filter() kwargs are always AND-ed. Q lets you express OR, NOT, and nested conditions.

from django.db.models import Q

# OR
Movie.objects.filter(Q(release_year=2010) | Q(rating__gt=8))
# SQL: WHERE release_year = 2010 OR rating > 8

# AND NOT
Movie.objects.filter(Q(rating__gt=8) & ~Q(release_year=2010))
# SQL: WHERE rating > 8 AND NOT release_year = 2010

# Complex nesting: (year=2010 OR year=2020) AND rating > 7
Movie.objects.filter(
    (Q(release_year=2010) | Q(release_year=2020)),
    rating__gt=7,
)
# SQL: WHERE (release_year = 2010 OR release_year = 2020) AND rating > 7

# Build conditions dynamically from API params
conditions = Q()
if search_term:
    conditions &= Q(title__icontains=search_term)
if min_year:
    conditions &= Q(release_year__gte=min_year)
Movie.objects.filter(conditions)

5.4 Aggregations and Annotations

aggregate() returns a single dict for the whole QuerySet. annotate() adds a per-row computed value.

from django.db.models import Avg, Count, Max, Min, Sum

# aggregate() — one value for the whole table
Movie.objects.aggregate(Avg('rating'))
# SQL: SELECT AVG(rating) FROM movies_movie
# Returns: {'rating__avg': 7.9}

Movie.objects.aggregate(avg=Avg('rating'), total=Count('id'))
# Returns: {'avg': 7.9, 'total': 142}

# annotate() — one value per row
Director.objects.annotate(movie_count=Count('movies'))
# SQL: SELECT director.*, COUNT(movie.id) AS movie_count
#      FROM movies_director
#      LEFT OUTER JOIN movies_movie ON movies_movie.director_id = director.id
#      GROUP BY director.id

# Filter on an annotation → becomes a SQL HAVING clause
Director.objects.annotate(
    movie_count=Count('movies')
).filter(movie_count__gte=3).order_by('-movie_count')
# SQL: ... HAVING COUNT(movie.id) >= 3 ORDER BY movie_count DESC

# Conditional count: count only highly-rated movies per director
Director.objects.annotate(
    good_movies=Count('movies', filter=Q(movies__rating__gte=8))
)

5.5 The N+1 Problem — select_related & prefetch_related

The N+1 problem
Looping over movies and accessing movie.director.name fires one query per movie — 1 for the list + N for each director. These two methods fix it by fetching related data up front. 
# BAD — N+1 queries: 1 for movies, then 1 per movie for director
for movie in Movie.objects.all():
    print(movie.director.name)   # fires a new SELECT each time!

# GOOD — select_related: for ForeignKey / OneToOne → single SQL JOIN
for movie in Movie.objects.select_related('director'):
    print(movie.director.name)   # no extra query — director already loaded
# SQL: SELECT movies_movie.*, movies_director.*
#      FROM movies_movie
#      INNER JOIN movies_director ON movies_director.id = movies_movie.director_id

# GOOD — prefetch_related: for ManyToMany / reverse FK → 2 queries, joined in Python
for movie in Movie.objects.prefetch_related('actors'):
    print(movie.actors.all())    # no extra query per movie
# SQL (query 1): SELECT * FROM movies_movie
# SQL (query 2): SELECT * FROM movies_actor
#                INNER JOIN movies_movie_actors ON ...
#                WHERE movies_movie_actors.movie_id IN (1, 2, 3, ...)

# Chain both
Movie.objects.select_related('director').prefetch_related('actors')

5.6 Multiple filter() Calls — The JOIN Trap

For ManyToMany or reverse FK relationships, single vs. chained filter() produce different SQL. This is one of the most common sources of unexpected bugs. 

# SINGLE filter() → both conditions must match the SAME actor row
Movie.objects.filter(actors__name='Tom', actors__age__gt=30)
# SQL: SELECT DISTINCT movie.*
#      FROM movies_movie
#      JOIN movies_movie_actors ma ON ma.movie_id = movie.id
#      JOIN movies_actor a ON a.id = ma.actor_id
#      WHERE a.name = 'Tom' AND a.age > 30
# → Only movies with an actor who is BOTH named Tom AND over 30

# CHAINED filter() → each generates its own independent JOIN
Movie.objects.filter(actors__name='Tom').filter(actors__age__gt=30)
# SQL: SELECT DISTINCT movie.*
#      FROM movies_movie
#      JOIN ... a1 WHERE a1.name = 'Tom'
#      JOIN ... a2 WHERE a2.age > 30
# → Movies that have SOME actor named Tom AND SOME actor over 30 (can be different people)
Rule of thumb
Same related row must match? → single filter(). Each condition may match a different related row? → chain. For conditions on the model's own columns, both styles are identical. 
# These three are all equivalent (all conditions are on Movie's own columns)
Movie.objects.filter(release_year=2010, rating__gte=8.0)
Movie.objects.filter(release_year=2010).filter(rating__gte=8.0)
Movie.objects.filter(release_year=2010).filter(rating__gte=8.0).filter(director__name='Nolan')

5.7 Multi-field order_by() and NULL Handling

# Multiple sort keys — evaluated left to right
Movie.objects.order_by('-rating', 'title')
# SQL: ORDER BY rating DESC, title ASC

# Sort across a FK
Movie.objects.order_by('director__name', '-release_year')
# SQL: ORDER BY movies_director.name ASC, release_year DESC

# NULL handling — use F() for portable, explicit control
from django.db.models import F

Movie.objects.order_by(F('budget').asc(nulls_last=True))
# SQL: ORDER BY budget ASC NULLS LAST

Movie.objects.order_by(F('rating').desc(nulls_first=True))
# SQL: ORDER BY rating DESC NULLS FIRST

# Clear any default ordering from Model.Meta
Movie.objects.filter(rating__gte=8).order_by()
# SQL: SELECT * FROM movies_movie WHERE rating >= 8   (no ORDER BY)

5.8 F() Expressions — In-Database Column Operations

An F() object represents the value of a column at query time — without loading it into Python. Use it to compare columns, do arithmetic in the DB, and avoid race conditions.

from django.db.models import F

# Column-to-column comparison
Movie.objects.filter(views__gt=F('budget'))
# SQL: WHERE views > budget

# Atomic increment — race-condition safe
# BAD: read → Python increment → write (two requests can collide)
m = Movie.objects.get(id=1); m.views += 1; m.save()

# GOOD: single SQL UPDATE, increment happens atomically in the DB
Movie.objects.filter(id=1).update(views=F('views') + 1)
# SQL: UPDATE movies_movie SET views = views + 1 WHERE id = 1

# Computed annotation (virtual column per row)
from django.db.models import ExpressionWrapper, BigIntegerField
Movie.objects.annotate(
    profit=ExpressionWrapper(F('revenue') - F('budget'), output_field=BigIntegerField())
).order_by('-profit')
# SQL: SELECT *, (revenue - budget) AS profit FROM movies_movie ORDER BY profit DESC

5.9 values() and values_list() — Projections

Return dicts or tuples instead of model instances. Faster when you only need a few fields. Combining with annotate() gives you a GROUP BY query.

# values() → list of dicts
Movie.objects.filter(rating__gte=8).values('title', 'rating')
# SQL: SELECT title, rating FROM movies_movie WHERE rating >= 8
# Returns: [{'title': 'Inception', 'rating': 8.8}, ...]

# Follow a FK with double underscore
Movie.objects.values('title', 'director__name')
# SQL: SELECT movies_movie.title, movies_director.name FROM ... JOIN ...

# values_list() → list of tuples
Movie.objects.values_list('id', 'title')
# Returns: [(1, 'Inception'), (2, 'Tenet'), ...]

# flat=True → plain list of one field (great for .filter(id__in=...))
ids = Movie.objects.filter(rating__gte=8).values_list('id', flat=True)
# Returns: [1, 3, 7, ...]

# GROUP BY: values() before annotate()
from django.db.models import Count, Avg

Movie.objects.values('release_year').annotate(count=Count('id')).order_by('release_year')
# SQL: SELECT release_year, COUNT(id) AS count
#      FROM movies_movie GROUP BY release_year ORDER BY release_year ASC

Movie.objects.values('director__name').annotate(avg_rating=Avg('rating')).order_by('-avg_rating')
# SQL: SELECT movies_director.name, AVG(rating) AS avg_rating
#      FROM movies_movie JOIN ... GROUP BY movies_director.name ORDER BY avg_rating DESC
values() before annotate() = GROUP BY
The fields in values() become the GROUP BY columns. Always call annotate() after values() to aggregate per group.

5.10 only() and defer() — Load Fewer Columns

# only() — SELECT just these columns (id always included)
Movie.objects.only('title', 'rating')
# SQL: SELECT id, title, rating FROM movies_movie
# Accessing m.release_year on a result triggers a NEW query — avoid it

# defer() — SELECT everything EXCEPT these columns
Movie.objects.defer('synopsis')
# SQL: SELECT id, title, release_year, rating, ... FROM movies_movie
#      (synopsis omitted; accessing it later fires one extra query per object)

# Combine with filter
Movie.objects.filter(rating__gte=8).only('title', 'rating').order_by('-rating')
When to use only() / defer()
Profile first. Use them only when a large column (TextField, JSONField, binary) is measurably slowing queries. Accessing a deferred field fires a separate SELECT per object — which makes things worse if you access it in a loop.

5.11 distinct() — Remove Duplicate Rows

# M2M filters return one row per matching child — distinct() collapses them
Movie.objects.filter(actors__name__startswith='Tom').distinct()
# SQL: SELECT DISTINCT movies_movie.* FROM movies_movie
#      JOIN movies_movie_actors ma ON ...
#      JOIN movies_actor a ON ...
#      WHERE a.name LIKE 'Tom%'

Director.objects.filter(movies__rating__gt=8).distinct()
# SQL: SELECT DISTINCT movies_director.* FROM movies_director
#      JOIN movies_movie m ON m.director_id = movies_director.id
#      WHERE m.rating > 8

5.12 get_or_create() and update_or_create()

Atomic fetch-or-insert and upsert operations. They avoid the check-then-create race condition that manual code would have.

# get_or_create() → returns (object, was_created: bool)
movie, created = Movie.objects.get_or_create(
    title='Inception',           # lookup fields — used in the WHERE
    defaults={                   # set ONLY on creation, not in the WHERE
        'release_year': 2010,
        'rating': 8.8,
    }
)
# SQL (if found):   SELECT * FROM movies_movie WHERE title = 'Inception' LIMIT 1
# SQL (if missing): INSERT INTO movies_movie (title, release_year, rating) VALUES (...)

if created:
    print('New movie added')
else:
    print('Found existing:', movie)
lookup vs defaults
Fields outside defaults are the lookup key (the WHERE). Fields inside defaults are applied only on creation. Never put a search field inside defaults. 
# update_or_create() — upsert: insert or update in one call
movie, created = Movie.objects.update_or_create(
    title='Inception',              # lookup
    defaults={'rating': 9.0},       # applied on BOTH create and update
)
# SQL (if found):   UPDATE movies_movie SET rating = 9.0 WHERE title = 'Inception'
# SQL (if missing): INSERT INTO movies_movie (title, rating) VALUES ('Inception', 9.0)

5.13 bulk_create() and bulk_update()

Insert or update many rows in a single SQL statement instead of N separate ones.

# bulk_create() — one INSERT for all rows
movies = [
    Movie(title='Oppenheimer', release_year=2023, rating=8.3),
    Movie(title='Barbie',      release_year=2023, rating=6.9),
    Movie(title='Dune Part 2', release_year=2024, rating=8.5),
]
Movie.objects.bulk_create(movies)
# SQL: INSERT INTO movies_movie (title, release_year, rating)
#      VALUES ('Oppenheimer', 2023, 8.3), ('Barbie', 2023, 6.9), ('Dune Part 2', 2024, 8.5)

Movie.objects.bulk_create(movies, batch_size=500)          # split into batches
Movie.objects.bulk_create(movies, ignore_conflicts=True)   # skip duplicate rows silently
bulk_create() skips signals
pre_save / post_save signals are not fired. Auto-increment PKs are set on most databases but not on older SQLite (< 3.35). 
# bulk_update() — one UPDATE for all rows; name the fields you changed
movies = list(Movie.objects.filter(release_year=2023))
for m in movies:
    m.rating += 0.1

Movie.objects.bulk_update(movies, fields=['rating'])
# SQL: UPDATE movies_movie SET rating = CASE WHEN id=1 THEN 8.4 WHEN id=2 THEN 7.0 ... END
#      WHERE id IN (1, 2, ...)

Movie.objects.bulk_update(movies, fields=['rating', 'views'], batch_size=200)

5.14 Subquery, OuterRef, and Exists

Embed one QuerySet inside another. OuterRef links the inner query to the current outer row — a correlated subquery.

from django.db.models import OuterRef, Subquery, Exists

# Annotate each Director with their most recent movie's title
latest_movie = Movie.objects.filter(
    director=OuterRef('pk')          # OuterRef('pk') = Director.id in the outer query
).order_by('-release_year').values('title')[:1]

directors = Director.objects.annotate(latest_title=Subquery(latest_movie))
# SQL: SELECT director.*,
#        (SELECT title FROM movies_movie
#         WHERE director_id = director.id
#         ORDER BY release_year DESC LIMIT 1) AS latest_title
#      FROM movies_director

# Filter using a subquery result
high_rated = Movie.objects.filter(
    director=OuterRef('pk')
).order_by('-release_year').values('rating')[:1]

Director.objects.annotate(
    latest_rating=Subquery(high_rated)
).filter(latest_rating__gt=8)

# Exists — efficient boolean membership test (no data fetched)
has_hit = Movie.objects.filter(director=OuterRef('pk'), rating__gt=9)
Director.objects.filter(Exists(has_hit))
# SQL: SELECT * FROM movies_director
#      WHERE EXISTS (SELECT 1 FROM movies_movie
#                    WHERE director_id = director.id AND rating > 9)

6. Seeing the SQL — str(qs.query) and EXPLAIN

The ORM generates SQL for you, but you should be able to see and understand what it produces. There are three tools for this, each suited to a different moment.

6.1 str(queryset.query) — quickest inspection

Print the raw SQL of any unevaluated QuerySet directly in the shell. No configuration needed.

qs = Movie.objects.filter(rating__gte=8).order_by('-rating').select_related('director')
print(str(qs.query))

# Output (formatted for readability):
# SELECT "movies_movie"."id", "movies_movie"."title", "movies_movie"."rating",
#        "movies_director"."id", "movies_director"."name"
# FROM "movies_movie"
# INNER JOIN "movies_director" ON ("movies_movie"."director_id" = "movies_director"."id")
# WHERE "movies_movie"."rating" >= 8.0
# ORDER BY "movies_movie"."rating" DESC
Limitation of str(qs.query)
The output may not be 100% valid SQL — Django interpolates parameters in a simplified way. Use connection.queries (below) for the exact executed statement.

6.2 connection.queries — see every query that ran

Django logs every SQL statement executed in the current thread when DEBUG = True in settings.py.

from django.db import connection, reset_queries

reset_queries()   # clear the log

movies = list(Movie.objects.select_related('director').filter(rating__gte=8))

# Inspect what was executed
for q in connection.queries:
    print(q['sql'])
    print('time:', q['time'], 'sec')

# Output:
# SELECT "movies_movie".*, "movies_director".* FROM "movies_movie"
#   INNER JOIN "movies_director" ON (...)
#   WHERE "movies_movie"."rating" >= 8.0
# time: 0.003 sec
Requires DEBUG = True
connection.queries only accumulates entries when settings.DEBUG = True. It's always available in the development shell. Never enable DEBUG in production.

6.3 EXPLAIN — understand how the database executes the query

EXPLAIN is a SQL command that shows the database's execution plan: which indexes it uses, how it joins tables, and the estimated cost. Use it when a query is slow and you want to know why. 

from django.db import connection

# Build your ORM query
qs = Movie.objects.filter(rating__gte=8).order_by('-rating').select_related('director')

# Get the raw SQL and parameters
sql, params = qs.query.sql_with_params()

# Run EXPLAIN against the database
with connection.cursor() as cursor:
    cursor.execute('EXPLAIN ' + sql, params)
    rows = cursor.fetchall()
    for row in rows:
        print(row)

# PostgreSQL output (example):
# ('Seq Scan on movies_movie  (cost=0.00..12.50 rows=50 width=120)',)
# ('  Filter: (rating >= 8.0)',)
# → "Seq Scan" means a full table scan — add an index on `rating` to fix this

# SQLite output (example):
# (5, 0, 0, 'SCAN TABLE movies_movie')
# (5, 0, 0, 'USE TEMP B-TREE FOR ORDER BY')

6.4 EXPLAIN ANALYZE — run the query and measure actual time

EXPLAIN ANALYZE actually executes the query and reports real timing alongside the plan. Use on PostgreSQL; SQLite does not support it.

with connection.cursor() as cursor:
    cursor.execute('EXPLAIN ANALYZE ' + sql, params)
    for row in cursor.fetchall():
        print(row[0])

# PostgreSQL output:
# Seq Scan on movies_movie  (cost=0.00..12.50 rows=50) (actual time=0.025..0.182 rows=48)
#   Filter: (rating >= 8.0)
#   Rows Removed by Filter: 94
# Planning Time: 0.3 ms
# Execution Time: 0.2 ms

Reading the EXPLAIN output

TermMeaningWhen it's a problem
Seq ScanFull table scan — reads every rowOn large tables; add an index
Index ScanUses an index to find rowsUsually good
Index Only ScanData fetched entirely from indexBest — no table access
Hash JoinJoins two sets by hashing oneNormal for larger joins
Nested LoopFor each row in A, scan BSlow without an index on B
cost=X..YEstimated startup..total costHigh Y = slow query
rows=NEstimated row countWrong estimate → bad plan
Workflow: debugging a slow query
1. str(qs.query) — confirm the ORM is generating what you expect.
2. connection.queries — count how many queries ran (spot N+1).
3. EXPLAIN — find Seq Scans on large tables; add db_index=True to the field.
