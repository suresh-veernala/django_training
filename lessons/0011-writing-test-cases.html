Why write tests?

In the org's codebase you build Clean Architecture interactors, presenters, and adapters. Tests give you three things that matter on a real team:

  1. Confidence your code works. "It works on my machine" is not a reliable signal — a test suite is.
  2. Safety when refactoring. Reorganise an interactor and immediately know if you broke a contract.
  3. Documentation through examples. A well-named test is a one-line spec of what the code should do.
Note on terminology
"Testing" in this course always means automated testing — code the machine runs without human input. It does not prove your code is correct under all conditions; it reports whether the conditions you thought of are handled correctly. Source: NxtWave Unit Tests guide

pytest basics & naming rules

All test packages are already declared in the shared repo's pyproject.toml. Install them with:

poetry install --with dev,test

pytest discovers tests by following three naming rules. Break any one of them and your test is silently skipped.

  1. File: name starts or ends with test — e.g. test_movies.py
  2. Class: name starts with Test and has no __init__ — e.g. class TestCreatePost
  3. Function: name starts with test_ — e.g. def test_returns_movie_details_for_valid_id()
Silent failure trap
Writing def check_prime() instead of def test_check_prime() gives you 0 tests collected with no error message. pytest just skips it.

Step 1 — List your cases before writing a single line

Before touching code, make a list of every scenario the function must handle. Group them into three kinds:

Normal cases

The happy path — inputs that should work and return a value.

Edge cases

Boundary values — zero, empty lists, None, floats where int is expected.

Error cases

Inputs that should raise an exception or trigger an error path.

Example — create_post(user_id, post_content)

  • normal Creates a post for a valid user with non-empty content — returns post_id
  • normal Returned post_id is an integer
  • edge Creates a post when content is exactly 1000 characters (max length)
  • edge Creates a post with content that contains only whitespace — should it raise?
  • error Raises InvalidUserException for a non-existent user_id
  • error Raises InvalidPostContent for an empty string content
The org rule
Write this list first — as a comment block at the top of your test file. It forces you to think about the contract before touching any code.

Step 2 — Write each test: Arrange · Act · Assert

Every test in the org's codebase follows this three-part structure. Follow it even when it feels mechanical — it makes failures much easier to diagnose.

Step 1

Arrange

Create all objects, data, and mocks needed before calling the function.

Step 2

Act

Call the function under test. Store the return value in a variable.

Step 3

Assert

Verify the result is what you expected using assert.

A complete example

# fb_post/tests/test_create_post.py
import pytest
from unittest.mock import Mock
from fb_post.interactors.create_post_interactor import CreatePostInteractor
from fb_post.exceptions import InvalidUserException


def test_create_post_with_valid_user_and_content_returns_post_id():
    # Arrange
    storage_mock = Mock()
    storage_mock.create_post.return_value = 42
    presenter_mock = Mock()
    presenter_mock.response.return_value = {"post_id": 42}
    interactor = CreatePostInteractor(storage=storage_mock)

    # Act
    result = interactor.create_post_wrapper(
        user_id=1, post_content="Hello world!", presenter=presenter_mock)

    # Assert
    assert result == {"post_id": 42}

Naming — the three-part formula

A good test name reads like a sentence: method · scenario · expected result.

# Bad — says nothing about scenario or outcome
def test_post(): ...

# Good
def test_create_post_with_valid_user_and_content_returns_post_id(): ...
def test_create_post_with_invalid_user_raises_exception(): ...
def test_create_post_with_empty_content_raises_invalid_post_content(): ...
Tests are documentation
When a PR shows 12 passing tests, a reviewer can read the names and understand every scenario the feature handles — without opening the implementation.

Common assert patterns

# Equality
assert result["post_id"] == 42

# Boolean / None
assert result is not None
assert dto.post_id is not None

# Length
assert len(comments) == 3

# Membership
assert post_id in [dto.post_id for dto in results]
One assert per test (usually)
If a test has 5 asserts, the first failure hides the rest. Split into separate functions so every failure is visible. Exception: asserting multiple fields on a single returned object is fine.

Asserting exceptions with pytest.raises

For error-case tests, never use a bare try/except. Use pytest.raises() — it is a context manager that passes only if the specified exception is raised inside the with block. 

def test_create_post_with_invalid_user_raises_exception():
    # Arrange
    INVALID_USER_ID = -999
    storage_mock = Mock()
    storage_mock.validate_user_id.side_effect = InvalidUserException
    interactor = CreatePostInteractor(storage=storage_mock)

    # Act & Assert
    with pytest.raises(InvalidUserException):
        interactor.create_post_wrapper(
            user_id=INVALID_USER_ID, post_content="Hello!", presenter=Mock())

If you also need to check the exception message:

with pytest.raises(InvalidUserException) as exc_info:
    interactor.create_post_wrapper(user_id=-999, post_content="Hello!", presenter=Mock())
assert str(exc_info.value) == "User with id -999 does not exist"
Don't use bare try/except in tests
try: call() except MyException: assert True will pass even if the wrong exception is raised. pytest.raises checks the exact exception type.

Assert vs Snapshot — which to use per layer

The org uses two different verification strategies depending on which layer is under test. Mixing them up is a common mistake.

LayerTest folderStorage dependencyVerification
Interactors tests/interactors/ Mock() — no real storage assert — check DTOs, return values, exceptions
Storages tests/storages/ StorageImplementation() — real DB assert — check saved row, returned DTO
Presenters tests/presenters/ — snapshot — pins entire JSON shape
Views (DSU) tests/views/ adapter mocked via mocker.patch snapshot — pins full HTTP status + body
Interactor — use assert (Mock storage) 
# tests/interactors/test_create_post_interactor.py
from unittest.mock import Mock
from fb_post.interactors.create_post_interactor import CreatePostInteractor

def test_create_post_returns_post_id():
    storage_mock = Mock()
    storage_mock.create_post.return_value = 42
    interactor = CreatePostInteractor(storage=storage_mock)

    result_dto = interactor.create_post_core(
        user_id=1, post_content="Hello!")

    assert result_dto.post_id == 42
    storage_mock.create_post.assert_called_once_with(
        user_id=1, post_content="Hello!")
Presenter / View — use snapshot 
# tests/presenters/test_create_post_presenter.py
import json
from fb_post.tests.factories.presenter_dtos import \
    CreatePostResponseDTOFactory

def test_create_post_success_response(snapshot):
    dto = CreatePostResponseDTOFactory(post_id=42)
    presenter = CreatePostPresenterImplementation()

    response = presenter.get_create_post_response(dto)

    data = json.loads(response.content)
    snapshot.assert_match(data, "response")
    snapshot.assert_match(
        response.status_code, "status_code")
The rule
assert when you know the exact expected value — for interactors and storages.
snapshot when you want to pin the full JSON shape — for presenters and views. Snapshots are stored in tests/snapshots/ and are committed to git. They act as a safety net against unintentional API contract changes.
Which tests import the storage implementation?
Storage tests only. Interactor tests never import CreatePostStorageImplementation — they use Mock() instead.
LayerStorage dependency
Interactor test (tests/interactors/)Mock() — no DB, no import of implementation
Storage test (tests/storages/)CreatePostStorageImplementation() — real DB, needs @pytest.mark.django_db

Any test that reads or writes the database must be decorated with @pytest.mark.django_db. This applies to storage tests: 

# fb_post/tests/storages/test_create_post_storage.py
import pytest
from fb_post.tests.factories.models import UserFactory
from fb_post.storages.create_post_storage import CreatePostStorageImplementation

@pytest.mark.django_db
def test_create_post_saves_to_database():
    # Arrange — real DB row via factory
    user = UserFactory()
    storage = CreatePostStorageImplementation()

    # Act
    post_id = storage.create_post(user_id=user.id, post_content="Hello world!")

    # Assert
    assert post_id is not None
Each test gets a clean database
pytest-django wraps every test in a transaction and rolls it back after. Tests never pollute each other's state.

Part B

Factory Boy — test data without noise

The problem with creating test data manually is that it leaks irrelevant details into your tests. If Post has required FK fields but you're testing content logic, you shouldn't have to specify all 12 every time.

Factory Boy defines sensible defaults for every model field once, then lets you override only what matters for a given test. It is included in the repo's dev/test dependencies — available after poetry install --with dev,test.

Two base classes — choose by layer

The org uses two different base classes depending on what you are building:

Base classSaves to DB?Factory file
DjangoModelFactoryYesfactories/models.py
factory.FactoryNofactories/storage_dtos.py, interactor_dtos.py, presenter_dtos.py

factories/models.py — real DB rows

Use DjangoModelFactory for Django model instances. It calls Model.objects.create(). Tests using these need @pytest.mark.django_db.

# fb_post/tests/factories/models.py
import factory
from factory.django import DjangoModelFactory
from fb_post.models import User, Post

class UserFactory(DjangoModelFactory):
    class Meta:
        model = User

    name        = factory.Sequence(lambda n: f"User {n}")
    profile_pic = factory.LazyAttribute(lambda o: f"https://pic.example.com/{o.name}.jpg")

class PostFactory(DjangoModelFactory):
    class Meta:
        model = Post

    posted_by = factory.SubFactory(UserFactory)
    content   = factory.Sequence(lambda n: f"Post content {n}")

factories/storage_dtos.py — DTO factories

Use factory.Factory (no Django) for plain dataclass DTOs — no DB is touched. This is used in storage, interactor, and presenter test files.

# fb_post/tests/factories/storage_dtos.py
import factory
from fb_post.interactors.storages.dtos import PostDetailsDTO

class PostDetailsDTOFactory(factory.Factory):
    class Meta:
        model = PostDetailsDTO       # a @dataclass, not a Django model

    post_id        = factory.Sequence(lambda n: n + 1)
    content        = factory.Sequence(lambda n: f"Post content {n}")
    comments_count = 0

# No @pytest.mark.django_db needed:
dto = PostDetailsDTOFactory(post_id=42, content="Hello!")
# Using it:
user_object = UserFactory()                       # name="User 0"
user_object = UserFactory()                       # name="User 1"  (auto-increments)

# Override only what matters for this test:
user_object = UserFactory(name="Alice")

# PostFactory with FK to UserFactory:
post_object = PostFactory()                       # auto-creates a User too
post_object = PostFactory(posted_by=user_object)  # reuse an existing user

Key factory declarations

DeclarationPurposeExample
factory.Sequence Unique value using auto-counter n name = factory.Sequence(lambda n: f"User {n}")
factory.LazyAttribute Derive one field from another email = factory.LazyAttribute(lambda o: f"{o.username}@example.com")
factory.SubFactory Auto-create a related FK object posted_by = factory.SubFactory(UserFactory)

SubFactory — FK relationships

class PostFactory(DjangoModelFactory):
    class Meta:
        model = Post

    # Creates a User automatically — no need to pass user_id manually
    posted_by = factory.SubFactory(UserFactory)
    content   = factory.Sequence(lambda n: f"Post content {n}")

# Creates a Post AND its User in one call:
post_object = PostFactory()

# Reuse an existing user:
post_object = PostFactory(posted_by=existing_user_object)

Part C

Fixtures — reuse your Arrange code

When multiple tests need the same setup, copy-pasting it is fragile. One model field change breaks every test separately. A fixture defines that setup once and injects it automatically.

Repeated Arrange 
def test_get_post_returns_content():
    # duplicated!
    post = PostFactory(content="Hello world!")
    result = storage.get_post_details(post.id)
    assert result.content == "Hello world!"

def test_get_post_returns_poster():
    # duplicated!
    post = PostFactory(content="Hello world!")
    result = storage.get_post_details(post.id)
    assert result.posted_by.name is not None
Fixture 
import pytest

@pytest.fixture
def post_object():
    return PostFactory(content="Hello world!")


def test_get_post_returns_content(post_object):
    result = storage.get_post_details(post_object.id)
    assert result.content == "Hello world!"

def test_get_post_returns_poster_name(post_object):
    result = storage.get_post_details(post_object.id)
    assert result.posted_by.name is not None

pytest sees that post_object is a parameter name that matches a fixture. It calls the fixture, gets the return value, and passes it as the argument. A fresh object is created for every test — they never share state.

Shared fixtures — conftest.py

Fixtures defined in the same file are only available in that file. To share a fixture across all tests in an app, put it in conftest.py — pytest discovers it automatically, no import needed.

fb_post/tests/
├── conftest.pyshared fixtures — auto-discovered by pytest
├── factories/
│ ├── models.pyDjangoModelFactory classes
│ ├── storage_dtos.pyfactory.Factory for storage DTOs
│ ├── interactor_dtos.pyfactory.Factory for interactor DTOs
│ └── presenter_dtos.pyfactory.Factory for presenter DTOs
├── interactors/test_create_post.py, test_get_post.py, …
├── storages/test_create_post_storage.py, …
└── presenters/test_create_post_presenter.py, …
# fb_post/tests/conftest.py
import pytest
from fb_post.tests.factories.models import UserFactory, PostFactory

@pytest.fixture
def user_object():          # org convention: _object suffix for model instances
    return UserFactory()

@pytest.fixture
def post_object(user_object):
    return PostFactory(posted_by=user_object)
Naming convention
The org names Django model instances with the _object suffix. So your fixture is user_object / post_object, not user / post.
Part D

Parametrize — one test, many inputs

Sometimes you need to verify the same behaviour across many input values. Adding a loop inside a test is bad practice — logic in tests hides bugs. @pytest.mark.parametrize is the right tool.

Loop in test 
def test_invalid_reaction_types():
    invalid = ["INVALID", "", "thumbs_up", "123"]
    for reaction_type in invalid:
        # first failure hides the rest
        with pytest.raises(InvalidReactionTypeException):
            storage.react_to_post(
                user_id=1, post_id=1, reaction_type=reaction_type)
Parametrize 
@pytest.mark.parametrize("reaction_type", [
    "INVALID", "", "thumbs_up", "123"
])
def test_react_to_post_with_invalid_reaction_type_raises_exception(
        reaction_type):
    with pytest.raises(InvalidReactionTypeException):
        storage.react_to_post(
            user_id=1, post_id=1, reaction_type=reaction_type)
# pytest runs this 4× — reports each
# failure separately with its input value

Multiple parameters at once

@pytest.mark.parametrize("content, expected_count", [
    ("Hello!",       1),
    ("Post 2 content", 2),
    ("Third post",    3),
])
def test_get_posts_returns_correct_count(content, expected_count):
    PostFactory.create_batch(expected_count)
    result = storage.get_all_posts()
    assert len(result) == expected_count
What pytest reports
Each parametrized run appears separately: test_react_to_post_with_invalid_reaction_type_raises_exception[INVALID]. Failures are pinpointed to the exact input.

Putting it all together — Factory Boy + Fixtures + Parametrize

# fb_post/tests/storages/test_get_post_storage.py
import pytest
from fb_post.tests.factories.models import UserFactory, PostFactory, CommentFactory
from fb_post.storages.get_post_storage import GetPostStorageImplementation


@pytest.fixture                              # Part C — fixture
def post_with_comments():
    user = UserFactory()
    post = PostFactory(posted_by=user)
    CommentFactory.create_batch(3, post=post, commented_by=user)  # Part B — factory
    return post


@pytest.mark.django_db
@pytest.mark.parametrize("extra_comments, expected_count", [  # Part D — parametrize
    (0, 3),
    (2, 5),
])
def test_get_post_returns_correct_comments_count(
        post_with_comments, extra_comments, expected_count):
    if extra_comments:
        CommentFactory.create_batch(extra_comments, post=post_with_comments)
    storage = GetPostStorageImplementation()
    result = storage.get_post_details(post_id=post_with_comments.id)
    assert result.comments_count == expected_count   # storage tests use assert

common_fixtures/adapters.py — adapter mock helpers

Adapter patches are centralised in tests/common_fixtures/adapters.py as plain functions that accept mocker. This keeps the patch target string in one place — if the adapter path ever changes, you fix it once. 

# fb_post/tests/common_fixtures/adapters.py
def create_post_mock(mocker):
    return mocker.patch(
        "fb_post.adapters.service_adapter.ServiceAdapter.create_post"
    )

How snapshots work in DSU view tests

make_api_call() fires the full HTTP request through Django's test client. It then calls snapshot.assert_match(response, "name") internally.

First run — no snapshot file exists yet:

  1. The response is serialised (status code + JSON body).
  2. It is written to snapshots/snap_test_case_01.py — the golden file.
  3. The test passes.

Every subsequent run:

  1. The response is serialised again.
  2. It is compared byte-for-byte against the golden file.
  3. Any difference — a new field, changed status code, renamed key — fails the test immediately.
Updating a snapshot
If you intentionally changed the API response, delete the snapshot file and re-run. The new response becomes the new golden file. Commit both the code change and the updated snapshot file together.

What a snapshot file looks like

# fb_post/tests/views/create_post/snapshots/snap_test_case_01.py
# This file is auto-generated — do not edit manually
snapshots = {
    'test_case 1': {
        'status_code': 400,
        'response': {
            'res_status': 'INVALID_USER_EXCEPTION',
            'response':   'Invalid user exception'
        }
    }
}

# snap_test_case_02.py — success case
snapshots = {
    'test_case 1': {
        'status_code': 201,
        'response': {
            'post_id': 1
        }
    }
}

Snapshot files are committed to git. A reviewer reading the PR sees not just the test code but also the exact JSON the endpoint will return — no manual documentation needed.

Why snapshot for views, not assert?
A JSON response can have 10+ fields. Writing assert response["post_id"] == 1 for every field is tedious and easy to miss. Snapshot captures the whole shape in one call. If the presenter accidentally drops a field or changes a key name, the snapshot diff catches it instantly.

One test case file per scenario

The org convention is one file per case — not one file per endpoint. Failures are pinpointed and diffs are readable in code review.

FileScenario
test_case_01.pyError: invalid user_id → 400 INVALID_USER_EXCEPTION
test_case_02.pySuccess: valid user + content → 201 with post_id
test_case_03.pyError: empty content → 400 INVALID_POST_CONTENT
DSU view tests vs interactor tests
View tests (TestUtils) go through the full HTTP stack — use them to test the request/response contract and snapshot the JSON shape.
Interactor tests (tests/interactors/) mock the storage and test business logic in isolation — much faster and more granular. Write both.
