Why write tests?

In the org's codebase you build Clean Architecture interactors, presenters, and adapters. Tests give you three things that matter on a real team:

  1. Confidence your code works. "It works on my machine" is not a reliable signal — a test suite is.
  2. Safety when refactoring. Reorganise an interactor and immediately know if you broke a contract.
  3. Documentation through examples. A well-named test is a one-line spec of what the code should do.
Note on terminology
"Testing" in this course always means automated testing — code the machine runs without human input. It does not prove your code is correct under all conditions; it reports whether the conditions you thought of are handled correctly. Source: NxtWave Unit Tests guide

pytest basics & naming rules

Install pytest and the Django plugin (already present in most org projects):

pip install pytest pytest-django

pytest discovers tests by following three naming rules. Break any one of them and your test is silently skipped.

  1. File: name starts or ends with test — e.g. test_movies.py
  2. Class: name starts with Test and has no __init__ — e.g. class TestGetMovie
  3. Function: name starts with test_ — e.g. def test_returns_movie_details_for_valid_id()
Silent failure trap
Writing def check_prime() instead of def test_check_prime() gives you 0 tests collected with no error message. pytest just skips it.

Step 1 — List your cases before writing a single line

Before touching code, make a list of every scenario the function must handle. Group them into three kinds:

Normal cases

The happy path — inputs that should work and return a value.

Edge cases

Boundary values — zero, empty lists, None, floats where int is expected.

Error cases

Inputs that should raise an exception or trigger an error path.

Example — get_movie_rating(movie_id)

  • normal Returns rating for a valid movie ID that exists in the DB
  • normal Returns a float value (not an integer)
  • edge Returns 0 when the movie has no ratings yet
  • edge Handles a movie with exactly one rating
  • error Raises MovieDoesNotExistException for an invalid ID
  • error Raises an exception when movie_id is None
The org rule
Write this list first — as a comment block at the top of your test file. It forces you to think about the contract before touching any code.

Step 2 — Write each test: Arrange · Act · Assert

Every test in the org's codebase follows this three-part structure. Follow it even when it feels mechanical — it makes failures much easier to diagnose.

Step 1

Arrange

Create all objects, data, and mocks needed before calling the function.

Step 2

Act

Call the function under test. Store the return value in a variable.

Step 3

Assert

Verify the result is what you expected using assert.

A complete example

# imdb/tests/test_get_movie.py
import pytest
from imdb.interactors.get_movie_interactor import GetMovieInteractor
from imdb.exceptions import MovieDoesNotExistException


def test_get_movie_with_valid_id_returns_movie_dto():
    # Arrange
    movie_id = 1
    interactor = GetMovieInteractor(movie_id=movie_id)

    # Act
    result = interactor.execute()

    # Assert
    assert result.movie_id == movie_id

Naming — the three-part formula

A good test name reads like a sentence: method · scenario · expected result.

# Bad — says nothing about scenario or outcome
def test_movie(): ...

# Good
def test_get_movie_with_valid_id_returns_movie_dto(): ...
def test_get_movie_with_invalid_id_raises_not_found_exception(): ...
def test_create_movie_with_duplicate_title_raises_validation_error(): ...
Tests are documentation
When a PR shows 12 passing tests, a reviewer can read the names and understand every scenario the feature handles — without opening the implementation.

Common assert patterns

# Equality
assert dto.movie_id == 42

# Boolean / None
assert result is True
assert result is not None

# Length
assert len(results) == 3

# Membership
assert movie_id in [dto.movie_id for dto in results]
One assert per test (usually)
If a test has 5 asserts, the first failure hides the rest. Split into separate functions so every failure is visible. Exception: asserting multiple fields on a single returned object is fine.

Asserting exceptions with pytest.raises

For error-case tests, never use a bare try/except. Use pytest.raises() — it is a context manager that passes only if the specified exception is raised inside the with block. 

def test_get_movie_with_invalid_id_raises_exception():
    # Arrange
    INVALID_MOVIE_ID = -999
    interactor = GetMovieInteractor(movie_id=INVALID_MOVIE_ID)

    # Act & Assert
    with pytest.raises(MovieDoesNotExistException):
        interactor.execute()

If you also need to check the exception message:

with pytest.raises(MovieDoesNotExistException) as exc_info:
    interactor.execute()
assert str(exc_info.value) == "Movie with id -999 does not exist"
Don't use bare try/except in tests
try: call() except MyException: assert True will pass even if the wrong exception is raised. pytest.raises checks the exact exception type.

Django setup — pytest.ini and @pytest.mark.django_db

Create pytest.ini in the project root (next to manage.py):

[pytest]
python_files = test*.py
DJANGO_SETTINGS_MODULE = imdb.settings.local

Any test that reads or writes the database must be decorated with @pytest.mark.django_db: 

import pytest

@pytest.mark.django_db
def test_get_movie_with_valid_id_returns_movie_dto():
    # Arrange — creates a real DB row
    from imdb.tests.factories import MovieFactory
    movie = MovieFactory()

    # Act
    from imdb.interactors import GetMovieInteractor
    result = GetMovieInteractor(movie_id=movie.id).execute()

    # Assert
    assert result.movie_id == movie.id
Each test gets a clean database
pytest-django wraps every test in a transaction and rolls it back after. Tests never pollute each other's state.

Running tests

# All tests
pytest

# Specific file
pytest imdb/tests/test_get_movie.py

# Specific function
pytest imdb/tests/test_get_movie.py::test_get_movie_with_valid_id_returns_movie_dto

# Verbose (shows full name per result)
pytest -v

Reading the output: a dot = pass, F = fail, E = error during setup.

Part B

Factory Boy — test data without noise

The problem with creating test data manually is that it leaks irrelevant details into your tests. If Movie has 12 required fields but you're testing rating logic, you shouldn't have to specify all 12 every time.

Factory Boy defines sensible defaults for every model field once, then lets you override only what matters for a given test. 

pip install factory_boy

DjangoModelFactory — the base class you always use

For Django models, always subclass factory.django.DjangoModelFactory — not plain factory.Factory. It calls Model.objects.create() and handles DB saving correctly.

# imdb/tests/factories/movie_factory.py
import factory
from factory.django import DjangoModelFactory
from imdb.models import Movie

class MovieFactory(DjangoModelFactory):
    class Meta:
        model = Movie

    title  = factory.Sequence(lambda n: f"Movie {n}")   # unique on every call
    rating = 7.5
    year   = 2020
# Using it:
movie_object = MovieFactory()                       # title="Movie 0", rating=7.5
movie_object = MovieFactory()                       # title="Movie 1"  (auto-increments)

# Override only what matters for this test:
movie_object = MovieFactory(rating=9.2)
movie_object = MovieFactory(title="Inception", year=2010)

Key factory declarations

DeclarationPurposeExample
factory.Sequence Unique value using auto-counter n title = factory.Sequence(lambda n: f"Movie {n}")
factory.LazyAttribute Derive one field from another email = factory.LazyAttribute(lambda o: f"{o.username}@example.com")
factory.SubFactory Auto-create a related FK object movie = factory.SubFactory(MovieFactory)

SubFactory — FK relationships

class RatingFactory(DjangoModelFactory):
    class Meta:
        model = Rating

    # Creates a Movie automatically — no need to pass movie_id manually
    movie  = factory.SubFactory(MovieFactory)
    score  = 8
    review = factory.Sequence(lambda n: f"Review {n}")

# Creates a Rating AND its Movie in one call:
rating_object = RatingFactory()

# Reuse an existing movie:
rating_object = RatingFactory(movie=existing_movie_object)

create() vs build() vs create_batch()

MethodSaves to DB?When to use
MovieFactory() / .create()YesTests that need a real DB row (requires @pytest.mark.django_db)
MovieFactory.build()NoTests that don't touch the DB — much faster
MovieFactory.create_batch(5)YesNeed multiple objects at once
Part C

Fixtures — reuse your Arrange code

When multiple tests need the same setup, copy-pasting it is fragile. One model field change breaks every test separately. A fixture defines that setup once and injects it automatically.

Repeated Arrange 
def test_get_movie_title():
    # duplicated!
    movie = MovieFactory(title="Inception")
    result = GetMovieInteractor(movie.id).execute()
    assert result.title == "Inception"

def test_get_movie_rating():
    # duplicated!
    movie = MovieFactory(title="Inception")
    result = GetMovieInteractor(movie.id).execute()
    assert result.rating == 8.8
Fixture 
import pytest

@pytest.fixture
def inception_movie():
    return MovieFactory(title="Inception", rating=8.8)


def test_get_movie_title(inception_movie):
    result = GetMovieInteractor(inception_movie.id).execute()
    assert result.title == "Inception"

def test_get_movie_rating(inception_movie):
    result = GetMovieInteractor(inception_movie.id).execute()
    assert result.rating == 8.8

pytest sees that inception_movie is a parameter name that matches a fixture. It calls the fixture, gets the return value, and passes it as the argument. A fresh object is created for every test — they never share state.

Shared fixtures — conftest.py

Fixtures defined in the same file are only available in that file. To share a fixture across all tests in an app, put it in conftest.py — pytest discovers it automatically, no import needed.

imdb/tests/
├── conftest.pyshared fixtures for all test files here
├── factories/Factory Boy factories
├── test_get_movie.pyuses fixtures from conftest.py
└── test_create_movie.pyalso uses fixtures from conftest.py
# imdb/tests/conftest.py
import pytest
from imdb.tests.factories import MovieFactory, ActorFactory

@pytest.fixture
def movie_object():          # org convention: _object suffix for model instances
    return MovieFactory()

@pytest.fixture
def actor_object():
    return ActorFactory()
Naming convention
The org names Django model instances with the _object suffix. So your fixture is movie_object, not movie.
Part D

Parametrize — one test, many inputs

Sometimes you need to verify the same behaviour across many input values. Adding a loop inside a test is bad practice — logic in tests hides bugs. @pytest.mark.parametrize is the right tool.

Loop in test 
def test_invalid_ratings():
    invalid = [-1, 0, 10.1, 100]
    for rating in invalid:
        # first failure hides the rest
        with pytest.raises(ValidationError):
            CreateMovieInteractor(
                rating=rating
            ).execute()
Parametrize 
@pytest.mark.parametrize("rating", [
    -1, 0, 10.1, 100
])
def test_create_movie_with_invalid_rating_raises_error(
        rating):
    with pytest.raises(ValidationError):
        CreateMovieInteractor(
            rating=rating
        ).execute()
# pytest runs this 4× — reports each
# failure separately with its input value

Multiple parameters at once

@pytest.mark.parametrize("title, expected_slug", [
    ("The Dark Knight", "the-dark-knight"),
    ("Inception",       "inception"),
    ("2001: A Space",   "2001-a-space"),
])
def test_slugify_title_returns_expected_slug(title, expected_slug):
    result = slugify_title(title)
    assert result == expected_slug
What pytest reports
Each parametrized run appears separately: test_slugify_title[The Dark Knight-the-dark-knight]. Failures are pinpointed to the exact input.

Putting it all together — Factory Boy + Fixtures + Parametrize

# imdb/tests/test_get_top_rated_movies.py
import pytest
from imdb.tests.factories import MovieFactory
from imdb.interactors import GetTopRatedMoviesInteractor


@pytest.fixture                              # Part C — fixture
def top_rated_movies():
    MovieFactory.create_batch(3, rating=9.0)  # Part B — factory
    MovieFactory.create_batch(2, rating=5.0)


@pytest.mark.django_db
@pytest.mark.parametrize("min_rating, expected_count", [  # Part D — parametrize
    (8.0, 3),
    (5.0, 5),
    (9.5, 0),
])
def test_returns_correct_count_for_various_thresholds(
        top_rated_movies, min_rating, expected_count):
    results = GetTopRatedMoviesInteractor(min_rating=min_rating).execute()
    assert len(results) == expected_count
