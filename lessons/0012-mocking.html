Lesson 14 Testing Track

Mocking — Test in Isolation

Replace unpredictable dependencies with controlled stand-ins so every test runs reliably, fast, and without hitting a real database or external service.

After this lesson you will be able to
  • Explain why uncontrolled dependencies make tests unreliable
  • Create a Mock() object and set its return_value
  • Use side_effect to force a mock to raise an exception
  • Assert that a mock was called with the right arguments
  • Patch a dependency that is imported inside a function with @patch
  • Patch a single method on a class with patch.object
  • Prevent misspelling bugs with create_autospec

Why mock?

Your interactors call storage methods that hit a database, other services, or external APIs. When a test calls those things for real, several things go wrong:

Tests fail randomly

A share-price API returns a different number each time. Your hard-coded assertion breaks by tomorrow morning.

Tests are slow

Every HTTP round-trip adds seconds. A suite with 200 tests that each make one network call takes minutes.

Hard-to-reach paths

You cannot easily force requests.get to raise TimeoutError in a real network environment.

Dirty side effects

If your test calls a real write API, it modifies production or staging data every time the suite runs.

Mocking solves all of these by replacing the real dependency with a fake that you control completely — returning the value you need, raising the exception you need, instantly.

Source
All concepts in this lesson come from the NxtWave Mocking guide, Mocking in Python, and Using Patch.

Three tools, three situations

Mock()

The dependency is passed in as an argument to the function under test (e.g., an injected storage object).

patch()

The dependency is imported inside the function — you cannot replace it by passing an argument.

patch.object()

You want to replace one method on a real object/class while keeping the rest of it intact.

Part A

The Mock object

unittest.mock.Mock creates an object that simulates any real object. Its key property: it creates attributes and methods on-demand — access any name on a Mock and it returns another Mock, never raising AttributeError. 

from unittest.mock import Mock

storage_mock = Mock()

# Access any attribute — always works:
storage_mock.get_movie_by_id     # <Mock name='mock.get_movie_by_id'>
storage_mock.get_movie_by_id()   # <Mock name='mock.get_movie_by_id()'>

In your org's Clean Architecture tests, the storage interface is injected into the interactor. That means you can just pass a Mock() straight in: 

# fb_post/tests/test_create_post_interactor.py
from unittest.mock import Mock
from fb_post.interactors.create_post_interactor import CreatePostInteractor

def test_create_post_with_valid_data_returns_post_id():
    # Arrange
    storage_mock = Mock()
    presenter_mock = Mock()
    interactor = CreatePostInteractor(storage=storage_mock)

    # Act — runs without hitting the real database
    result = interactor.create_post_wrapper(
        user_id=1, post_content="Hello world!", presenter=presenter_mock)

    # Assert
    ...
Why this works in Clean Architecture
The interactor only knows the interface, not the implementation. A Mock satisfies any interface because it accepts every attribute access and every call. This is the main payoff of Dependency Inversion.

return_value — controlling what a mock returns

By default, calling a mock method returns another Mock — not useful for assertions. Set return_value to make the mock return exactly what you need: 

from unittest.mock import Mock
from fb_post.tests.factories import UserFactory

def test_create_post_returns_post_id():
    # Arrange
    storage_mock = Mock()
    storage_mock.create_post.return_value = 42   # pin the return value

    storage_mock = Mock()
    storage_mock.get_movie_by_id.return_value = movie_object  # pin the return value

    presenter_mock = Mock()
    presenter_mock.response.return_value = {"post_id": 42}

    interactor = CreatePostInteractor(storage=storage_mock)

    # Act
    result = interactor.create_post_wrapper(
        user_id=1, post_content="Hello world!", presenter=presenter_mock)

    # Assert
    assert result == {"post_id": 42}
The syntax rule
mock_obj.method_name.return_value = <value> — always on the method attribute, not on the mock itself.

side_effect — raising exceptions & more

side_effect is more powerful than return_value. You can use it to make a mock method raise an exception, call another function, or return different values on consecutive calls.

Raising an exception (the most common case in your org's tests)

import pytest
from unittest.mock import Mock
from fb_post.exceptions import InvalidUserException

def test_create_post_with_invalid_user_raises_exception():
    # Arrange
    storage_mock = Mock()
    storage_mock.validate_user_id.side_effect = InvalidUserException

    interactor = CreatePostInteractor(storage=storage_mock)

    # Act & Assert
    with pytest.raises(InvalidUserException):
        interactor.create_post_wrapper(
            user_id=-999, post_content="Hello!", presenter=Mock())
return_value vs side_effect
If both are set, side_effect wins — it takes priority. To clear a side_effect later, set it to None.

The iterable form — different outputs on each call

# First call returns a movie, second call raises the exception
storage_mock.validate_user_id.side_effect = [None, InvalidUserException]

storage_mock.validate_user_id(1)    # valid — returns None
storage_mock.validate_user_id(99)   # raises InvalidUserException

Assertions — verifying the mock was called correctly

A Mock records every call made to it. After the Act step, you can assert how the storage was called — verifying the interactor passed the right arguments.

AssertionPasses when
.assert_called()Method was called at least once
.assert_called_once()Method was called exactly once
.assert_called_with(*args, **kwargs)Last call used these exact arguments
.assert_called_once_with(*args, **kwargs)Called exactly once with these exact arguments
.assert_not_called()Method was never called

Example — asserting the storage was called with the right ID

def test_interactor_passes_correct_args_to_storage():
    # Arrange
    storage_mock = Mock()
    storage_mock.create_post.return_value = 42
    interactor = CreatePostInteractor(storage=storage_mock)

    # Act
    interactor.create_post_wrapper(
        user_id=1, post_content="Hello!", presenter=Mock())

    # Assert — did the interactor forward args correctly?
    storage_mock.create_post.assert_called_once_with(user_id=1, post_content="Hello!")
Positional vs keyword arguments matter
assert_called_with(42) and assert_called_with(movie_id=42) are different assertions — they check exact call signatures. Match exactly how the code calls the method.

Inspection attributes (for debugging)

storage_mock.create_post.call_count        # how many times called
storage_mock.create_post.call_args         # arguments from the last call
storage_mock.create_post.call_args_list    # list of all calls
Part B

patch() — replacing an imported dependency

Mock() works when the dependency is passed in. But what if the function you're testing imports the dependency internally?

Dependency injected — use Mock() 
def is_weekend(calendar_obj):
    day = calendar_obj.get_current_day()
    return day in [6, 7]

# You can pass a Mock() directly
Imported inside — must use patch() 
def is_weekend():
    from my_calendar import get_current_day
    day = get_current_day()
    return day in [6, 7]

# Cannot pass anything — use patch()

patch() looks up the target object in its module and temporarily replaces it with a Mock for the duration of the test.

The target string format

my_calendar . get_current_day
module where the name lives name to replace in that module

patch as a decorator (most common)

from unittest.mock import patch
from is_weekend import is_weekend

@patch('my_calendar.get_current_day')
def test_is_weekend_on_saturday(get_current_day_mock):
    # Arrange — the mock is injected as the first argument
    get_current_day_mock.return_value = 6

    # Act
    result = is_weekend()

    # Assert
    assert result is True
    get_current_day_mock.assert_called_once()
Decorator argument order rule
Each @patch decorator injects one argument. Multiple decorators inject from bottom-up — the decorator closest to the function is the first argument.

patch as a context manager (when you only need it for part of the test)

def test_is_weekend_on_saturday():
    with patch('my_calendar.get_current_day') as mock:
        mock.return_value = 6

        result = is_weekend()

        assert result is True
When to use context manager vs decorator
Prefer the decorator when the mock is active for the entire test. Use the context manager when you only want to patch for part of the test body, or when you already have many decorator arguments that hurt readability.

patch.object() — patching one method on a class

Use patch.object() when you want to mock only one method of a class, leaving the rest of the class working normally. 

from unittest.mock import patch
from my_calendar import MyCalendar
from is_weekend import is_weekend_v2

@patch.object(MyCalendar, 'get_current_day', return_value=6)
def test_weekend_v2(get_current_day_mock):
    result = is_weekend_v2()
    assert result is True
ArgumentValue
FirstThe class or object to patch
SecondThe method/attribute name as a string
Third (optional)return_value, side_effect, etc.

Pitfall: misspellings & create_autospec

Because Mock creates attributes on-demand, a misspelling silently creates a new attribute instead of raising an error: 

storage_mock = Mock()
# Intended: assert_called_once()
# Actual typo: asert_called_once() — no error, test passes vacuously
storage_mock.get_movie_by_id.asert_called_once()  # silent bug!

The fix: use create_autospec when mocking a real class. It makes the mock enforce the real interface — accessing an attribute that doesn't exist raises AttributeError. 

from unittest.mock import create_autospec
from fb_post.storages.create_post_storage import CreatePostStorageImplementation

# Creates a mock that enforces the real interface
storage_mock = create_autospec(CreatePostStorageImplementation)

storage_mock.create_post.return_value = 42           # fine — method exists
storage_mock.cerate_post.return_value = 42           # AttributeError — typo caught!
Org recommendation
Use create_autospec whenever mocking a storage class or any injected dependency that has a real implementation. It makes your tests resilient to interface changes — if a method is renamed, the mock immediately flags the mismatch.
