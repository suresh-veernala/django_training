DTOs, Enums, Inter-App Communication, Interactor Structure & Authentication

The five building blocks you touch every time you write a feature end-to-end.

1 Data Transfer Objects (DTOs)

A DTO (Data Transfer Object) is a named, typed container for data that travels between layers. In the org's codebase, DTOs are plain Python @dataclass classes.

Why not dict / tuple?
Dicts fail silently on typos (KeyError at runtime). Tuples require you to remember field order. DTOs give you IDE auto-complete, type checking, and a meaningful repr — at zero cost.

How a DTO looks

from dataclasses import dataclass
from typing import List, Optional

@dataclass
class PostDTO:
    post_id: int
    content: str
    author_id: int
    image_url: Optional[str] = None    # Optional field — default to None

That's it — plain @dataclass, typed fields, Optional[X] = None for fields that may be absent. No extra decorator arguments needed.

Naming conventions

  • Class names end in DTO — e.g. PostDTO, UserDetailsDTO
  • Variable names end in _dto — e.g. post_dto, user_dto
  • Factory classes end in Factory — e.g. PostDTOFactory (in tests)

Where DTOs live

interactors/business logic
storages/dtos.pystorage ↔ interactor DTOs
presenters/dtos.pyinteractor → presenter DTOs
dtos/dtos.pyDTOs shared across multiple interactors
# DTOs used only within one interactor → defined in that interactor file itself

2 Enums

An enum is a set of symbolic names bound to unique, constant values.

Python enum basics

import enum

class GenderEnum(enum.Enum):
    MALE   = "MALE"
    FEMALE = "FEMALE"
    OTHER  = "OTHER"

# Compare by identity (preferred)
if user.gender == GenderEnum.MALE:
    ...

# Access the string value (e.g. to store in DB)
GenderEnum.MALE.value    # → "MALE"

Where enums live — constants/enums.py

import enum

    DELETED   = "DELETED"

class ReactionTypeEnum(enum.Enum):
    LIKE  = "LIKE"
    WOW   = "WOW"

Enums in DTOs, storages, and API specs

class UserDTO:

    name: str

# In a storage interface (sorting/filtering)
class SortByEnum(enum.Enum):
    NAME = "NAME"

3 Inter-App Communication

A Django project typically has multiple apps. When App A needs data from App B, it must not reach directly into App B's models or interactors. Instead, it goes through two gateways:

  1. app_interfaces/service_interface.py in App B — what App B promises to expose.
  2. adapters/ in App A — the translator that speaks App B's language and converts it to App A's language.
App A Interactor → App A Adapter → App B ServiceInterface → App B Interactor

App B — exposing a ServiceInterface

# app_b/app_interfaces/service_interface.py
from app_b.interactors.storage_interfaces.dtos import UserDetailsDTO

class ServiceInterface:

    def get_user_details(self, user_id: int) -> UserDetailsDTO:
        from app_b.interactors.get_user_details_interactor import GetUserDetailsInteractor
        from app_b.storages.storage_implementation import StorageImpl
        interactor = GetUserDetailsInteractor(storage=StorageImpl())
        return interactor.get_user_details_wrapper(user_id=user_id)

App A — writing an Adapter

# app_a/adapters/auth_service_adapter.py
from dataclasses import dataclass

@dataclass
class AuthUserDTO:              # App A's own DTO — NOT imported from app_b
    user_id: int
    username: str
    is_verified: bool

class AuthServiceAdapter:

    @property
    def interface(self):
        from app_b.app_interfaces.service_interface import ServiceInterface
        return ServiceInterface()               # ← local import keeps things loosely coupled

    def get_user(self, user_id: int) -> AuthUserDTO:
        try:
            user_details_dto = self.interface.get_user_details(user_id=user_id)
        except app_b_exceptions.UserDoesNotExist:
            raise app_a_exceptions.InvalidUserId(user_id=user_id)  # ← re-raise as own

        return AuthUserDTO(                     # ← always convert, never pass through directly
            user_id=user_details_dto.user_id,
            username=user_details_dto.username,
            is_verified=user_details_dto.email_verified,
        )
Three rules of Adapters
  1. Always convert DTOs — even if App B's DTO looks identical to what you need. This prevents accidental coupling.
  2. Always re-raise exceptions — catch App B's exceptions in the adapter and raise your own app's exceptions instead.
  3. DTOs live in the adapter file

ServiceAdapter — the single entry-point

Each adapter is exposed as a @property on ServiceAdapter, with the adapter class imported locally inside the property. A module-level get_service_adapter() factory is what interactors actually call.

# app_a/adapters/service_adapter.py

class ServiceAdapter:

    @property
    def auth(self):
        from app_a.adapters.auth_service_adapter import AuthServiceAdapter
        return AuthServiceAdapter()

    @property
    def notifications(self):
        from app_a.adapters.notifications_adapter import NotificationsAdapter
        return NotificationsAdapter()

    @property
    def payments(self):
        from app_a.adapters.payments_service_adapter import PaymentsServiceAdapter
        return PaymentsServiceAdapter()


def get_service_adapter():
    return ServiceAdapter()
Why @property + local import?
The local import inside the property runs only when that adapter is actually accessed. This avoids circular imports and keeps startup cost low — the heavy service imports only load if that adapter is used in the current request.

Using the adapter in an interactor

# Inside an interactor method
from app_a.adapters.service_adapter import get_service_adapter

adapter = get_service_adapter()
user_dto = adapter.auth.get_user(user_id=user_id)
# adapter.notifications.send_welcome(user_id=user_id)

4 Interactor Structure — Wrapper & Main

Every interactor has a predictable two-layer structure:

MethodResponsibilityWho calls it
create_post_wrapper
(wrapper method)		 Receives presenter as an argument (wired by the view).
Calls the core method, gets back a DTO, passes it to the presenter.
Catches domain exceptions and routes them to presenter error methods.
Returns the final HttpResponse. 		api_wrapper in the DSU view
create_post
(core method)		 Pure business logic — validates, calls storage, builds and returns a DTO.
Raises domain exceptions; never catches them, never calls presenter.
Easy to unit-test in isolation. 		Called by the wrapper method

Full example

# interactors/create_post_interactor.py
from dataclasses import dataclass
from typing import Optional

from fb_post.interactors.storages.storage_interface import StorageInterface
from fb_post.interactors.presenters.presenter_interface import PresenterInterface
from fb_post.exceptions.custom_exceptions import InvalidUserId, PostContentTooLong
from fb_post.interactors.storages.dtos import PostDTO

@dataclass
class CreatePostInputDTO:          # internal DTO — lives in this file
    user_id: int
    content: str
    image_url: Optional[str] = None


class CreatePostInteractor:

    def __init__(self, storage: StorageInterface) -> None:
        self.storage = storage

    # ── PUBLIC (wrapper) ────────────────────────────────────────
    def create_post_wrapper(
        self,
        user_id: int,
        content: str,
        presenter: PresenterInterface,
        image_url: Optional[str] = None,
    ):
        try:
            post_dto = self.create_post(
                CreatePostInputDTO(user_id=user_id, content=content,
                                   image_url=image_url)
            )
            return presenter.get_create_post_response(post_dto=post_dto)
        except InvalidUserId:
            return presenter.raise_exception_for_invalid_user_id()
        except PostContentTooLong:
            return presenter.raise_exception_for_post_content_too_long()

    # ── CORE (business logic) ───────────────────────────────────
    def create_post(self, input_dto: CreatePostInputDTO) -> PostDTO:
        self.storage.validate_user_id(user_id=input_dto.user_id)
        self._validate_content(input_dto.content)

        post_id = self.storage.create_post(
            user_id=input_dto.user_id,
            content=input_dto.content,
            image_url=input_dto.image_url,
        )
        return self.storage.get_post(post_id=post_id)  # returns PostDTO

    def _validate_content(self, content: str) -> None:
        if len(content) > 1000:
            raise PostContentTooLong(length=len(content))
The key split
Core method returns a DTO. The wrapper receives it and calls presenter.some_response(dto). Presenter is never passed into the core method.

How the view (api_wrapper) wires it up

# views/create_post/api_wrapper.py
@validate_decorator(validator_class=ValidatorClass)
def api_wrapper(*args, **kwargs):
    user_id   = kwargs['user_id']
    body      = kwargs['request_data']
    content   = body['content']
    image_url = body.get('image_url')

    from fb_post.interactors.create_post_interactor import CreatePostInteractor
    from fb_post.storages.storage_implementation import StorageImpl
    from fb_post.presenters.presenter_implementation import PresenterImpl

    storage    = StorageImpl()
    presenter  = PresenterImpl()
    interactor = CreatePostInteractor(storage=storage)

    return interactor.create_post_wrapper(
        user_id=user_id,
        content=content,
        presenter=presenter,
        image_url=image_url,
    )
Key rules recap
  • Wrapper = exception handler + dependency wirer. Core method = pure logic.
  • If a method has > 3 arguments, group them into a DTO.
  • When interactor B is composed inside interactor A, catch B's exceptions in A's wrapper method.
  • Never create new instance variables mid-flow — pass new values as arguments to private methods.
  • Write one test file per public method (per wrapper).

# constants/enums.py
class ReactionTypeEnum(enum.Enum):
    LIKE  = "LIKE"
    WOW   = "WOW"
    ANGRY = "ANGRY"

# interactors/storage_interfaces/dtos.py
@dataclass
class ReactionDTO:
    reaction_id: int
    post_id: int
    user_id: int
    reaction_type: ReactionTypeEnum

# interactors/add_reaction_interactor.py
@dataclass
class AddReactionInputDTO:
    user_id: int
    post_id: int
    reaction_type: ReactionTypeEnum

class AddReactionInteractor:

    def __init__(self, storage):
        self.storage = storage

    def add_reaction_wrapper(
        self,
        user_id: int,
        post_id: int,
        reaction_type: ReactionTypeEnum,
        presenter,
    ):
        try:
            reaction_dto = self.add_reaction(
                AddReactionInputDTO(user_id, post_id, reaction_type)
            )
            return presenter.get_add_reaction_response(reaction_dto=reaction_dto)
        except InvalidUserId:
            return presenter.raise_exception_for_invalid_user_id()
        except InvalidPostId:
            return presenter.raise_exception_for_invalid_post_id()

    def add_reaction(self, input_dto: AddReactionInputDTO):
        from fb_post.adapters.service_adapter import get_service_adapter
        adapter = get_service_adapter()

        adapter.auth.get_user(user_id=input_dto.user_id)  # validates user exists
        self.storage.validate_post_id(post_id=input_dto.post_id)

        return self.storage.add_or_update_reaction(  # returns ReactionDTO
            user_id=input_dto.user_id,
            post_id=input_dto.post_id,
            reaction_type=input_dto.reaction_type,
        )

5 The ib_users Package

ib_users is the internal package that manages all user accounts and profiles. Your app never queries its models directly — it calls ib_users.interfaces.service_interface.ServiceInterface.

Key method: get_user_profile

from ib_users.interfaces.service_interface import ServiceInterface

profile_dto = ServiceInterface().get_user_profile(user_id="abc123")
# Returns a GetUserProfileDTO

GetUserProfileDTO fields you'll use most often:

FieldType

last_nameOptional[str]
emailstr
phone_numberstr
profile_pic_urlstr
genderGender (enum)

Writing the adapter

Follow the same adapter pattern: define your own DTO in the adapter file, expose the ServiceInterface as a @property, convert the returned DTO.

# your_app/adapters/ib_users_adapter.py
from dataclasses import dataclass
from typing import Optional


@dataclass
class UserProfileDTO:           # your app's own DTO
    user_id: str
    name: str
    email: Optional[str]
    profile_pic_url: Optional[str]


class IbUsersServiceAdapter:

    @property
    def interface(self):
        from ib_users.interfaces.service_interface import ServiceInterface
        return ServiceInterface()

    def get_user_profile(self, user_id: str) -> UserProfileDTO:
        ib_profile_dto = self.interface.get_user_profile(user_id=user_id)

        return UserProfileDTO(
            user_id=ib_profile_dto.user_id,
            name=ib_profile_dto.name,
            email=ib_profile_dto.email,
            profile_pic_url=ib_profile_dto.profile_pic_url,
        )

Then plug it into your service_adapter.py:

class ServiceAdapter:

    @property
    def ib_users(self):
        from your_app.adapters.ib_users_adapter import IbUsersServiceAdapter
        return IbUsersServiceAdapter()


def get_service_adapter():
    return ServiceAdapter()

And call it from an interactor:

from your_app.adapters.service_adapter import get_service_adapter

adapter = get_service_adapter()
user_profile_dto = adapter.ib_users.get_user_profile(user_id=user_id)

DB entries that must exist

get_user_profile reads from two tables. Both rows must exist or it raises an exception.

WhatTableHow to create (locally)
User Account ib_users_useraccount Django Admin → Ib Users → User Accounts → Add
or ServiceInterface().create_user_account_with_phone_number(...)
User Profile ib_users_userprofile Django Admin → Ib Users → User Profiles → Add (link to the User Account above)
or ServiceInterface().create_user_profile(user_id, CreateUserProfileDTO(...))
Common local testing mistake
Creating a User Account is not enough — get_user_profile also needs a User Profile row. If only the account exists, the call raises UserProfileDoesNotExist.

6 Authentication

DSU handles auth for you — you just need to declare it in two places: the API spec and the api_wrapper. DSU then validates the Bearer token before your code ever runs.

Step 1 — Declare securityDefinitions in the API spec (once per app)

Add this at the top level of api_specs/api_spec.json, alongside paths:

{
    "swagger": "2.0",
    ...
    "securityDefinitions": {
        "oauth": {
            "tokenUrl": "http://auth.ibtspl.com/oauth2/",
            "flow": "password",
            "scopes": {
                "read":   "read",
                "write":  "write",
                "update": "update",
                "delete": "delete"
            },
            "type": "oauth2"
        }
    },
    ...
}

This is boilerplate — copy it once and forget it. It tells Swagger UI how to acquire a token.

Step 2 — Add security to each protected endpoint

Inside the path definition, add a "security" key. Use "read" for GET endpoints, "write" for POST/PUT/DELETE.

"/user/fresh_chat/v1/": {
    "get": {
        "operationId": "get_user_fresh_chat_details",
        "summary": "Get user fresh chat details",
        "security": [{"oauth": ["read"]}],
        "parameters": [...],
        "responses": {...}
    }
}

// POST endpoint:
"/post/v1/": {
    "post": {
        "operationId": "create_post",
        "security": [{"oauth": ["write"]}],
        ...
    }
}
Public endpoints (login, refresh token)
Omit the "security" key entirely. DSU will not require a token for that endpoint.

Step 3 — Read user_dto in api_wrapper

Once "security" is in the spec, DSU validates the token and injects a user_dto into kwargs. You extract the user ID like this:

@validate_decorator(validator_class=ValidatorClass)
def api_wrapper(*args, **kwargs):
    user_id = str(kwargs['user_dto'].user_id)   # ← authenticated user
    request_data = kwargs['request_data']
    ...

The ValidatorClass also receives user_dto, user, and access_token — available as self.user_dto etc. if you need them for custom validation.

Step 4 — Create a token from Django Admin (for local testing)

You need a valid OAuth2 access token to call authenticated endpoints in Swagger UI or via curl locally.

  1. Open Django Admin at http://localhost:8000/admin/
  2. Navigate to Django OAuth Toolkit → Access tokens
  3. Click Add — select a User, set Token to any string (e.g. testtoken123), set Expires to a future date, leave Scope as read write
  4. In Swagger UI, click Authorize → enter Bearer testtoken123
  5. Or in curl: -H "Authorization: Bearer testtoken123"
What DSU does behind the scenes
When your endpoint has "security": [{"oauth": [...]}], DSU reads the Authorization: Bearer <token> header, looks up the token in the oauth2_provider_accesstoken table, resolves the owner User, and packs it into kwargs['user_dto'] before calling your api_wrapper. If the token is missing or expired, DSU returns 401 automatically — you write zero auth logic.
