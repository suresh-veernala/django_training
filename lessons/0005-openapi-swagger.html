1. What is OpenAPI?

OpenAPI (formerly Swagger) is a standard, language-agnostic format for describing a REST API in a single document (YAML or JSON). It is the contract for your API: every endpoint, parameter, request body, and response shape, written down. See the OpenAPI 3.0 spec.

OpenAPI vs Swagger — clearing the confusion
OpenAPI = the specification (the standard). Swagger UI = a tool that reads an OpenAPI document and renders interactive, try-it-out documentation in the browser.

Why document an API?

Shared contract

Frontend and backend agree on shapes before code is written.

Interactive docs

Swagger UI lets anyone test endpoints in the browser.

Codegen

Generate client SDKs and server stubs from the spec.

Single source of truth

One file describes the whole API surface.

2. The OpenAPI Document Structure (Swagger 2.0 style)

Based on your sample, this is the Swagger 2.0 structure you'll see in your codebase:

KeyPurpose
swaggerSpec version, e.g. 2.0
host + basePathAPI host and base URL prefix
schemesProtocol list (http / https)
consumes / producesRequest and response content types
infoAPI title, version, description
securityDefinitionsAuth schemes (oauth2, apiKey, etc.)
securityGlobal/default security requirements
pathsThe endpoints — the heart of the spec
definitionsReusable schema models
parametersReusable parameter objects
responsesReusable response objects

In Swagger 2.0, request body payloads are described via parameters with in: body (not requestBody).

3. A Complete Spec for the IMDb API

This version follows your sample style (Swagger 2.0 with definitions, reusable parameters, and reusable responses).

{
  "swagger": "2.0",
  "host": "127.0.0.1:8000",
  "basePath": "/api/imdb/",
  "info": {
    "version": "1.0.0",
    "title": "IMDb API",
    "description": "Movie and actor APIs"
  },
  "schemes": ["http"],
  "consumes": ["application/json"],
  "produces": ["application/json"],

  "securityDefinitions": {
    "apiKey": {
      "type": "apiKey",
      "in": "header",
      "name": "x-api-key"
    }
  },
  "security": [],

  "definitions": {
    "Movie": {
      "type": "object",
      "properties": {
        "id": { "type": "integer" },
        "title": { "type": "string" },
        "release_year": { "type": "integer" },
        "rating": { "type": "number" }
      },
      "required": ["id", "title", "release_year"]
    },
    "Actor": {
      "type": "object",
      "properties": {
        "id": { "type": "integer" },
        "name": { "type": "string" }
      },
      "required": ["id", "name"]
    },
    "CreateMovieRequest": {
      "type": "object",
      "properties": {
        "title": { "type": "string" },
        "release_year": { "type": "integer" },
        "rating": { "type": "number" }
      },
      "required": ["title", "release_year"]
    },
    "ErrorResponse": {
      "type": "object",
      "properties": {
        "response": { "type": "string" },
        "http_status_code": { "type": "integer" },
        "res_status": { "type": "string" }
      },
      "required": ["response", "http_status_code", "res_status"]
    }
  },

  "parameters": {
    "MovieIdPathParam": {
      "name": "movie_id",
      "in": "path",
      "required": true,
      "type": "integer",
      "description": "Movie ID"
    },
    "CreateMovieBodyParam": {
      "name": "CreateMovieBodyParam",
      "in": "body",
      "required": true,
      "description": "Payload to create a movie",
      "schema": {
        "$ref": "#/definitions/CreateMovieRequest"
      }
    }
  },

  "responses": {
    "GetMovieSuccessResponse": {
      "description": "movie details success response",
      "schema": { "$ref": "#/definitions/Movie" }
    },
    "GetMovieActorsSuccessResponse": {
      "description": "movie actors success response",
      "schema": {
        "type": "array",
        "items": { "$ref": "#/definitions/Actor" }
      }
    },
    "CreateMovieSuccessResponse": {
      "description": "create movie success response",
      "schema": { "$ref": "#/definitions/Movie" }
    },
    "GenericFailureResponse": {
      "description": "failure response",
      "schema": { "$ref": "#/definitions/ErrorResponse" }
    }
  },

  "paths": {
    "/movies/{movie_id}/": {
      "get": {
        "summary": "Get movie details",
        "operationId": "get_movie_details",
        "security": [{ "apiKey": [] }],
        "parameters": [
          { "$ref": "#/parameters/MovieIdPathParam" }
        ],
        "responses": {
          "200": { "$ref": "#/responses/GetMovieSuccessResponse" },
          "404": { "$ref": "#/responses/GenericFailureResponse" }
        }
      }
    },
    "/movies/{movie_id}/actors/": {
      "get": {
        "summary": "Get actors for a movie",
        "operationId": "get_movie_actors",
        "security": [{ "apiKey": [] }],
        "parameters": [
          { "$ref": "#/parameters/MovieIdPathParam" }
        ],
        "responses": {
          "200": { "$ref": "#/responses/GetMovieActorsSuccessResponse" },
          "404": { "$ref": "#/responses/GenericFailureResponse" }
        }
      }
    },
    "/movies/": {
      "post": {
        "summary": "Create a movie",
        "operationId": "create_movie",
        "security": [{ "apiKey": [] }],
        "parameters": [
          { "$ref": "#/parameters/CreateMovieBodyParam" }
        ],
        "responses": {
          "201": { "$ref": "#/responses/CreateMovieSuccessResponse" },
          "400": { "$ref": "#/responses/GenericFailureResponse" }
        }
      }
    }
  }
}
$ref = don't repeat yourself
Define reusable objects once (in definitions, parameters, or responses) and reference them with $ref. This keeps large Swagger files maintainable.

5. Rendering with Swagger UI

Feed the document above into editor.swagger.io and it renders interactive docs: expandable endpoints, a "Try it out" button, example payloads, and response schemas.

In a Django project

Keep your API spec in a versioned file (for example swagger.json), and load it in Swagger UI. Teams usually review this file in pull requests just like code.

# Example workflow
# 1) update swagger.json when endpoint/request/response changes
# 2) open Swagger UI and point it to swagger.json
# 3) verify request/response examples and status codes
Hands-on goal
In the hands-on session you'll hand-write a Swagger/OpenAPI document for your IMDb endpoints, then render and test it in Swagger UI.
