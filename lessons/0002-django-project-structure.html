1. Project vs App — the key distinction

This is the single most important idea in this lesson. Django splits your code into two levels:

ProjectApp
WhatThe whole website / configurationA single feature module
How manyOne per siteMany per project
Created bydjango-admin startprojectpython manage.py startapp
Exampleimdb_projectmovies, actors, accounts
ContainsSettings, root URLs, WSGI/ASGIModels, views, app URLs, migrations
Mental model
A project is the building. Apps are the rooms inside it — each one self-contained and ideally reusable in another building. The official tutorial reuses its "polls" app exactly this way.

2. What the files look like

Most common confusion
You usually see two folders with the same name. The outer one is your project root folder. The inner one is the Django configuration package (where settings.py and root urls.py live).
project root
imdb_project/ project root folder
├── manage.py CLI entry point — run, migrate, shell
config package (same name as project)
├── imdb_project/ Django config package
│ ├── settings.py apps, database, middleware, secret key
│ ├── urls.py root URL router — delegates to apps
│ ├── wsgi.py WSGI server entry point
│ └── asgi.py ASGI server entry point (async)
app — one per feature module
└── movies/ the "movies" feature app
    ├── models.py database tables as Python classes
    ├── views.py request → response logic
    ├── urls.py app-level URL routes
    ├── admin.py register models in the admin panel
    ├── apps.py app metadata & configuration
    ├── migrations/ versioned schema change history
    └── tests.py unit & integration tests

3. File-by-file walkthrough

manage.py

Your command-line Swiss army knife. Every command you run goes through it:

python manage.py runserver        # start the dev server
python manage.py startapp movies  # create a new app
python manage.py makemigrations   # generate migration files
python manage.py migrate          # apply migrations to the DB
python manage.py createsuperuser  # create an admin login
python manage.py shell            # interactive Python + ORM

settings.py

Central configuration: database connection, installed apps, middleware, secret key, timezone. When you create an app, you register it here:

# settings.py
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    # ...
    'rest_framework',   # third-party
    'movies',           # ← your app
]

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': BASE_DIR / 'db.sqlite3',
    }
}
Easy to forget
A new app does nothing until you add it to INSTALLED_APPS. Migrations and the admin won't pick it up otherwise.

urls.py (root vs app)

The root urls.py (in the config package) usually delegates to each app's own urls.py using include():

# imdb_project/urls.py  (ROOT)
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('movies/', include('movies.urls')),   # delegate to app
]

# movies/urls.py  (APP)
from django.urls import path
from . import views

urlpatterns = [
    path('<int:movie_id>/', views.movie_detail),
]

models.py

Defines your database tables as Python classes. This is the "M" in MVT. We go deep here in Module 3.

views.py

The logic that handles a request and returns a response. This is the "V" in MVT (the Controller in MVC terms). Covered in Module 4.

admin.py

Register a model here and Django gives you a full CRUD admin dashboard for free:

# movies/admin.py
from django.contrib import admin
from .models import Movie

admin.site.register(Movie)

apps.py

App-specific configuration (the app's name, default auto field). You rarely touch it early on.

migrations/

A version-controlled history of your schema. makemigrations writes files here; migrate applies them to the database. Think of it as "git for your DB schema." More in Module 3.

4. How a request flows through these files

  1. Request arrives → Django hands it to the root urls.py.
  2. Root urls.py matches a prefix and include()s the app's urls.py.
  3. App urls.py matches the rest of the path and points to a view in views.py.
  4. View runs logic, calls the ORM in models.py if it needs data.
  5. Model / ORM queries the database and returns Python objects.
  6. View builds a response (JSON for an API) and returns it to the client.
