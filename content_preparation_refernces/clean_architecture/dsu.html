          # Django Swagger Utils (DSU)

          Django Swagger Utils is an internal library built to ease application development.

          With DSU we can

          • Generate a boilerplate code for an app.
          • Generate serializers, URLs, and views automatically from OAS 2.0
          • Automate the scopes validation for each API
          • Ease request and response wrapping
          • Ease writing integrating tests for views
          • Ease handling API failures
          • Generate client libraries for python, java and mobx models easily
          • Generate documentation for OAS ... and many more.

          # Handling Multiple Settings Files

          Till now your Django project has a single settings.py file, but in a typical Django project, we have multiple settings files each having a different purpose.

          For example alpha.py, beta.py … are for alpha, beta environments respectively, and similarly base_pg_db.py storing all the database related settings.

          Below are the most commonly used settings files for development:

          1. local.py- Consists of all the settings required for local development, for example, it has database settings configured to sqlite3 for local development, etc.
          2. base_swagger_utils.py - Consists of all the settings required to configure the behavior of DSU ‌

          To specify Django, to use a specific settings file, we have to set an environment variable as mentioned below

           export DJANGO_SETTINGS_MODULE=<project_name>.settings.local

          Execute the above command in your terminal

          # Making APPS Discoverable

          1. Add the app name to APPS variable as below
          APPS = [
  "fb_post"

    "HOST": os.environ.get('APIGATEWAY_ENDPOINT', '127.0.0.1:8000'),

          # Understanding the generated files

          • create_post.py: This python file with the name same as OperationId is used to parse the request and prepares the response. It is automatically generated and you need to do any changes to it.

          • api_wrapper.py: Api Logic can be written in this file

          • request_response_mocks.py: Sample request and response can be viewed in this file

          • validator_class.py: Validations which need to be satisfied before executing our API can be written here

          # Working with the generated views

          # Control flow

          Let's understand how the control flows between these files.

          On request from the client, control flows as below:

          urls.py → view file → validator_class.py → api_wrapper.py

          1. Requested URL is matched to the listed patterns in urls.py and a function in the view file of the corresponding handles the request
          2. view file (create_post.py from our current example) does basic parsing and sends it to api_wrapper to execute business logic.
          3. api_wrapper function in api_wrapper.py file calls validator_class to validate the data in the request.
          4. The validated data is sent back to the api_wrapper function to execute business logic and return an appropriate response to the client.

          # Models in multiple files

          In a typical Django project, we need to define multiple models, which is ridiculous in maintaining all of them in one file called models.py which might cause circular import errors some times. So we can split them up into multiple files and all can be maintained in models folder

          For example, we can split our fb_post models into different files such as user.py, post.py, comment.py... then the directory structure would look as follows:

          | models
|   |-- __init__.py
|   |-- post.py
|   `-- user.py
|   `-- ....

          To make the models accessible to different files we need to define them in __init__.py with global scope as follows

          from .user import User
from .post import Post

__all__ = [
    "Post",
    'User'

]
