                    # Explanation

                    # adapters

                    • Write Adapter classes in this package i.e AuthServiceAdapter, ProductsServiceAdapter etc. Each class in a separate file.

                    • All the dtos related to the Adapter classes are written in dtos.py module in this package.

                    • All the individual Adapter classes are plugged into the ServiceAdapter class in service_adapter module.

                    # admin.py

                    • Django admin classes should be added to this module.

                    # api_specs

                    • OAS API specification of a Django app should be written in api_spec.json file.

                    # app_interfaces

                    • Interfaces that an app exposes to rest of the Django Apps in a project are written in ServiceInterface class in service_interface module of this package.

                    # constants

                    Constants package consists of the following modules:

                    • config: Consists of configurations i.e date time formats, number of hours after a form expires etc.
                    • enums: Define all the enums used by the app in this module.
                    • exception_messages: Define all the exception message in this module.

                    # exceptions

                    • Define all the custom exceptions used in the app should be defined in custom_exceptions module.

                    # interactors

                    • Define all the interactors should be defined in this package as mentioned in the Writing Interactors section. Along with the following packages

                      • mixins: Define all the mixin classes required for the interactors in this package following Writing Mixins section.
                      • presenter_interfaces: Define all the presenter interfaces and the required DTOs in this package following guidelines in Writing Presenters section.
                      • dto: Define all the DTOs that interactors define for their internal usage or for controller - interactor communication.
                      • storage_interfaces: Define all the presenter interfaces and the required DTOs in this package following guidelines in Writing Storages section.

                    # models

                    • Define all the Django models of an APP in this package.

                    # populate

                    • Define all the populate wrappers that wrap the required interactors here.

                    # presenters

                    • Define all the presenter implementations in this package.

                    # storages

                    • Define all the storage implementations in this package.

                    # API Specs

                    • While writing the api_spec, all possible exceptions raised by the API should be mentioned in the response with corresponding enums.

                    # Interactors

                    • If an entity requires ONLY plain crud operations (without logic), all such operations can be written as methods in a single interactor class. (instead of creating one class per operation)

                    • When an interactor B is called inside an interactor A (composition), all possible exceptions that are raised by the interactor B (the one being called) should be handled appropriately (with try-catch blocks) inside the 'wrapper method' of Interactor A.

                      • Exceptions raised from interactor B (the one being called) that are handled within the interactor A (in the method which calls B) need not be handled in the wrapper method again.

                    • When an interactor A inherits another interactor B, the return type of any method (especially public methods) which is overridden in the child class A must be the same as the one defined in the base/parent class B. When there are some additional keys that needs to be returned by a method in A, the return type DTO of that method should inherit the return type DTO specified in the corresponding method of parent class B.

                      • If inheriting the return type DTO of corresponding method in the base/parent class B isn't possible, then it could be a sign that interactor A shouldn't inherit interactor B in the first place.

                    • In case a new variable (created in the middle of a logical flow) needs to be passed to other private methods in an interactor, pass the variable as an argument. DON'T create a new instance variable for this purpose.

                    • If a method has too many arguments(i.e > 3), you can group related arguments into a dto.

                    • If an interactor needs sorted or filtered data from the storage, the interactor should send the filtering and sorting information as parameters to the storage in the below mentioned DTO formats.

                    • DTOs used by an interactor other than those related storage, presenter, adapter should be defined in the interactor file.

                      • Ex: DTOs which are used to communicate between different methods within an interactor should be defined in that interactor file itself.

                    # Mixins

                    • Try to keep the usage of mixins as minimum as possible (as they hamper code readability). Move a method to a mixin only if it is used in more than 3 interactors.

                    • If a Mixin class size grows over 200 lines, follow the SRP & OCP principles to split it into multiple classes.

                    # Presenters

                    • DTOs to dictionaries conversion logic should be written in Presenters. Interactors should only retrieve the required data, make necessary modifications (if needed), and send the data to the presenter (in DTOs format).

                      • Logic related to preparing the nested structures required by an API should also be written in the presenter.
                      • Presenter tests are equally important to test the conversion logic properly.

                    • Write one presenter class (Interface & Implementation) per API. Move the common methods which are used across multiple presenters to mixin classes. Presenter classes can inherit the required mixin classes.

                      • Note: All common methods need not go into one mixin class, follow SRP & OCP appropriately.

                    • DTOs specific to each presenter class, should be written in the interface file itself.

                      • If there are any DTOs that are used across several presenters, they should be added/moved to dtos.py file in the presenter_interfaces folder.

                    • Specific exception messages should be sent based on the context of the API call, even when the same enum is used to raise the exception.

                      • These exception messages should be defined at the top of the presenter implementation class.

                    • If an interactor needs to pass a storage DTO to the presenter without any modification, the DTO need not be duplicated in the presenter interface. (Trade-off is taken to avoid code duplication)

                    • The return type of a presenter interface method should be an HTTPResponse. With this, a consistent signature will be maintained across different type of presenters.

                    # Tests

                    • All the tests should be written in appropriate folders in tests folder in django app.

                    • All the snapshot tests should inherit the updated base class from DSU.

                    • Prefer class based tests.

                      • Don't repeat the class name in each of the test method name.
                      • Define fixtures which are used only to test a specific API/Interactor in the corresponding test class.

                    • Tests should be written using factories, fixtures & mocker wherever needed, appropriately.

                    • For an interactor with multiple public methods (example create_post_wrapper, delete_post_wrapper, update_post_wrapper), write a separate test file for each of the public method.

                    # Storages

                    • As the storage interface class grows, split it into multiple classes (and files) appropriately. Discuss with your RP / Domain Expert before making the changes.

                      • When splitting storage interfaces, the corresponding DTOs should also be split appropriately.

                    • If there are multiple storage interfaces in an app, the corresponding DTOs should also be split into multiple files in a similar structure.

                    # Adapters (Inter-App Communication)

                    • Write all the DTO conversions and exception handling related code in the adapter file. Also define the required DTOs in the adapter file itself.

                      • If there are any common DTOs across multiple adapters (>3), move them to the dtos.py file in the adapters folder.

                      • This avoids code duplication across multiple interactors which use the adapter.

                    • When app A calls an app B, DTOs returned by the app_interface of B should not be directly used in the app A. They should be converted (based on app A's structure and requirements) in the adapter of A before passing to the interactors.

                      • The conversion should be done even if app A needs a similar DTO structure. (to avoid accidental duplication)

                    • When app A calls an app B, exceptions raised by the app_interface of B should be handled in the adapter of app A. The adapter should then raise exceptions which are defined in app A (based on app A's structure and requirements).

                    # PopulateUtils

                    • The logic separation between the interactors and population utils should be such that, the interactor can be used for a create API call.

                      • Population utils should handle reading from sheets, format related validations, field validations (data types, enum validations, required checks, etc.) and DTO construction.

                    Note

                    Exception handling and Rollbacks should be handled appropriately in population utils.

                    # Common Python Code Guidelines

                    # Naming

                    • For factories & DTOs, the class name should include “Factory”, “DTO” respectively.
                      • variable names of objects instantiated using 'models' or 'model factories' should include "_object" at the end.
                      • Ex: Objects instantiated using Post(model), PostFactory should be named "post_object".
                      • variable names of objects instantiated using 'DTOs' or 'DTO factories' should include "_dto" at the end.
                      • Ex: Objects instantiated using PostDTO, PostDTOFactory should be named "post_dto".

                    # Raising Errors

                    • When some data needs to be included in an exception, it should be stringified and sent in the "message" key along with the message.

                    # Import Guidelines

                    • Local imports should be done with the import being as close to the usage as possible (in most of the cases).

                      • Increases readability
                      • Code refactoring will be easier
                      • Reduces scope for circular dependencies

                    • Global imports should be preferred only when the same import is being used in multiple methods.

                      • Ex: In test cases Storage Impl, Presenter Impl, Interactors have to be imported in all test methods. So, these imports can be done globally.

                    • Standard python packages should be imported globally (one per line).

                      • Ex: os, uuid, datetime, json

                    • Absolute import paths should be used (no relative imports).

                      • Ex: from myapp.interactors.presenter_interfaces.presenter_interface import PresenterInterface is preferred over from .presenter_interfaces.presenter_interface import PresenterInterface

                    • When importing third party packages, import only till module. Don't import the classes/methods directly.

                      • Use import x for importing packages and modules.
                      • Use from x import y where x is the package prefix and y is the module name with no prefix.
                      • Use from x import y as z if two modules named y are to be imported or if y is an inconveniently long name.
                      • Use import y as z only when z is a standard abbreviation (e.g., np for numpy)
                      • Reference: https://google.github.io/styleguide/pyguide.html#23-packages

                    # Typing guidelines

                    • All method signatures (input args & return) must include the typing information. This applies to private methods as well.

                    • Instance variables of a class must have the type specified.

                    • Typing should not be used for local variables (causes overhead & increases the line lengths)

                    # Writing Enums

                    • While writing Enums in constants/enums.py, should be properly organized i.e write all the related Enums together, to ease developers to find a particular enum is defined or not.

                    • If the number of enums in constants/enums.py are increasing, organize them into multiple files.

                      • All related enums should be grouped into a single file.
                      • Within a file, related enums should be written close to each other. (for better readability).

                    # Maintaining Utils

                    • Keep the code in the utils module as minimal as possible.

                    • If there are any common methods (used in more than 3 classes/files) that are being used only within a module (say interactors), write those common methods in a utils module inside the interactors folder itself.

                    # Refactoring Guidelines

                    • As the size of the interface grows, Storage interface class should be divided into multiple interface classes.

                      • Division should be done foreseeing the changes in the underlying datastore.

                    • Split the interactor when the size of a single interactor class grows. Use the 'number of lines' in the class as a checkpoint. Splitting should be done based on SRP and OCP principles.

                      • If there are multiple public methods in an interactor class, they are potential candidates to separate into individual classes.
                      • Composition/Inheritance/Mixin patterns should be used to refactor a large interactor class with only one public method.

                    • As the custom_exceptions file grows in size, related exception classes can be split into separate modules (possibly based on entities)

                    • Common functionality (not accidental duplication) in child classes should be moved to the parent class.

                      • If any one of the child classes need a different behaviour for the same method, it can override/extend the parent class method.

                    • General guideline to follow: If a particular file/class size increases, split it appropriately and move the files/classes into a module (folder).

                    • Related views should be grouped into subfolders.
