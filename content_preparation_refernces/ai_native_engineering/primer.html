Skills

Skills are domain knowledge files the agent loads on-demand before working on a specific area of your codebase. Each skill documents the patterns, conventions, and rules for a particular concern — how to write an interactor, how to structure a DTO, how to handle datetime values.

Skills are not injected into every request — they are loaded only when the task touches the relevant domain, keeping the context window focused.


Commands

Commands are slash-command recipes that define the agent's goal and process for a specific type of task. When you invoke a command, the agent switches into a structured mode — it knows what kind of thinking to do, what steps to follow, and what the output should look like.


Rules

Rules are always-apply/pattern based guidelines that are injected into every agent session. They define the core behavior that must hold regardless of the task — how to use tools, what scope to stay within, naming conventions to follow, and how to interact with you as the developer.

Unlike skills (loaded on-demand) and commands (invoked explicitly), rules are passive. The agent never has to be told to follow them — they are always present. This makes them the right place for non-negotiable standards that should never be bypassed.


Hooks

Hooks are shell scripts that fire at specific lifecycle events during an agent session — before a shell command executes or before a tool is used. They act as hard guardrails that enforce constraints the agent cannot bypass, even if instructed to.

The difference between hooks and rules is enforcement: rules guide the agent's behavior through instructions, hooks enforce constraints through code. A hook can physically block a destructive command from running, prevent writes outside the workspace, or check code patterns before a push.

Examples: blocking rm and destructive shell commands, blocking writes outside the project, preventing pushes to protected branches, checking for anti-patterns before commit


Sub-Agents

Sub-agents are spawned by the main agent to handle isolated, parallelizable work. When a task involves deep research across independent areas — like scanning multiple parts of the codebase simultaneously — the main agent can delegate each area to a sub-agent and get back focused results without doing all the work sequentially.

Sub-agents run in their own context with a fresh scope. They do not share the main session's conversation history and only return the specific output the main agent asked for. This keeps the main context lean while still covering broad ground.

Sub-agents are used sparingly — only when parallel research genuinely saves time. For most tasks, the main agent handles everything directly.


MCPs (Model Context Protocol)

MCPs are external tool integrations that extend what the agent can do beyond reading and writing code. They give the agent access to live data and external systems — like capturing Figma design screenshots, navigating a browser, or querying an issue tracker — when the task requires it.

MCPs are not always-on. Enabling all MCPs in every session consumes context window tokens with tool definitions that may be irrelevant to the current task. The right approach is to enable only the MCPs relevant to what you're working on.
