    About

    Prompt engineering is the process of crafting effective instructions or inputs to guide AI models to produce the output you want.

    There is no fixed syntax like in programming languages, the model will respond as long as the requirement is stated properly. However a well framed prompt directly impacts the quality, consistency, and reliability of the output.

    While models are getting better at understanding vague or casually written prompts by guessing intent, a well-structured prompt significantly increases the chances of getting correct and usable results especially in complex or layered scenarios.

    You may sometimes get the expected result with a minimal or informal prompt, but a well-formed prompt:

    • Increases consistency across multiple runs
    • Works more reliably across different models and versions
    • Reduces guesswork and misinterpretation
    • Helps you iterate faster with fewer corrections

    # Best Practices

    Note: A More Effective Prompt often uses one or more of these techniques depending on the situation.


    Be Explicit and Descriptive in Instructions

    Clearly State the Requirement or Problem

    • The model doesn’t know your codebase, your domain logic, or your edge cases unless you describe them. Unlike a teammate who’s read the design doc or has historical context, the model only sees what you explicitly include in the prompt.

    • When describing a requirement or problem, don’t just state what needs to be added or fixed but also explain what is the change for or why the issue happens.

    Example 1

    # ❌ Less Effective Prompt

    Add concurrency support for this storage method.

    This prompt is technically valid, but it leaves too much unsaid. What kind of concurrency problem? What's the business rule being violated? What does "support" mean locking, atomic writes?

    # ✅ More Effective Prompt

    In this storage method, when multiple reactions are added to the same discussion with no time gap, an integrity error is thrown due to a unique constraint.

The method should update the existing reaction if it already exists. can lock row if needed.

    This version includes:

    • What is happening: Integrity error from concurrent writes
    • When it happens: Multiple reactions added simultaneously
    • What should happen instead: Update instead of insert
    • How to approach it: Use locking or transaction handling

    By clearly stating the technical and business context, the model can now generate a useful, correct solution that fits into your actual systemnot a generic guess.


    Include Step-by-Step Instructions and Give Direction of Changes When Applicable

    • When your prompt involves a non-trivial implementation like model changes, feature additions, or integration work then it's important to go beyond “what” and specify how and where.

    • The model doesn't have awareness of your service boundaries, file structure, or architecture decisions unless you explain them. A More Effective Prompt should direct the model on what you expect across different layers like storage, DTOs, interactors, presenters etc.

    Example 1

    # ❌ Less Effective Prompt

    Add a new field `location` in the `UserProfile` model. Make it optional by default.  
Make changes in all the files that are using the data from the model and return data in the APIs.  
Fix tests also of all files.

    This is a reasonable request but it’s far too open-ended. It doesn’t specify how data flows through the system, which layers need updates, or how the field should behave in different contexts (e.g., optional in DB, required in DTO, etc.)

    # ✅ More Effective Prompt

    Add a new field `location` in the `UserProfile` model. Make it optional by default.  
Make changes in all the files that are using the data from the model and return data in the APIs.

1. Add the field in sheet util and take it as optional
2. Update the create user profile interactors code
3. Update the storage queries to save and return the data
4. Update the DTOs to include this field  but don’t make it optional in the DTO
5. Return the newly added key in the presenters of APIs that return user profile data

    Why this works:

    • It gives clear indication over layers where changes are required even though its on overview level.
    • It communicates how the data should behave in different parts of the system (optional in DB, required in DTO)
    • It ensures that model make changes in all the applicable files from loading to apis.

    This lets the model reason about the structure of your project, and respond with targeted, relevant code rather than scattered guesses.


    Use Clear Categorization in Long Contexts

    • When giving instructions to the model, remember that along with your requirement a lot of other guidelines will be dumped to the model as system instructions.

    • Adding lightweight categorization with clear labels like sections or tags helps the model focus on each aspect independently and reduces mistakes.

    • While the content might be correct, it becomes harder for the model to reason through it clearly. It might hallucinate on every requirement in the midst of all the other instructions on very long prompts.

    Example 1

    # ❌ Less Effective Prompt

    Implement a sheet util to insert interview data for the ai_interview app. The util should process a sheet with fields like interview_id, org_id, title, max_attempts_allowed, is_video_recording_enabled, is_proctoring_enabled, and a few others. Add validations like UUID format for IDs, max_attempts_allowed should be greater than 0, and reject duplicates. If the org_id doesn’t exist or the interview_id already exists, it should fail. Update the interactor if needed and modify the storage interface or implementation only if new methods are required. Don’t use placeholders, follow the conventions from the sheet util guide, and make sure everything is production-ready. Start with the sheet util and handle each layer cleanly.

    This kind of prompt dumps everything together, making it difficult for the model to separate concerns or follow logical structure.

    # ✅ More Effective Prompt

    Implement a sheet util to upsert interview data.

App Name: ai_interview

Input Format:
Sheet columns (in order):  
- interview_id (required)  
- org_id (required)  
- max_attempts_allowed (required, must be > 0)  

Validation Rules:
- interview_id and org_id must be valid UUIDs  
- max_attempts_allowed must be > 0  

Additional Context:
- can use same dto for both create and update data

    Why this works:

    • Adds clear labels and sections like Input Format, Validation Rules, and Context
    • Helps the model separate logic and responsibilities instead of interpreting a wall of text
    • Makes it easier to maintain and extend later by clearly isolating each requirement
    • Reflects how tasks are scoped in real-world teams, reducing confusion in layered changes

    Using lightweight tags like , or structured blocks helps the model make fewer mistakes and produce cleaner, layered code.


    Focus on One Task at a Time

    Break Problems into Smaller Tasks

    • Treat the model like a capable, eager intern who knows how to write code but needs guidance on what to build, how to structure it, and which part to focus on first.

    • Don’t throw the entire feature in one go and expect the right output. Break the problem into smaller sub-tasks ideally file-by-file or layer-by-layer so that the model can implement each part independently and cleanly. This makes debugging easier and avoids mistakes propagating across files.

    Example 1

    # ❌ Less Effective Prompt

    Write a sheet util which creates data in the UserProfile model. Fields in the sheet are user_id and profile_url.  
Implement all the files necessary till creation and add tests.

    This is too large in scope. It asks for:

    • A sheet parser
    • Interactors to validate and call business logic
    • Storage code to persist data
    • Tests for all of the above

    The model will likely skip validations, miss integration boundaries, or bundle logic in the wrong place.

    # ✅ More Effective Prompt

    Write a sheet util which creates data in the UserProfile model. Fields in the sheet are user_id and profile_url.  
First, let's write only the sheet util.

Here we will attach the sheet util implementation guide and what interaction and storage method it should create.  
But doing file-wise will ensure any mistake happening in one file will not affect others  we will review and fix it before moving to the next file.

    Why this works:

    • It narrows the task to just one file or one unit of work
    • It clarifies what to expect in terms of output
    • It leaves room to review, validate, and iterate, just like in a real team code review

    Use this pattern when asking for anything multi-layered: start with one part (e.g., util, interactor, storage, DTO, test), and move to the next only after validating the last.


    Step By Step Implementation

    • When a requirement requires edits across multiple files or involves complex business logic asking the model to do it directly in one go often may lead to incorrect results.

    • Similar to how a we break down a feature into smaller subtasks, asking the model to break the requirement to sub tasks or asking it to implement step by step will have the higher chance to get better results.

    Example 1

    # ❌ Less Effective Prompt

    Update the sheet util to support user_role. Add validations and changes across files as required.

    The model is expected to decide the structure and flow on its own this often leads to missed layers, shallow validation, or tightly coupled code across files.

    # ✅ More Effective Prompt

    We need to update the sheet util to support user_role.  
First, give a step-by-step plan breaking the task into sub-tasks.

    This puts the model in a mode where it thinks in smaller scopes and executes cleanly across layers just like a developer would.


    Example 2

    # ❌ Less Effective Prompt

    Implement soft delete for projects and make sure deleted ones don’t appear in any API response.

    This leaves it up to the model to decide what files to change and how it might miss query filters, affect unrelated APIs, or skip updating presenter layers.

    # ✅ More Effective Prompt

    We want to add soft delete support for the Project model.  
Give a step-by-step breakdown of what needs to change across model, storage, interactor, and presenter.

    In next step can ask the model to implement each step in the logical order.

    Why this works:

    • It mimics real-world workflows, where tasks are decomposed into sub-tasks
    • Reduces risk of missing key parts or introducing tightly coupled logic
    • Each sub-task is easier to review, fix, or adjust independently
    • It makes the model to a follow a structured execution giving you better control over multi-layered implementation

    Understand Before You Implement

    Plan Before You Implement

    • Just like how jumping straight into implementation without thinking leads to messy code or rework, the same applies when working with AI models.

    • If you immediately ask the model to "write the code," it might miss fine grained details like where to make the changes or scope of changes etc.

    • Instead, give the requirement first, and then ask the model to generate a plan before writing any code. This gives you a chance to review its understanding, correct any assumptions, and keep the implementation aligned.

    Example 1

    # ❌ Less Effective Prompt

    Update the sheet util to support max_focus_change_warnings_count and is_proctoring_enabled.

    # ✅ More Effective Prompt

    Update the sheet util to support max_focus_change_warnings_count and is_proctoring_enabled.  
First present me the plan without code changes on how you will implement this.

    Why this works:

    • It encourages structured thinking and avoids premature coding
    • You get a chance to validate assumptions before any implementation begins
    • It reduces back-and-forth caused by misunderstandings or missing details

    Ask for Analysis and/or Solution

    • Not every time you need the model to do some action on behalf of you or write code.
    • Sometimes you need to analyze or debug an issue, or brainstorm a solution for a requirement considering the current flow.
    • In those scenarios, asking the model to analyze or plan will help in controlling the model’s default behaviour of jumping directly into code or implementation.
    • Instead, when facing a bug, regression, or design ambiguity, explicitly ask the model to analyze the cause or propose a plan.
    Example 1

    # ❌ Less Effective Prompt

    I am getting a KeyError in this interactor. This is the traceback. Fix it.

    # ✅ More Effective Prompt

    I am getting a KeyError in this interactor. This is the traceback.  
First identify what could be the root causes of it and tell me a plan on how to fix it.

    Why this works:

    • It prevents the model from jumping into blind fixes or assumptions
    • Encourages it to go through a cause-and-effect analysis process
    • Mimics real-world debugging or pair programming approach
    • Helps you stay in control of the problem-solving flow

    This will help the model reason through the issue first similar to how you would pair with someone during a debugging session or planning.


    Model Context Handling

    Provide right context to model.

    • When you're asking the model to help with a specific task, avoid dumping full files, unrelated configs, or environment details unless they are directly needed.

    • Give just enough context for the model to reason accurately. If you give too much, the model might get distracted or make incorrect assumptions. If you give the wrong kind of context, it will solve the wrong problem.

    Example 1 - Too Much Context

    # ❌ Less Effective Prompt

    Add new field profile_pic in user model and make changes in the get profile interactor

    here the entire auth directory is added into context

    • Might hit model context limits and useful info might be truncated in the model request.

    # ✅ More Effective Prompt

    Add new field profile_pic in user model and make changes in the interactor

    here only the get profile interactor file is added into context

    Why this works:

    • It limits the input to only the relevant file(s) involved in the change
    • It avoids overwhelming the model with unrelated code from the rest of the module
    • It keeps the focus tight so the model can reason specifically about the interactor logic

    Example 2 - Irrelevant Context

    # ❌ Less Effective Prompt

    We need to update the sheet util to support a new field `is_proctoring_enabled`.  
FYI, the app is hosted on AWS Lambda, and we use Postgres, Redis, and Celery  plus we recently upgraded Django.

    here given example of get api instead of another update interactor

    # ✅ More Effective Prompt

    We need to update the sheet util to support a new field `is_proctoring_enabled`.  
This field should be parsed from the sheet (optional), passed through the interactor, and saved in storage.

    here given example of another update interactor as reference

    Why this works:

    • It keeps the example focused on a similar type of implementation, rather than shifting to unrelated patterns
    • It guides the model toward the correct type of change (update flow → update reference)
    • It avoids introducing noise that can confuse the model’s intent matching

    Always provide just enough input to focus the model's attention on the actual task. nothing more, nothing less.


    Having Project-Specific Guides for Conventions & Implementations

    • Preparing a guide on common patterns and implementation practices followed in your project helps in getting more accurate and aligned outputs from the model.

    • Even when you attach another file as a reference, the model might not always extract the right patterns unless those conventions are clearly documented or explicitly mentioned.

    Example 1

    # ❌ Less Effective Prompt

    Write a test for this interactor. Take reference from existing codebase.

    Assume no reference files or guidelines is attached in context

    The prompt doesn’t include any reference to the project’s testing style, naming patterns, or structural expectations. The model may generate something syntactically correct but misaligned with your actual codebase.

    # ✅ More Effective Prompt

    Write a test for this interactor. Take reference from existing codebase and implement as per the guidelines.

    Assume the other test files or guidelines is attached in the context

    Why this works:

    • It tells the model to align with your existing structure instead of guessing
    • Reduces inconsistencies across test implementations
    • Reinforces the use of shared guides or reference patterns
    • Helps scale consistent code generation as your project grows

    This works because it tells the model to align with established conventions either through a shared guide or an existing example which makes the output more consistent with your codebase.


    The Model Only Knows What You Tell It

    • The model only has access to what you send in the input. It doesn’t know your project structure, file layout, internal naming conventions, or why a particular rule exists unless you include that information explicitly.

    • Everything outside of the current prompt is a black box to the model.

    • So if you assume “it should already know this” or “it can figure that out from earlier runs,” the output will drift or be incomplete.

    • Always give the minimal context it needs to reason correctly nothing more, nothing less.


    Start New Chats When the Session Becomes Too Large

    • As the chat session grows, the model may start to lose track of earlier details or carry over unintended context from previous prompts.

    • To avoid confusion, stale assumptions, or degraded output quality, start a new chat when:

      • You’re switching to a new feature or module
      • You notice the model making incorrect assumptions based on earlier context
      • The responses become inconsistent or less focused

    Resetting the session ensures the model only works with the relevant context you provide, keeping outputs clean and scoped.


    Be Aware of the Model's Context Limits

    • Every model has a limit to how much input and conversation history it can retain and reason about in a single session this is called the context window.

    • If your input is too long or if you’ve had an extended chat, earlier parts of the conversation may be partially or fully dropped. This can cause the model to lose key assumptions or generate incomplete answers.

    • Being aware of the context limits of a particular model and mode helps you:

      • Scope your prompts more effectively
      • Decide when to restart the session
      • Prioritize what context to include and what to leave out

    Iterate and Refine

    Verify Model Understanding Before Proceeding

    • When you feel your input may not be detailed or structured enough, don’t immediately ask the model to proceed with implementation.
    • Instead, ask the model to explain what it has understood about the task so far.
    • This gives you a chance to verify its interpretation, correct assumptions early, and avoid wasted iterations or incomplete outputs later.

    Use this approach especially when:

    • You’re dealing with a slightly ambiguous task
    • You’re not sure if your context was scoped properly
    • You’re working across multiple files or layers and need alignment first

    # Example Prompt:

    We need to update the sheet util to support `is_proctoring_enabled`.  
Before you start, explain what you’ve understood about the task.

    Why this works:

    • It turns the model into a collaborative partner instead of a passive executor
    • Lets you debug the prompt, not just the output
    • Helps avoid going too far in the wrong direction
    • Reduces rework across large, layered implementations

    Reset When Iteration Fails

    • Most often, if the model assumes something incorrectly and continues to repeat the same mistake even after you've tried refining or correcting it, it's better to start a new chat.

    • Some assumptions get “locked in” due to previous messages, and trying to override them mid-session can lead to confusion or wasted time.

    Starting fresh helps:

    • Clear any misleading context
    • Let you reframe the task with clean intent on what model/your requirement missed previously
    • Improve the chances of getting accurate and unbiased output
