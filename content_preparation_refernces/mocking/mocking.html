            # Mocking

            # Introduction

            To build robust applications, tests are essential to verify the correctness, reliability and efficiency of the application logic. However, the value of tests depends on how well they demonstrate a given scenario. Obstacles such as complex logic and unpredictable dependencies make writing valuable tests difficult. In this section, you will learn how to handle such scenarios.

            # Why Mocking?

            How would you test the below function?

            def get_share_price(company_id):
    import requests
    import json
    
    response = requests.get( ....)
    api_response = json.loads(response.content)
    
    return api_response['share_price']

            You might write a test like the below one, where today_share_price is the actual share price of the company "Test_Co", today.

            def test_get_share_price():
    today_share_price = 101.23
    
    share_price = get_share_price("Test_Co")
    
    assert today_share_price == share_price

            Here, 101.23 is today's share price of "Test_Co" which is hard-coded into the testcase. Luckily, your tests might pass now. But they will not pass forever.

            Below are some of the problems in writing such tests.

            # Problems

            • As the code makes HTTP requests to external services, your tests execute predictably only as long as the services are behaving as you've expected them to. (i.e, until the share price of the company changes)
            • A temporary change in the behavior of these external services can cause intermittent failures within the test suite.
            • An HTTP call in a testcase makes the test suite much slower (as it involves a network round trip time).

            In the above test, we are reading data from an external source. What if we were supposed to write/send data to an external source? If we follow a similar pattern, every time we execute our tests, we would be updating the share prices to an inappropriate value. Clearly, it's not a desired behaviour to write values to external sources when a test suite is run.

            The underlying pattern in all of the above problems is that we aren't using a controlled test environment. That is, we cannot control the behaviour of external sources (in this case, data in the http response). So, how do we overcome this? We use mocking in such scenarios.

            # What is Mocking?

            Mocking is primarily used in unit testing, where we want to test the methods of a class (unit) in isolation. But, usually, classes are not isolated. They might use methods from other clases and also some external services when required. So, to isolate the behavior of the class (a unit in general) which we want to test, we mock the methods from other classes and external services. That is, we simulate the real behavior of them using some mocking frameworks. In short, mocking is creating methods/objects that simulate the behavior of real methods/objects.

            # Benefits of Mocking

            # Speed

            The first reason to use mocks is speed. A mock of a method or an external service would cut down unnecessary resource(network, disk, etc.) usage during testing. It reduces the test run time significantly.

            # Avoiding Dependencies

            When some components aren't ready yet, you might not be able to write tests for another component which depends on the first component.

            For instance, if you have a function A that depends on a function B, but function B has not been implemented at the moment. In such cases, you can simply mock function B that simulates it's expected behavior.

            In this way, you can write unit-tests even without having a fully implemented system.

            # Availability

            When a unit depends on a service, it often happens that services from third parties crash or are not available to the application for a period of time.

            For example, the first time you implement a unit-test which uses an external service, it might work fine. But, the next morning you come in and the service was not available and the unit-tests which depended on it fails.

            To avoid such scenarios, you can mock the service call and simulate it's behavior.

            # Reproducible states

            When you require database(or an external webservice for that matter) calls in test cases, it’s quite hard to reproduce the same result (which might require the same state of database) for each test every time you run it.

            Typically, we use one database for the whole system to test on. The state of the database might change with each test and there’s no guarantee that the tests are run in the same order each time you run them. This requires developers to write code to restore the database state for each test. This is not only difficult to do, but can also cause technical issues with a failing 'database restore' or a database server that is 'out of disk space'.

            # Code coverage

            Sometimes, it is difficult to test certain areas of your codebase. Such areas include except blocks and conditional blocks that are hard to reach in normal scenarios. Using mocking can help you control the execution path of your code to reach these areas and improve your code coverage.
