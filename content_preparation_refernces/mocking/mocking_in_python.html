            # Mocking in Python

            We use a python library unittest.mock for mocking. It provides an easy way to introduce mocks into the tests.

            # The Mock Object

            • unittest.mock offers a base class for mocking objects called as Mock. Mock is very flexible and can be used in many scenarios.

            Let's look at some examples to understand the usage of Mock. Consider the following is_weekend function implementation.

            • calendar.py
            # my_calendar.py
class MyCalendar:
    
    @staticmethod
    def get_current_day():
        import datetime
        now = datetime.datetime.now()
        return now.isoweekday()
            • is_weekend.py
            # is_weekend.py
def is_weekend_using_calendar(calendar_obj):
    day_as_number = calendar_obj.get_current_day()
    return day_as_number in [6, 7]

            How would you test the is_weekend_using_calendar function? You might write the following tests.

            
from is_weekend import is_weekend_using_calendar
from my_calendar import MyCalendar


def test_is_weekend_using_calendar_on_weekend():
    calendar_obj = MyCalendar()

    result = is_weekend_using_calendar(calendar_obj)

    assert result is True


def test_is_weekend_using_calendar_on_weekday():
    calendar_obj = MyCalendar()

    result = is_weekend_using_calendar(calendar_obj)

    assert result is False

            Clearly, atleast one of the tests fail on any given day. This is because we are not testing the is_weekend_using_calendar function in isolation. It depends on the get_current_day method which returns differntly based on the day we run tests. Let's mock the get_current_day method to create a controlled environment for tests on is_weekend_using_calendar function.

            We can create a mock_calendar_obj as shown in the following code snippet.

            from unittest.mock import Mock

def test_weekend():
    mock_calendar_obj = Mock()
    mock_calendar_obj.get_current_day.return_value = 6

    result = is_weekend_using_calendar(mock_calendar_obj)
    
    assert result is True
            • mock_calendar_obj = Mock() - Creating a mock object to mimic Calendar class object.
            • mock_calendar_obj.get_current_day.return_value = 6 - Sets return value of the get_current_day method to 6 in mock_calendar_obj object.

            Note: mock_calendar_obj.get_current_day.return_value = 6 is the syntax to specify a return_value for a method in a particular mock object.

            Similarly, update the second test so that the mock object returns a weekday value and change the test appropriately.

            This is possible because the attributes and methods of a mock object are lazy, this behavior of Mock object is detailed in the next section.

            # Lazy Attributes and Methods

            • A Mock must simulate any object that it replaces. To achieve such flexibility, it creates its attributes when you access them.

              >>> mock = Mock()
>>> mock.some_attribute
<Mock name='mock.some_attribute' id='4394778696'>
>>> mock.do_something()
<Mock name='mock.do_something()' id='4394778920'>

            • In the below example, you’re mocking the json library and calling the loads() function. The Python mock object will create the method so that its interface can match the library’s interface.

              >>> json = Mock()
>>> json.loads()                <-- No args

            • Unlike the real loads(), this mocked method will accept any arguments that you pass to it.

              >>> json = Mock()
>>> json.loads()                <-- No args
>>> json.loads('{"k": "v"}')    <-- Single argument
>>> json.loads('{"k": "v"}', 1, 12, 4, [])    <-- Any args

            • The return value of a mock object is also a Mock.

              >>> json = Mock()
>>> json.loads('{"k": "v"}')
<Mock name='mock.loads()' id='4379599424'>  <--- A mock instance

            • As the return value of each mocked method is also a Mock, you can use your mocks in a multitude of ways which you will learn in further sections.

              >>> json.loads('{"k": "v"}').get('k')
<Mock name='mock.loads().get()' id='1234871923'>  <--- A mock instance

            # Managing a Mock’s Return Value

            To control the behavior of the code, we have to control its return value. One way to do this using Mock is to specify the function’s return_value.

            >>> json = Mock()
>>> json.loads()
<Mock name='mock.some_attribute' id='4394778696'>

            In the above example, loads method returns a mock object. To force the json library to return '{"key": "value"}' on calling the loads method, we should set return_value attribute of the method in the mock as shown below

            >>> json = Mock()
>>> json.loads()
<Mock name='mock.some_attribute' id='4394778696'>
>>> json.loads.return_value = '{"key": "value"}'
>>> json.loads()
'{"key": "value"}'

            # Managing a Mock’s Side Effects

            • side_effect is a more powerful way of controlling a function/method behavior. It can be used to
              • raise an exception
              • call another function
              • return from a list of values (returing a new output iteratively on each execution)

            # Side Effect - Raise Exception

            Using side_effect, you can force a method of the mock object to raise an exception as shown below.

            >>> json = Mock()
>>> json.loads()
<Mock name='mock.some_attribute' id='4394778696'>
>>> json.loads.side_effect = TypeError
>>> json.loads()

...

            # Side Effect - Call another function

            Using side_effect, you can force a method of the mock object to call another function as shown below.

            >>> json = Mock()
>>> json.loads()
<Mock name='mock.some_attribute' id='4394778696'>
>>> def my_function():
        print("Call to my function")
>>> json.loads.side_effect = my_function
>>> json.loads()
Call to my function

            # Side Effect - Iterable

            To force a method of the mock object to behave differently on each method call, we can assign an iterable to the method's side_effect as shown in the below example.

            Every time we call the mock method, we get an item from the iterable. The iterable can contain return values, exceptions, or a mixture of both.

            >>> json = Mock()
>>> json.loads()
<Mock name='mock.some_attribute' id='4394778696'>
>>> json.loads.side_effect = ["Some string", TypeError]
>>> json.loads()
"Some string" 
>>> json.loads()

            If you call the method more times than the number of values given in the assigned iterable, then the mock method raises the StopIteration error.

            # Assertions and Inspection

            Mock instances capture information about how they are invoked. For example, you can see if you have called a method, with what arguments you have called the method, and so on.

            # Check the arguments with which a method is called.

            We can also assert the arguments. Both assert_called and assert_called_once functions have variants that let you inspect the arguments passed to the mocked method:

            • .assert_called_with(*args, **kwargs)
            • .assert_called_once_with(*args, **kwargs)
            >>> from unittest.mock import Mock

>>> json = Mock()  # Create a mock object

>>> json.loads('{"key": "value"}')
<Mock name='mock.loads()' id='4550144184'>

>>> json = Mock()
>>> json.loads(s='{"key": "value"}')
>>> json.loads.assert_called_with('{"key": "value"}')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "/usr/local/Cellar/python/3.6.5/Frameworks/Python.framework/Versions/3.6/lib/python3.6/unittest/mock.py", line 814, in assert_called_with
    raise AssertionError(_error_message()) from cause
AssertionError: Expected call: loads('{"key": "value"}')
Actual call: loads(s='{"key": "value"}')
>>> json.loads.assert_called_with(s='{"key": "value"}')
            • The assertions are passed only when the mocked method is called with the same arguments (including order and type) that you would pass to the actual method. Positional and keyword arguments are considered different.

            In the above code, json.loads.assert_called_with('{"key": "value"}') raised an AssertionError because it expected a call to loads() with a positional argument. But, it was called with a keyword argument. json.loads.assert_called_with(s='{"key": "value"}') assertion would have passed in the above scenario.

            # Special attributes

            >>> from unittest.mock import Mock

>>> json = Mock()  # Create a mock object
>>> json.loads('{"key": "value"}')
<Mock name='mock.loads()' id='4391026640'>

>>> json.loads.call_count  # Number of times you called loads():
1

>>> json.loads.call_args  # args of the last loads() call
call('{"key": "value"}')

>>> json.loads.call_args_list  # list of loads() calls
[call('{"key": "value"}')]

>>> json.method_calls  # list of calls to json's methods
[call.loads('{"key": "value"}')]
            • You can write tests using these attributes to make sure that the mock objects are behaving as you've intended them to.
