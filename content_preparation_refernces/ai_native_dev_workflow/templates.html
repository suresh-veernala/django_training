# Templates

Agent Input Format
  • Use this for every task you are giving to the agent.
  • This format will give agent additional context in structured way.
  • This format will help us to not miss any essential context about the task which is neither covered by PRD/Codebase/Docs and applies specifically to this task.

Show Template

Remove the Example Heading and its content when giving to the Agent.


## 1. Core Requirement

**What are we building?**

**Example:**
```
Build an API endpoint to fetch user's completed learning units. The API should return a list of units the user has completed, along with completion timestamps and scores if available.

can copy paste the business prd directly here or can give the path like prds/abc
```

---
 
## 2. Architectural Context

**Where does this code live and why? Are there existing patterns or structures to follow?**

**Example:**
```
This belongs in nkb_resources app because it deals with learning unit data. Follow the existing view structure in nkb_resources/views/. There's already a get_user_resources_v1 pattern we should mirror. Storage layer should use the ResourceStorage class.
You can also give app docs path here. but do only if the docs are necessary for current task.
```

---

## 3. Frontend/UI Context

**Client-side behavior that affects backend**

**Example:**
```
Mobile app displays this in a timeline view and expects data sorted by completion_date DESC. The app caches this data for 5 minutes, so include a last_updated timestamp. Frontend expects empty array if no data, not null.

it will auto navigate to another tab and they will call x api
```

---

## 4. External Systems & Integrations

**What external projects, services, or APIs does this interact with?**

**Example:**
```
This needs to call the analytics service (nkb_analytics project) to log completion events. Use the existing AnalyticsClient in common/clients/. Also, when a user completes a unit, we send a webhook to the notification service (runs separately on port 8001).
Or
The coupons service does all the validations and stores the coupon details in the database. We just have to pass the coupon to that service and it will return the coupon details. 

Sometimes we have to mention the other project/external api contracts
```

---

## 5. Misc

**What are we assuming? Any decisions, gotchas, or special context?**

**Example:**
```
- User can only see their own completed units, not others

- Don't use the old CompletionLog model, it's deprecated. Use UserUnitProgress instead.
- This query can be slow for power users (1000+ completions), so add pagination
```

PRD Format

What is a Good PRD?

  • There should be no conflicting flows/cases in the PRD.
  • There should be no missing flows/cases in the PRD.
  • There should be no unexplained or assumed flows/cases in the PRD.
  • Having only relevant details for the task in the PRD.

Example Output 

# Application Fee Discount Offer

## Problem Statement

Application portal shows significant delay and drop-off between OTP verification and application fee payment, especially within the first 30 minutes. Users complete OTP (demonstrating high intent) but delay or abandon before paying. The gap allows distraction, second thoughts, and loss of momentum. Currently no immediate incentive to act right after login.

**Opportunity:** First 30 minutes post-login is the highest intent moment. Introduce one-time incentive to convert intent into immediate action. Expected impact: improve OTP → payment conversion, shift more payments into 0-30 min window, faster revenue realization, lower re-marketing dependency.

---

## Solution (Goal)

Offer users a one-time discount immediately when they first log in to encourage completing payment right away.

---

## Pre Context (Omit if not applicable)

- Application is already created (via phone flow or portal)
- Payment flow exists where users pay application fee

**Starting State:**
User just logged in for the first time and is about to see the personal details page.

---

## Key Terms

**NIAT_200** - Our promotional discount code offering ₹200 off the application fee

**Application Fee** - The ₹1200 fee users pay to submit their application

**Discount Available** - User has access to a discount code but it's not currently reducing the fee amount (fee displays ₹1200)

**Discount Applied** - Discount is active and fee displays the reduced amount (₹1000 instead of ₹1200)

**Discount Used** - User completed payment successfully with discount. That specific code cannot be used again by this user.

**One-time** - Each discount code can only be used once per user. After payment succeeds with that code, it's permanently consumed.

**First Login** - The very first time a user logs in after completing OTP for this application

---

## User Journeys

### Journey 1: User Arrives Without Coupon, Clicks "Copy"

#### What User Sees
- Personal details page loads
- Popup appears immediately showing:
  - Heading "Offer Unlocked!"
  - Discount code "NIAT_200"
  - Message "Get ₹200 OFF on your application fee"
  - "Copy" button
  - "×" close icon at top-right

#### What User Can Do
- Click "Copy" button to copy code
- Click "×" to close popup
- Ignore and continue to form

#### What Happens Next

**User clicks "Copy":**
- Code "NIAT_200" copies to clipboard
- Popup shows "Copied!" confirmation next to the code
- Popup can be closed
- User fills personal details and continues through application
- When user reaches payment page later:
  - Fee breakdown shows "Application Fee: ₹1200"
  - Discount line shows "Discount (NIAT_200): -₹200"
  - Total shows "Amount to Pay: ₹1000"
  - "Clear" link appears next to discount
- User can proceed to payment or clear the discount

#### Rules
- Popup appears only on first login for this application
- If user logs out and logs back in later, popup does not appear again
- Clicking "Copy" makes the discount automatically applied at payment (user doesn't manually enter code)
- Discount state is consistent across all devices (phone, laptop, tablet)

---

### Journey 2: User Arrives Without Coupon, Clicks "×"

#### What User Sees
- Personal details page loads
- Popup appears immediately showing:
  - Heading "Offer Unlocked!"
  - Discount code "NIAT_200"
  - Message "Get ₹200 OFF on your application fee"
  - "Copy" button
  - "×" close icon at top-right

#### What User Can Do
- Click "×" to dismiss popup
- Ignore and continue to form

#### What Happens Next

**User clicks "×":**
- Popup closes immediately
- User fills personal details and continues through application
- When user reaches payment page later:
  - Fee shows "Application Fee: ₹1200"
  - Total shows "Amount to Pay: ₹1200" (no discount applied)
  - Below fee section, a discount card appears:
    - Shows "Get ₹200 OFF"
    - Shows "Code: NIAT_200"
    - Has "Apply" button
  - Coupon input box is available for manual entry
- User can click "Apply" on card or manually enter code

#### Rules
- Popup appears only on first login for this application
- If user logs out and logs back in later, popup does not appear again (discount card still available at payment)
- Clicking "×" does not remove discount eligibility, just means it's not automatically applied
- Discount remains available to apply manually at payment page anytime before payment succeeds

---

### Journey 3: User Arrives With Coupon in URL

#### What User Sees
- User clicks link with coupon parameter: `www.app.com/?coupon=NAT400`
- Completes OTP if needed, then logs in
- Personal details page loads
- **No popup appears** (different from Journey 1 and 2)
- User sees regular personal details form

#### What User Can Do
- Fill personal details and continue through application sections
- Navigate to payment page

#### What Happens Next

**User progresses through application:**
- User fills personal details, uploads documents, completes other sections
- No popup interruption
- When user reaches payment page:
  - Fee breakdown shows "Application Fee: ₹1200"
  - Discount line shows "Discount (NAT400): -₹400" (discount from URL coupon)
  - Total shows "Amount to Pay: ₹800"
  - "Clear" link appears next to discount

**If user clicks "Clear":**
- Discount is removed
- Fee reverts to "Amount to Pay: ₹1200"
- Discount card appears below showing:
  - "Get ₹200 OFF"
  - "Code: NIAT_200"
  - "Apply" button
- User can now apply NIAT_200 or re-apply NAT400
- User can switch between coupons multiple times before payment

#### Rules
- When URL contains coupon parameter, popup is never shown
- URL coupon is automatically applied at payment page
- Our standard NIAT_200 offer is still available if user clears the URL coupon
- User can only have one coupon active at a time (cannot apply both simultaneously)

---

## Edge Cases & Special Situations

### Payment Gateway Timeout or Failure

**Situation:** User clicks "Pay Now" but payment gateway times out, fails, or shows error

**What Happens:**
- User returns to payment page (either automatically or manually)
- Discount still shows as applied if it was active before payment attempt
- User can retry payment immediately with same discount
- User can also clear discount and re-apply if needed
- Discount is only marked as "used" when payment is confirmed successful by gateway

**Key Point:** Technical payment failures or gateway issues do not consume the discount

---

### Invalid URL Coupon

**Situation:** User clicks link with coupon that doesn't exist or is expired: `www.app.com/?coupon=INVALID123`

**What Happens:**
- No popup shows (because URL contains coupon parameter)
- User fills application sections normally
- When user reaches payment page: System attempts to auto-apply INVALID123
- Error message appears: "Invalid or expired code"
- Discount card appears below showing NIAT_200 as alternative option
- User can apply NIAT_200 from the card

**Key Point:** Invalid coupons from URL don't break the flow. Standard offer is still available as fallback.

---

### User Tries Already-Used Coupon

**Situation:** User manually enters a coupon code they already used in a previous successful payment

**What Happens:**
- User types the code in coupon input box
- Clicks "Apply"
- Error message appears: "This code has already been used"
- Fee remains at ₹1200 (no discount applied)
- User must enter a different valid code or proceed without discount

**Key Point:** Each coupon code can only be successfully used once per user

# Module Name

nw_assessments_core

## Responsibility

NW Assessments Core is the central app for managing the full lifecycle of organizational assessments. It handles assessment creation, user participation, attempt tracking, proctoring, integrity monitoring, and access control.

This app owns the answers to:
- What assessments exist and how are they configured per organization?
- Who can access an assessment and under what conditions?
- How do users start, attempt, and complete assessments?
- How are attempts tracked, scored, and reviewed?

---

## Key Entities

**Assessment** — A platform-level definition for group of exams (sections).

**Organisation Assessment** — An org-specific instance of an assessment. This is which holds custom settings: duration, start/end times, passing scores, IP restrictions, integrity rules, and review policies.

**User Organisation Assessment** — This is the link between  candidate and  assessment. Tracks the attempt status, start time, scores, and time spent. This is the core record for determining where a user stands in an assessment.

**Assessment attempt lifecycle** — Attempts move through states: `NOT_STARTED → IN_PROGRESS → ATTEMPTED → EVALUATED`. The transition logic and guards live in this app.

**Integrity monitoring** — A set of mechanisms to ensure honest attempts: activity logging (tab switches, fullscreen exits), IP whitelisting, PIN gates, and check-in requirements. Each has its own config model and runtime records.

**Check-in** — An optional pre-assessment attendance step used for offline or proctored settings. Candidates must check in before they can start. Without a check-in record, access is blocked.

**Entity Profile** — Allows an organization to present a different identity (logo, name) to candidates across different assessments they publish. Useful when the same org runs multiple branded drives.

**Assessment PIN** — A one-time gate that prevents candidates from accessing an assessment before the proctor shares the PIN. Commonly used in offline/proctored drives where the coordinator reveals the PIN at the start time, making it impossible to begin early or retry without re-entry.

**Activity log** — A per-event record of what a candidate does in the browser during an attempt. Captures tab switches, fullscreen exits, copy/paste, window blur, and screenshot attempts. This is what integrity analysis and violation scores are built from.

**Psychometric trait** — For assessments that measure personality or cognitive traits, each trait (e.g., Openness, Conscientiousness) gets its own score record tied to the exam attempt. Used for candidate behavioral profiling and not part of the regular scoring flow.

---

## Key Features & Flows

### 1. Starting an Assessment

Before a candidate can begin, the system validates:
- The assessment is within its active time window
- The user has remaining attempts
- The user's IP is in the whitelist (if network restrictions apply)
- The correct PIN has been entered (if PIN is required)
- The user has checked in (if check-in is mandatory)

Once all checks pass, the `UserOrganisationAssessment` is moved to `IN_PROGRESS` and the candidate proceeds to the exam.

### 2. Integrity Monitoring

During the attempt, the frontend sends events to the backend as the user interacts with the browser (tab switches, fullscreen exits, copy/paste). Each event is saved as a `UserOrgAssessActivityLog` entry. These logs feed into integrity analysis and violation scoring, visible to the org admin.

### 3. Quit Assessment

There are two ways an attempt ends early: the user exits themselves, or an admin terminates it. Both go through the same API. The presence of admin metadata in the request determines which path is taken.

- User exit: status is updated, score is finalized.
- Admin termination: same as user exit, plus the candidate receives a real-time WebSocket notification in their browser so their session closes immediately.

If the attempt was already ended (or already scored), the request is silently ignored to prevent double processing.

### 4. Assessment Review

After submission, candidates may be allowed to review their answers. The `OrgAssessReviewConfig` controls this:
- Review may be disabled entirely
- Review may only be available after results are officially published
- Correct answers and scores may or may not be shown

### 5. Check-In Management

Orgs can require candidates to mark their presence before starting. The system records the check-in and blocks access for users who haven't done it. Used primarily for offline or campus-based drives.

### 6. Post-Assessment Schedulers

- Reminder emails are sent to invited candidates who haven't started
- Post-assessment actions (scoring finalization, result publishing triggers) are handled via scheduled jobs

## Data Layer

All models inherit from `AbstractDateTimeModel` (provides `created_at` and `updated_at` timestamps). Most models use auto-increment integer primary keys or UUIDs. Only non-obvious fields are documented below.

---

### Configuration Models

#### `Assessment`
Base platform-level assessment record. Organizations don't configure this directly.

#### `AssessmentLevel`
Links an assessment to an exam unit on the nkb platform. One row per section of the assessment.

| Field | Notes |
|-------|-------|
| `unit_id` | FK to `Exam` in the nkb_exam service |

#### `OrganisationAssessment`
The main configuration record for an org's assessment. Most other config models reference this.

```
{
    "custom_data_form_link": str,
    # extensible
}
```

#### `OrgAssessReviewConfig`
Controls whether and how candidates can review their submission. Determines if correct answers and scores are visible, and when review becomes available.

#### `OrgAssessLogoutConfig`
Controls what happens if a candidate logs out mid-attempt — whether the attempt is auto-submitted or can be resumed.

#### `OrgAssessIntegrityConfig`
Holds integrity settings for the assessment, such as whether a PIN is required before starting.

#### `OrgAssessmentNetworkConfig`
Stores whitelisted IPs or IP ranges for the assessment.

| Field | Notes |
|-------|-------|
| `allowed_ip_addresses` | JSON array, supports individual IPs and CIDR ranges |

#### `OrgAssessPinDetail`
Stores the PIN a candidate must enter before starting. The PIN is typically shared by the exam coordinator at the start time to prevent early access.

---

### Runtime Models

#### `UserOrganisationAssessment`
One row per user per assessment. Central record for tracking attempt progress and outcomes. Status transitions use `UserOrgAssessStatusEnum`.

#### `UserOrgAssessExitDetail`
Created when an attempt ends early — either by the user or an admin. Acts as an idempotency guard to prevent double-processing. Stores who triggered the exit and in what capacity (`ExitedByEnum`).

#### `UserOrgAssessCheckInDetail`
Records that a user has checked in for an offline assessment. Only created when check-in is enabled for the assessment.

#### `UserOrgAssessActivityLog`
One row per browser event during the attempt. Used for integrity analysis. Activity types are defined in `ActivityTypeEnum`.

#### `ExamAttemptTraitDetail`
Stores per-trait scores for psychometric assessments. Each row corresponds to one measured trait for a completed exam attempt.

| Field | Notes |
|-------|-------|
| `exam_attempt_id` | FK to `ExamAttempt` in nkb_exam |

#### `UserFeedback`
Captures candidate feedback after an assessment. Used by orgs to improve future assessments.

#### `UserOrgAssessConfig`
Stores per-user state for the assessment configuration flow (e.g., which assessments a user has previewed). Not relevant to candidates.

```
metadata: {
    "previewed_org_assess_ids": ["..."],
}
```

#### `OrgAssessAccessConfig`
Grants automatic access to an org assessment based on a user's active subscription plans or learning path enrollments.

```
default_access_subscription_plans_config_str: {
    "lp_enroll_plans": [],
    "subscription_plans": [],
}
```

---

### Organization Models

#### `OrganisationDetail`
Master record for an organization on the platform.

#### `EntityProfile`
Represents a named identity shown to candidates. An org can have multiple profiles for different assessments or drives.

#### `UserOrganisation`
Links a user to an organization with their role within it. Role values are in `OrgUserRoleEnum`.

---

### Metadata Models

#### `OrgAssessmentTag`
Tags attached to an assessment for categorization. Tag types are defined in `AssessmentTagTypeEnum`.

#### `OrgAssessmentReport`
Stores a candidate's report of a content issue (wrong answer, broken test case, etc.) against a specific question in the assessment.

```
metadata: {
    "org_assessment_title": "...",
    "exam_id": "...",
    "question_id": "...",
}
```

## Other Apps Dependencies

nkb_exam — The exam execution service. Assessment sections map to exams in nkb_exam. Starting or ending an attempt involves calling nkb_exam. Scores are pulled from here after evaluation.

nkb_exam_extensions — Extends nkb_exam with exam slots (time-window-based scheduling).

nkb_fullstack_exam — Provides HTML coding question content for assessments that include that question type.

nkb_primitive_coding — Provides submission data for primitive coding questions.

ib_user_accounts_helper — Used to fetch or create user profiles for candidates and admins interacting with assessments.

nw_assess_config — Handles publishing assessments and inviting candidates. This app is the upstream that creates `UserOrganisationAssessment` records. Also used to fetch exam config details.

nw_auth — Used to validate user roles (admin vs candidate), subscription plans, and entity API keys.

nw_organisation — Used to resolve a user's organization and to reserve/debit credits when candidates are invited.

nw_assess_stats — Reads activity logs and attempt data from this app to generate candidate reports and integrity analysis.

nw_question — Provides org-specific question references (`OrgQuestion`) used when building assessments.

## External Dependencies

AWS SES — Sends reminder and notification emails to invited candidates.

AWS Step Functions — Powers the assessment schedulers: reminder emails and post-assessment lifecycle actions.

AWS S3 — Stores assessment reports, and proctor images. Presigned URLs are generated for secure access.

Segment — Receives assessment result events for downstream CRM and notification workflows.

LP Service — Used to check a user's active access plans when evaluating `OrgAssessAccessConfig` auto-access rules.

Hasura GraphQL — Assessment publish status is synced to the GraphQL layer for the frontend dashboard.
