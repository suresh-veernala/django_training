# Mar 30 - Apr 26


# Workflow updates

  • The /write-tests command is now split into layer-specific commands:

    • /write-interactor-tests - Writes unit tests for the interactor layer.
    • /write-presenter-tests - Writes unit tests for the presenter layer.
    • /write-storage-tests - Writes unit tests for the storage layer.
    • /write-view-tests - Writes unit tests for the view layer. Use these commands based on the layers changed in your task.

  • The /plan command is now replaced with two focused commands:

    • /propose-approaches
      • Suggests 1-2 approaches for your requirement.
      • Use this when you want to evaluate multiple approaches and do not already have one in mind.
    • /plan-approach
      • Expands a selected approach into an execution-ready plan when more structure is needed before implementation.
      • Use this when you already have an approach in mind and want to evaluate it in more detail, or use it after /propose-approaches to create a more detailed plan.

  • Use /implement when you already have a detailed plan and want the agent to implement it.

  • Use the new /implement-with-plan command when you have a high-level approach and want the agent to plan briefly before writing code.

  • You can use these combinations based on the task:

    • /propose-approaches -> /plan-approach -> /implement
    • /propose-approaches -> /implement-with-plan
    • /implement-with-plan when you already have a high-level approach.
    • /implement when the task is simple enough to implement directly without a separate plan.

  • /debug is now replaced with /resolve-issue.

  • /breakdown is renamed to /write-tech-prd for clearer naming.

  • /refactor is updated to improve readability.

  • Added a new db-design skill to improve table design decisions.

  • Naming conventions are improved.

  • The /review-plan command is now removed.

  • Design analysis support is removed from all prompts because the related use cases are very limited.


# Documentation updates

  • The Suggested Prompts and Workflows pages are updated to reflect the changes above.
  • Use the helper prompts in Suggested Prompts for each step, and add your own additional instructions when needed.

# Optimizations

  • All prompts are optimized to reduce tool calls and prompt sizes.

Note:

  • If the propose-approaches command produces multiple approaches, tag only the selected approach for the next step. Remove supporting sections like Why this approach and Trade-offs as they are not relevant to the next step.
  • Use /plan-approach only when your task needs a layer-wise plan. You can continue in the /propose-approaches chat if context window usage is minimal.
  • Do not use /build-api when you do not need an API contract.
  • The /plan command is deprecated and will be available until next week. If you face any issues with new commands can use that.

# Toolkit Upgrade

Problem: Previously all the backend prompts are loaded to global cursor folder and this caused some conflicts with frontend prompts and non-backend projects.

Solution: The sync script now copies the prompts directly to the specific backend projects you have configured. Follow below steps:

  • Remove the previous script and in the same path download the new one using:
export AWS_PROFILE=<profile name>
aws s3 cp s3://nw-agent-toolkit-backend/client/nw-agent-toolkit-sync.py nw-agent-toolkit-sync.py
  • Open the folder .nw-agent-toolkit-backend in your system home folder.
  • Open the file named config.json and add the list of backend projects there and save it. Example:
"backend_projects": [
  <Absolute path to repo. eg: "/home/nw-tech-prasanth/work/nw_projects/agent-toolkit-backend">
]
  • In your backend projects add .cursor/ to gitignore to avoid the changes appearing in your git whenever prompts are updated. Push the updated .gitignore to remote to ensure changes doesn't appear distinct to each member of your team.

WARNING

The next command will delete the entire folders previously managed by toolkit. (~/.cursor/commands/, ~/.cursor/rules/, ~/.cursor/skills/, etc.). If you have any custom prompts or rules saved in those folders, copy them to a safe location first and restore them after the sync completes.

  • Now go to the path where the new script is downloaded, activate venv and run:
python nw-agent-toolkit-sync.py link --profile <aws profile name>
  • Now open your backend projects and check if all the prompts have appeared in .cursor folder.
  • Close and re-open the cursor windows for the prompts to be synced with cursor ide. To validate open cursor settings -> Rules, Skills, SubAgents -> the count of rules should be 2.
  • Replace the .pre-commit-config.yaml in each of your backend project with the updated one from ~/.nw-agent-toolkit-backend/cursor/git_hooks/ push it to remote and complete the setup mentioned on top of the file.

# Recommendations

  • Keep all the app and project docs in the project_docs folder in your root for models to easily navigate through them and having them at a centralized place will make it easier to refer in the prompts.
  • Remove all the unrelated/deprecated config files from the project to avoid models reading them and make wrong choices. why? -> Though we use poetry, models sometimes read the requirements*.text and referring the old dependencies. Examples:
    • all requirements files. requirements.txt, requirements_db.txt, requirements_lambda.txt etc.
    • env*.encrypted
    • Dockerfile, .dockerignore
    • api_doc.json
    • codeship*.yml
  • Upgrade your Readme.md with clear mention of how to setup the project and any other useful info like common issues which can help agents to navigate through the project.
  • Better documentation of HISTORY.md. If your project doesn't have one, Its recommended to implement it from now on.
  • Prefer the naming convention .venv for virtual environments in your projects. Having different namings in different systems making the model to do multiple tool calls just to figure out the name of virtual environment to activate it.
  • Avoid reviewing the tests with the model, which may not be effective on a cost-basis considering the tokens they consume. Prefer generating them after the code is reviewed.
  • If you have any custom instructions you want to add, can append it to respective step or add as your own custom rules in the cursor
