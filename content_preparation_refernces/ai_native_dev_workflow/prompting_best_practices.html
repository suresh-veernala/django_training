# Cursor IDE & Prompting - Best Practices

# Cursor IDE

# Close Unrelated Files

  • Close files that are not relevant to your current task. Cursor sends all open file paths to the model by default, and models tend to pull those in while reasoning, even when they are unrelated to what you are working on. These unnecessary files content might add noise to model decisions, and increase the overall context window.

# Use Inline Chat for Targeted Edits

  • Use inline chat (Ctrl + K) for targeted, self-contained edits, such as refactoring a method or making straightforward changes that do not need:
    • context outside the current file
    • previous conversation context
    • updates across multiple files
  • Cursor Agent chat includes broader context by default, which is useful for complex tasks but often overkill for simple modifications.

# Prefer IDE Functions

  • When applicable, use IDE functions for deterministic or trivial operations like Refactor/Rename, Find Usages etc. instead of the Agent.
  • These operations are deterministic by nature, so IDE-native actions are usually faster, more reliable, and do not require additional prompting loops.

Incorrect usage examples:

  • Change this method name to xyz and update all other places.
  • List all the places where this method is used.
  • Move this file to xyz folder and update references.

# Model Usage

# Model Selection

  • Use the right model for the task. Not every task needs the most powerful model. Use Opus/Sonnet for genuinely complex work: architectural decisions, multi-step reasoning, and hard debugging.
  • For writing tests, generating boilerplate, or making small edits, a lighter model is often enough.
  • Different models have different strengths. Using a high-capability model for minor tasks like below is unnecessary:
    • Write tests for this file.
    • Fix linter issues.
    • Improve naming/readability of this method/class/file.
    • Suggest better names for this method/class/file/variable.
  • Models like Sonnet are also unnecessary when the task does not require any reasoning or when instructions are straightforward. Newer models like composer-2 can handle no-complexity tasks with good accuracy.
  • At the same time, low-capability models are not ideal for planning or code reviews because they are not designed for deeper, multi-layer reasoning and large context handling.
  • Using a high-capability model does not guarantee the best result. Most of the time, it is overkill.
  • Try different models for different task types and build a practical understanding of each model's strengths to avoid rework.

# Match Reasoning Effort to Task Complexity

  • Select the right reasoning effort based on your task. High or Medium reasoning should not be the default choice just because the model can think more. Examples:

    • Keep effort low/off for non-critical changes that do not span multiple files.
    • Use medium effort for planning or review tasks, based on complexity.
    • Keep effort low when you already have clear, step-by-step implementation instructions.
    • Turn reasoning off (use a non-thinking version) for straightforward refactors, writing tests, fixing linter issues, and similar tasks.

  • High reasoning effort on easy tasks often leads to overthinking and slower execution without meaningful gains. It can also produce long reasoning traces that dilute focus in later turns.

    Example: Sonnet/Opus 4.6x series can spend 2-5 minutes thinking and produce 5-10k output tokens just for reasoning.

# Context Window

# Add Right Context

  • Add only the context that is directly relevant to the task. Do not load docs the task does not need. Attaching documentation "just in case" inflates context and can make the model worse at the exact task you are asking it to do.Examples:
    • Adding the entire app doc for an API change or a new feature that does not affect app flow adds noise.
    • Copy-pasting the full PRD (including business sections like metrics/admin operations or frontend-only UI details) is unnecessary for a backend task when only a small section is relevant.
  • More context is not always better. As context grows, output quality can degrade because the model has to rank more competing signals before deciding what matters.

# Minimize Iterations

  • Give feedback in one pass when reviewing code/plans or discussing approaches. Do not send feedback one message at a time as you remember points.
  • Each message adds another model call with all accumulated context, even when the model could have completed everything in one iteration.
  • Consolidate feedback into a single message to keep the session focused and give the model a complete picture upfront.

# Start Fresh with New Chats

  • Ask yourself whether the code changes you want to make actually depend on this session. If not, start a new chat instead of continuing the same thread. Start a new chat when:
    • The new changes do not depend on current conversation context.
    • The new feedback/update to code or plan does not require previous conversation context.
    • The current conversation is already more than half full.
  • Carrying unneeded context from a long thread makes future changes slower and less efficient, and it increases the chance that the model follows stale assumptions from earlier discussion.

# Avoid Auto-Adding Large Files

  • Do not keep large files (such as app docs) set to auto-attach in every conversation. Most tasks can be completed without that full context.
  • Large files can drift the model toward unrelated scenarios, especially when those files contain broad architecture details not needed for the current task. Examples:
    • Instead of dumping full server logs or asking the model to read all terminal output, paste only the last few relevant error lines.
    • Adding the entire folder in context instead of a specific file
    • Adding the entire file in context instead of pointing to a specific lines or method

# Avoid Delegating Trivial Tasks

  • Avoid asking the model to run commands for simple operations like:
    • generating migrations for model changes
    • running the server or building the API
    • repeatedly checking whether a file generates output
  • Use the model where reasoning and code understanding matter most. For routine command execution, direct IDE/terminal usage is often cleaner and avoids unnecessary context-heavy loops.

# MCP

  • Pasting a ClickUp/Sentry link and asking the model to fetch/fix something is convenient, but enabling all MCPs and tools by default is not practical.

  • Enable only the MCPs and tools needed for the current task.

  • Every MCP has descriptions, and each MCP can include many tool descriptions. This helps with tool selection, but irrelevant tools also add noise to context and make tool choice less precise.

      Calculation: If 2-3 MCPs in your IDE consume around 2-3k context, then for a planning task with ~15-30 tool calls, total billed input can become large (for example, 3*20 = 60k). Caching may help, but it depends on provider infra and TTL is often short.

Examples:

  • Many plugins/vs code extensions now bundle MCP tools and add them to your MCP config without you noticing, which creates hidden overhead. Example: Installing GitKraken or Console Ninja may update MCP config and enable tools regardless of task.
  • By default, an MCP server can include many tools that are not needed for your task. Disable irrelevant tools even when the MCP is enabled. Example: Official Figma MCP has ~13 tools, while you may only need 3-5 for frontend design analysis.
  • Disable MCPs like Sentry or ClickUp when they are not needed.

# Prompting

# Give Right Direction

  • Give the model clear direction. Do not expect it to guess missing pieces by:
    • not giving enough context
    • dumping content without saying what you want to achieve or what the model should focus on
    • not mentioning the approach/plan when you already know it
    • not mentioning the app name or specific files when you already know where changes are needed
    • not referencing patterns the model should follow for the new task
    • not saying patterns like common out or remove existing files when you already know the task involves them as a direction
  • Models are capable, but pointing to a known approach or existing code helps them follow the expected pattern faster and more accurately.
  • Models read many files to understand the codebase and patterns, which can drift output toward unrelated patterns. Based on what they read, they may choose a different (still valid) pattern that is not what you expected, leading to extra iterations.

# Avoid Conflicting Instructions

  • Avoid conflicting instructions in your prompt. Do not give directions that contradict each other in different sections. Example: The PRD says "send score to frontend," but later you add "we will not send score to frontend."
  • Conflicting instructions can cause inconsistent output and rework.

# Avoid Vague Instructions

  • Always give clear, detailed instructions instead of making the model guess your intent.
  • Vague instructions often lead to unexpected output and rework. Examples:
    • "make the code more readable" (without saying what the current readability issue is)
    • "improve the performance of the API" (without giving supporting context)
  • If detailed typing takes time, use voice input first and then format it.

# Cache Behavior

  • Prefer creating a new chat if:
    • a chat has been idle for more than ~5 minutes and previous context is not important enough to continue in the same conversation
    • you want to edit a message in a long conversation to reduce context window and the model supports cache behavior
    • you want to change the model in between the conversation.
  • Editing a previous message or model change invalidates cache and re-sends everything from that point onward. In most cases, it is better to send a new message at the end of the thread.
  • Only edit an older message when it meaningfully reduces context or corrects a direction that has been wrong for a while.
  • Continuing an idle long conversation can trigger a fresh cache rewrite, which is often less efficient than starting a new chat.

# Right Mindset

  • When using any prompt or step, first develop a basic understanding of what it does, how it works, and whether you actually need it. Using prompts without understanding them can increase delivery time and consume more tokens or context window space leading to higher cost.
  • Do not spend time testing whether the model can do something or not. That usually adds unnecessary time, whereas giving it the right direction and context is more likely to produce the desired output.
  • Treat Models/Agents as capable junior developers but keep in mind that:
    • They may not be familiar with the codebase or broader context, but they can read the codebase and identify patterns.
    • They can propose approaches, but they may sometimes miss scalability concerns or the best practices followed in the codebase.
    • They may overlook certain cases because of missing context, business decisions, or by models behavior itself.
    • Their code may sometimes miss established best practices or optimizations, or it may become too tightly coupled to the immediate requirement.
  • Do not expect the model to be perfect or one-shot your entire task, especially when it does not have the same level of understanding that you do.
  • Even if the model misses a few things, such as loops, naming, or other basics, treat it the same way you would treat a peer review for a fellow developer's work: iterate and improve it rather than criticizing the model for missing something basic.
  • Just as you would ask specific questions to understand why a developer missed something, do the same here to identify gaps in the predefined prompts, context to improve on the next iterations.
  • Use the model as a tool to support your thinking, not as a replacement for it. Avoid getting stuck in the mindset of "that's what the model suggested".
  • Models are capable of completing the assigned task but also highly susceptible to the phrasing of context. When you feel that model output is not what you expected, iterate and understand why it drifted in unexpected direction and improve your prompting further.
