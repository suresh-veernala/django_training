# Suggested Invocation Prompts

These are the commands available in the toolkit and example prompts showing how to invoke each one effectively. The pattern is consistent — provide the requirement or context, attach the relevant files, and invoke the command.


/Write Tech PRD — Business PRD to Tech PRD
  • Give the input in Agent Input Format with agent mode
  • This generates a tech prd in the feature specific folder.
  • Before running this keep only relevant PRD content and remove irrelevant content to backend like success metrics, DA team notes, frontend specific sections etc.

Example:

Write the tech PRD for the requirement in
```
@requirement.md
```
by following the guidelines in
```
/write-tech-prd
```
word-to-word and ask me questions as per the guidelines criteria to avoid rework.
- Keep the tool calls and codebase research to minimum. 
- Do not ask implementation level questions
- Do not include implemenation phrasing or what to do. Focus only on what capabilities and not how to do.

/propose-approaches — Evaluate mutliple approaches
  • Use this when the solution direction is not fixed yet and you want the agent to research and propose 1-2 viable approaches.
  • Use this in a new chat session with mode as agent.
  • Can skip to implement-with-plan if you already know the implementation approach.
  • Give the business PRD or Tech PRD as input to this step.
  • Attach app docs only if needed.

Example:

Propose approaches for the requirement in
```
@jd_based_interview_k9mf/tech_prd_3x7hp2qw.md
```
by following the guidelines in
```
/propose-approaches
```
word-to-word and ask me questions as per guidelines criteria to avoid rework.
- Consider Scalability & Extensibilty and proppose optimized approaches by following existing codebase patterns.
- This is not low level implemention plan. Keep the tool calls and codebase research to minimum.
- Evaluate what can be extended rather than just inclining to new models/apis just because the requirement naming doesn't match.
- Do not focus on what the requirement currently asks, think in backend domain and codebase generalization on how it might change later.
- Do not ask questions about - What to do, How to do, Which might be better.

/plan-approach — Detailed layer wise plan for the selected approach
  • Use this in a new chat session with agent mode or when /propose-approaches context window consumption is minimal. Using in same chat reduces cost.
  • Use propose-approaches and then keep only the selected approach in the file to save context before invoking this step.
  • If you directly have an implementation approach give clear direction and references if applicable.

Example:

Plan for the approach
```
@jd_based_interview_k9mf/selected_approach.md
or
approach in your own words
```
for the requirement in
```
@jd_based_interview_k9mf/tech_prd_3x7hp2qw.md
```
by following the guidelines in
```
/plan-approach
```
word-to-word and ask me questions as per guidelines criteria  to avoid rework.
- Follow codebase patterns strictly when ambiguous follow `skill-map`.
- minimize tool calls by optimizing the no.of file read and read only relevant lines
- Do not mention low level implementation detail like method names, typing etc. explain in easy to understand natural language
- Avoid verbose outputs and keep it concise giving enough details for next implementation
/implement-with-plan — Implement from Selected Approach
  • Use this in a new chat session with agent mode.
  • If you have used propose-approaches then keep only the selected approach in the file to save context before invoking this step.
  • If you directly have an implementation approach give clear direction and references if applicable.
  • Attach app docs only if needed. Most often you shouldn't be needing them at this step.
  • Use implement if you have used plan-approach already.

Example:

Plan and write the code for the approach
```
@jd_based_interview_k9mf/selected_approach.md
or
approach in your own words
```
for the requirement in
```
@jd_based_interview_k9mf/tech_prd_3x7hp2qw.md
```
by following the guidelines in
```
/implement-with-plan
```
word-to-word and ask me questions as per guidelines criteria to avoid rework.
- Follow codebase patterns strictly when ambiguous follow `skill-map`.
- Write Optimized and efficient code. Avoid Re-fetching data or writing orm queries or external service calls in loops.
- Follow existing codebase pattern namings and strip requirement specific namings.
- Avoid default arguments for dtos, method args etc.
- Write readable and minimal code and avoid complex code patterns.
/implement — Implement
  • Use this in a new chat session with agent mode.
  • If you directly have an implementation approach give clear direction and references.
  • Attach app docs only if needed. Most often you shouldn't be needing them at this step.

Example:

write the code for approach
```
@jd_based_interview_k9mf/selected_approach.md
or
approach in your own words
```
for the requirement in
```
@jd_based_interview_k9mf/tech_prd_3x7hp2qw.md
```
by following the guidelines in
```
/implement
```
word-to-word and ask me questions if needed to avoid rework.
- Follow codebase patterns strictly when ambiguous follow `skill-map`.
- Write Optimized and efficient code. Avoid Re-fetching data or writing orm queries or external service calls in loops.
- Follow existing codebase pattern namings and strip requirement specific namings.
- Avoid default arguments for dtos, method args etc.
- Write readable and minimal code and avoid complex code patterns.

/build-api — API Specification Builder
  • Use this in new chat session with agent mode.
  • Not a mandatory step. Can use based on the level of API changes and the complexity of the task. Can skip if the changes are minimal or need not be communicated to frontend separately.
  • Use only if this needed otherwise implement already takes care of this.

Example:

Build the API spec for the endpoints defined in
```
@jd_based_interview_k9mf/selected_approach.md
```
by following the guidelines in
```
/build-api
```
and follow existing api spec file conventions

/review-impl — Code Review
  • Use this in new chat session.
  • Can skip if the changes are to minor and can be manually reviewed faster than using this command.
  • Can skip if the code changes are good and additional review is not needed.
  • Can attach the tech prd if required.

Performs a deep, systematic code review of a branch's changes against the selected approach and requirement — checking data contracts, tracing flows, and surfacing issues that matter.

Example:

Review the [last commit / current branch against development]
against the selected approach in
```
@jd_based_interview_k9mf/selected_approach.md
```
for the requirement in
```
@jd_based_interview_k9mf/tech_prd_3x7hp2qw.md
```
by following the guidelines in
```
/review-impl
```
word-to-word.
- Evaluate if the new code followed existing codebase patterns and namings avoiding requirement specific terminology.
- Identify complex code patterns and naming that hamper the readability.
- Focus on optimizations like avoiding unnecessary loops, db/external service calls in loops, n+1 data access etc.
- Do not give opiniated nitpicks when they doesn't improve readability, maintainability when they are just good to have kind.
- Ensure any suggestions do not introduce new issues
- Refer related layer specific instructions in `.cursor/review_standards/review-map.md`.

/write-{interactor/storage/presenter/view}-tests — Write Unit Tests< for each layer/strong>
  • Use this in new chat session.
  • Can skip if the changes are to minor or tests update is not needed.
  • Can add tech prd if required.
  • If you already know which test cases to write, can provide them.
  • Prefer using this after the code is reviewed to avoid rework.

Example:

Write the tests for the code changes in
[
    Last 1/2 commits
    or
    Current branch against development
    or 
    Specific files/modules
]
by following the guidelines in
```
/write-{interactor/storage/presenter/view}-tests
```
word-to-word.
- If multiple tests need same setup, extract it to common_setup as per the guidelines (not applicable for views tests)
- For views layer tests, see if the interactor have tests written already, then use them as reference to decide what scenarios needed integration testing
- Avoid writing tests for every logical branch, each test method should test a possible business scenario
- A test should fail when any part of the logic is modified to ensure test coverage
- Avoid inline factory creation, complex data setups and longer methods in tests.
- Tests should be as readable as code with right namings and patterns.
- Do not do hacky fixes or initiating with none or empty string arguments just for sake of fixing a failed Test. Test should always represent the correctness of the logic.
- Keep tool calls to minimum by reading only relevant lines and files.

/resolve-issue — Fix a Concrete Issue
  • Use this in a new chat session.
  • Use this when you already have a concrete issue, bug ticket, or tracked problem and want the agent to drive the fix.

Example:

Resolve the below issue 
```
{detailed explanation of issue, the flow in which its triggered, the expected behavior, the actual behavior & any logs/traceback}
```
by following the guidelines in 
```
/resolve-issue
```
word-to-word.
- Do not speculate and overthink unrealistic scenarios.
- Do not add preventive safeguards or hacky fixes just for the sake of it.
- Do not modify tests unless the current issue you have fixed is a valid branch that needs explicit coverage.

/update-docs — Documentation Update
  • Use this in new chat session.
  • Prefer this only when doc need considerable changes, if for few words can edit it manually.
  • Can skip if the changes doesn't need any documentation update.

Updates project and module documentation after completing a feature, enhancement, or bug fix that changed flows or capabilities.

Example:

Update the app doc in
```
project_docs/abc_app.md
```
for the changes made in current branch compared to [development]
by following the guidelines in /update-docs

/what-to-test — Manual Testing Scenarios
  • Use this in new chat session.
  • Use this only when feature has multiple changes or complex that need a testing checklist .
  • Can skip if the feature is small or the cases to test are clear/obvious.
  • Can provide business context like PRD also for the agent to understand the feature better.
  • DO not use this to generate test cases scenarios for unit tests.

Gives a list of testing scenarios that need to be tested manually for the code changes.

Example:

Write the testing scenarios for the code changes in
[
    Last 1/2 commits
    or
    Current branch against development
    or 
    Specific files/modules
]
by following the guidelines in 
```
/what-to-test
```

/refactor — Code Refactoring
  • Use this in new chat session.
  • Use this when code needs to be refactored for readability.
  • This can be used after implement-with-plan or individually by pointing to the files/modules that need to be refactored.
  • If you have a very targeted refactoring or want to do a specific thing then need not use this.

Improves naming,readability, and organization of code without changing its behavior.

Example:

Refactor the code in
[
    Last 1/2 commits
    or
    Current branch against development
    or 
    Specific files/modules
]
by following the guidelines in 
```
/refactor
```
word-to-word.
- Do not change the behaviour of the code flow
- Focus only readability & maintainability not on logical issues or opiniated formatting
- Do not increase verbosity when it doesn't increase readability or the current version also conveys same meaning
- Identify complex code patterns and naming that hamper the readability.
- Strictly follow existing codebase patterns and when in doubt, do not update and convey it in summary.

/explain — Explain Code or Flow

Explains how a piece of code works, traces a request flow end-to-end, or clarifies an architectural decision. Use when you need to understand before changing.

Example:

Explain how the interview attempt creation flow works from the API to the DB
by following the guidelines in /explain

/visualize — Visualize Code or Flow

Visualizes a code structure, data flow, or architectural decision. Use when you need to understand the codebase or flow before making changes.

Example:

Explain the code structure of the interview attempt creation flow with /visualize
