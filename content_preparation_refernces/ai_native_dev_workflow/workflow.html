# Workflows

Common Mistakes

  • Giving conflicting instructions/context to the Agent that contradict what is written in the PRD
  • Giving irrelevant content like success metrics, DA team notes, or other task-irrelevant content to the Agent.
  • Missing explanation of frontend behavior when it's relevant to the task flow
  • If the PRD contains a lot of frontend sections and only part of the content is backend-relevant, giving the entire PRD to the Agent will cause it to process all frontend-relevant requirements that are not needed.
  • Not giving any extra architectural context or pointing to reference implementations when you already know the direction of implementation
  • Intentionally excluding context from the Agent to see if it figures it out from the codebase. (This causes it to spend more time on the task and reduces efficiency)
  • Not mentioning external integration request/response contracts beforehand. e.g., Sheet utils, IB Users, or other external API calls.
  • Not mentioning the task depends on any external service/package when you already know it.
  • Not mentioning when you already have a plan or implementation cases or flow

# Dev Workflow

# Note:

  • Based on the task, you can directly start with propose-approaches, write-tech-prd, or implement-with-plan.
  • If you already know the implementation approach, use implement-with-plan instead of first generating approach exploration docs.
  • If you already have a detailed plan, use implement directly.
  • If you need a more detailed plan after selecting an approach, use plan-approach before implement.
  • If the task is small or the changes are limited to a specific file or module, you can directly move to implement.
  • When multiple approaches are generated and one is selected, give only the chosen approach to the next command and remove sections like How this fits, Trade-offs, and Recommendation. These sections help with decision-making but should be removed to optimize context.
  • Use build-api before or alongside implement-with-plan when the task changes APIs enough that frontend/backend alignment is needed.
  • The journeys and business context are re-explained in Tech PRD and selected approach outputs to serve as additional context to the Agent.
  • When the task is not significant and does not require changes in user or app flow docs, you can skip documentation updates.
  • Attach app docs only when relevant. Do not add them for every minor or focused change, so the context stays lean.
  • If the changes are minimal and can be manually reviewed faster than using /review-impl, you can skip it.
