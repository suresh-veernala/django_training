      # Web Application - Revisited

      # Frontend & Backend

      An application generally has two parts - The frontend and the backend.

      Let’s take an analogy to understand them - Suppose you are sitting in a car, and the basic things that you can see and use are the brakes, clutch, accelerator, gear, etc. These are called the frontend components through which you can interact with the car. But, when you use any of the basic things, there is a series of actions happening behind the scenes which you are not aware of. Everything that is happening behind the scenes is known as the backend.

      In the same way, in any Web Application, you interact with the frontend. But, the processing of your request is done at the backend. Backend is the server-side architecture of the web.

      In short:

      # Frontend

      The front-end is everything involved with what the user sees, including design. It is usually built using languages like HTML, CSS, JavaScript, etc.,

      # Backend

      The backend (or “server-side”) is the portion of the website which you don’t see. It’s responsible for storing and organizing data and ensuring that everything on the client-side works. It is usually built using languages like Python, Go, etc.,

      In the above car example, whenever a user does an action i.e., apply brakes or press the accelerator, etc. This input or request is communicated to the appropriate system (which is behind the scene) to take an appropriate action.

      Similarly, all the user requests are sent to Backend by Frontend. In turn, Backend responds with a response if necessary. This is usually done by requesting an API.

      # Why should backend and frontend be separate?

      # Scalability

      It is easier to scale an application with a separate frontend and backend. As the code is divided into two parts, you can optimize the code faster. You can also increase the resources for front-end and back-end independently. Usually, back-end needs to be ramped up at a relatively faster pace as you grow.

      # Resource Optimization

      In a combined backend-frontend application, your servers do all the work, from parsing requests, retrieving relevant data from a database, applying calculations or amendments to the data based on what is requested and then putting it all together nicely in an HTML format using a template and sending it to the browser.

      Whereas, in an application with separate Backend and Frontend, servers just send raw data -- mostly in JSON format(you will learn more about it in the next sections) -- to the browser. The frontend part of the application handles this data and puts it together nicely and seamlessly.

      Servers need to work less and the user experience is better as the whole page need not be updated with every request. Win-win!

      # Easier Upgradation

      Upgrading the application framework can be a real pain. Keeping the back-end and front-end separate will lessen your chances of breaking the entire website during this process. It would be easier to debug as you would already know if it's a front-end issue or a back-end one.

      # Consolidation Of APIs

      In this ever-growing world of devices, you have more and more versions of your code to manage -- your website, iOS app, Android app, etc. In most cases, you're supplying the same information to all these applications. So, it only makes sense to serve them from the same code base. Having an API-based website makes developers' lives simpler. Having less code to deal with makes it more manageable.

      # Faster Deployment

      Since a front-end developer's work is not dependent that much on a back-end developer's work and vice-versa, their code can be tested and potentially deployed whenever they're done without waiting for the other person's completion.

      # Modularity

      If your front-end is separate from the back-end, it becomes easier to work on one module keeping the other one untouched. Modularity also makes it easier for two teams or two people to work on front-end and back-end simultaneously without worrying about overwriting or messing up other person's work. They can work at a different pace as well.

      # API

      (opens new window)

      API is an acronym that means: Application Programming Interface. It's the interface that is used by an application, typically used by a front-end application to communicate to the back-end application.

      APIs are methods and functions that wrap some operations/logic.

      We normally talk about APIs in the context of internet network applications, but API is a more generic term that describe interfaces between any kind of applications.
