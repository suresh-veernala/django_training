      # Guidelines

      • Don't repeat your self by writing the same specification again and again, use allOf keyword wherever it is required

        Example

        • Char fields with enums can be used in multiple definitions
        • Same Object definition can be used for multiple Responses/Parameters
        • Don't over optimize by convert each and every key in an object as individual definition, Separate only that is necessary.

      • Naming conventions

        • Definitions/Parameters/Responses name should be in Camel Case.

            Ex:
      UserProfile, not userProfile or user_profile or Userprofile

        • OperationId should explicitly convey the logic/behaviour of an API and use snake case.

            Ex:
      get_user_details, not GetUserDetails or UserDetails or user_details or user_data

        • Parameters/Responses should explicitly show which API they belong to, if possible. It is suggested to follow the below format.

            API: get_user_details
  PARAMETER: GetUserDetailsParameter
  RESPONSE: GetUserDetailsResponse

      • It is suggested to use appropriate HTTP METHODS.

      • Try avoiding reuse of Parameters and Responses until it is seriously needed.

      • Always append version to your API path as multiple versions of an API is a common scenario.

          Example:
      /users/details/v1/ is preferred over
      /users/details/

      • Use proper format for url path.

          Example:
      /user/login/v1/ - instead of /user_login/v1/
      /user/profile/v1/ - instead of /user/get_profile/v1/
      /user/profile/update/v1/ - instead of /user/update_profile/v1/

      • Each API should be given proper tags so that it would be ease the navigation.

          Example:
      All user related API's can be given user tag
      if
          group/register/v1/
          user/login/v1/
          user/v1/
          group/v1/
          user/v2/
          group/v2/
      are API's then
      user/login/v1/ can have user as tag. (as there is only one login api)
      user/register/v1/ can have user as tag. (as there is only one register api)
      user/v1/ can have user, details and v1 as tags.
      group/v1/ can have group, details and v1 as tags.
      user/v2/ can have user, details and v2 as tags.
      group/v2/ can have group, details and v2 as tags.

      # Checklist

      • Make sure that you use proper naming convention which any developer can explicitly understand.

      • Proper documentation is written for the schema fields, parameters and responses.

      • All required field are listed under required key.

      • Use enums whenever required.

      • Make sure that you added necessary oauth scopes for your API.

      • Make sure that all the possible fields are given default values.

      • Make sure that all related API's are written one after the other. This will improve readability of OAS.

      • Include all possible responses of an API i.e 400, 404, 417, 401, 403 etc. whatever the API might return. And add proper documentation.

      • Make sure that sensible descriptions, summary / documentation is provided for each of the Definition/API spec/Parameters/Responses.

      • Make sure that the order of keys for the parameters to be in the following order

        "name": ...
"description": ...
"in": ...
"required": ...
