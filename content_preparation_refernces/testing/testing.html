              # Unit Tests

              # What is Testing?

              Before discussing why testing is useful and how to do it, let's take a minute to define What unit testing actually is.

              "Testing", in general programming terms, is the practice of writing code (separate from your actual application code) that invokes the code it tests to help determine if there are any errors.

              It does not prove that the code is correct (which is only possible under very restricted circumstances). It merely reports if the conditions the tester thought of are handled correctly.

              Note: when we use the term "testing", we are always referring to "automated testing", where the tests are run by the machine. "Manual testing", where a human runs the program and interacts with it to find bugs, is a separate subject.

              # What kinds of errors/issues(in the code) can be identified during testing?

              • Syntax errors: These are the typos in the code or unintentional misuses of the language.

                • For example, the extra '.' in my_list..append(foo).

              • Logical errors: These are created when the algorithm is not correct. Here, in this case, algorithm refers to "the way the problem is solved or the logic behind the solution".

                • For example, a programmer might have forgot that Python is "zero-indexed" and tried to print the last character in a string using print(my_string[len(my_string)]). This will raise an IndexError.

              • Larger and more systemic errors can also be checked for.

                • For example, a program always crashes when the user inputs a number greater than 100, or hangs if the web object it's trying to retrieve is not available.

              All of these errors can be caught through careful testing of code. Unit testing, specifically tests a single "unit" of code in isolation. The size of a unit could be an entire module, a single class or function, or almost anything in between.

              What's important, however, is that the code, which we are planning to test, is isolated from rest of the code which we're not testing. It is important because this 'rest of the code' itself could have errors and would thus confuse test results.

              # Why Testing?

              Consider the following example:

              def is_prime(number):
    """Return True if *number* is prime."""
    for element in range(number):
        if number % element == 0:
            return False
    return True

              Before we continue with the example, it's important to ask the question - "Why is testing a valuable use of my time?"

              It's a fair question, and it's the question those unfamiliar with testing code often ask. After all, testing takes time that could otherwise be spent on writing application code, and isn't that the most productive thing to be doing?

              Tests assure you that your code works. They give you the confidence that all the changes that you add to the software are not creating any bugs.

              There are a number of valid answers to this question. We'll look at a few here:

              # To be sure that your code works

              Testing assures correctness under a basic set of conditions. Syntax errors will almost certainly be caught by running tests, and the basic logic of a unit of code can be tested to ensure the correctness under certain conditions.

              Again, it's not about proving that the code is correct under any set of conditions. We're simply aiming for a reasonably complete set of possible conditions. For example, you may write a test for what happens when you call my_addition_function(3, 'refrigerator), but you needn't test all possible integers/strings for each argument.

              # To be sure that changes to the code didn't introduce any bugs

              This is especially helpful when you are refactoring the code. Without tests in place, you have no assurance that your code changes did not break things (introduce bugs) that were previously working fine. If you want to be able to change or rewrite your code and know that you didn't break anything, proper unit testing is essential.

              # Forces you to think about the code

              Writing tests forces you to think about the non-normal conditions your code may encounter.

              For example, my_addition_function adds two numbers. A simple test of basic correctness would call my_addition_function(2, 2) and assert that the result was 4.

              Further tests, however, might test that the function works correctly with floats by running my_addition_function(2.0, 2.0).

              It is suggested that your code should be able to gracefully fail on invalid input, so you write tests to check that an exception is properly raised when strings are passed as arguments to the function.

              In essence, you are forced to think about different conditions under which your function/class/module may be used in. This makes the code more robust.

              # Summary

              Good testing requires modular, decoupled code, which is a hallmark of a good system design. The whole practice of unit testing is made much easier by code that is loosely coupled (less dependent on each other).

              Applications designed with test-ability in mind usually end up being more modular and loosely coupled out of necessity.
