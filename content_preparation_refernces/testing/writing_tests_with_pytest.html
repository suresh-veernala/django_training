              # Installation

              Use the following pip command to install pytest.

              (venv) cloud9$ pip install pytest

              That’s it, pytest is installed and we can move forward with writing tests using pytest.

              # Pytest syntax for writing tests

              • File names should start or end with "test".
                • Example: test_example.py or example_test.py
              • If tests are defined as methods in a class, the class name should start with "Test". Also, the class should not have an __init__ method.
                • Example: TestSampleClass
              • Test method names or function names should start with "test_".
                • Example: test_example

              When we follow these conventions, the test framework (pytest in this case) identifies the tests in a project and are run appropriately. This process is called as test discovery, which can be configured.

              WARNING

              Methods with names that don’t match this pattern won’t be executed as tests.

              # Writing tests

              Let's write test cases for the following use case.

              Use case: Write an 'is_prime' method which takes a number(not string) as input and 
returns True if the given number is prime, else returns False.

              In file primes.py, let’s write is_prime function as below.

              # my_app/primes.py
def is_prime(number):
    """Return True if *number* is prime."""
    for element in range(2, number):
        if number % element == 0:
            return False
    return True

              # Step 2: Write tests for each case

              # What to write in a test case?

              Each test should test a single, specific property of the code and should be named accordingly.

              Each Test case has 3 parts as mentioned below:

              • Arrange: In this part, we create/define all the necessary objects/variables required for the function that we are going to test.

                Example: To test accelerate method of a Car we need a car instance

              • Act - In this part, we call the function/class that is to be tested.

              • Assert - In this part, we check if the function/class (unit) has done its job as expected or not.

              # Assert Statement:

              By now you must be familiar with how to write code in Arrange & Act parts of the code.

              In the Assert part, we write assert statements.

              Python’s assert statement is a debugging aid(i.e tool built for debugging/testing code) that tests a condition. If the condition is true, it does nothing and your program just continues to execute. But if the assert condition evaluates to false, it raises an AssertionError exception with an optional error message.

              • Assert statement evaluated to true
              >>> assert 1 in [1, 2, 3]
              • Assert statement evaluated to false
              >>> assert 4 in [1, 2, 3]

              # Test Case

              Test methods should be prepended by test_ (this is configurable, but the purpose is to differentiate test methods and non-test utility methods)

              # my_app/tests.py
# make sure to start function name with test
def test_is_prime_for_prime_number_returns_true():
    # Arrange
    from .primes import is_prime
    number_to_check = 3
    
    # Act    
    result = is_prime(number_to_check)
    
    # Assert
    assert result is True

              # Fixing Code During Tests

    # Assert

              Now that the error is fixed, does that mean we should delete the test method (since clearly it will now always pass)? NO.

              Unit tests should rarely be deleted, as passing tests are the end goal. We've tested that the syntax of is_prime is valid and, at least in three cases (3, 4, 0), it returns the proper result. Our goal is to build a suite (a logical group of unit tests) of tests that all pass, though some may fail at first.

              # Fix Your Code Properly

              Let us write a test for Case 5

              def test_is_prime_for_negative_number_returns_false():
    # Arrange
    from .primes import is_prime
    number_to_check = -10

    # Act
    result = is_prime(number_to_check)

    # Assert
    assert result is False

              You might fix your is_prime by writing something like this

              def is_prime(number):
    """Return True if *number* is prime."""

    if number == 0:
        return False

    if number == 1:
        return False

    if number < 0:
        return False

    for element in range(2, number):
        if number % element == 0:
            return False
    return True

              The purpose of writing unit tests is NOT to blindly add code (several if conditions) to check for edge cases. After passing the test, stop and find the best way to update your code so that your tests are always passing.

              In this case, we can update the code as below:

              def is_prime(number):
    """Return True if *number* is prime."""

    if number <= 1:
        return False

    for element in range(2, number):
        if number % element == 0:
            return False
    return True

              # Summary

              1. Process
                1. Choose a function/class to be unit tested.
                2. List all the possible cases the function/class should handle
                  • List down the possible cases which are given in the question/usecase directly.
                  • Think of cases where your code could raise Exceptions.
                  • Think of edge cases
                    • Handling other data types, if there is any scope for such inputs
                    • Handling large or small inputs
                    • Handling empty list cases
                    • Handling Object does not exist cases
                    • None values
                    • etc.
              2. Arrange, Act & Assert parts for each test case.
              3. Naming File, class & functions properly so that pytest can identify them (test discovery).
              4. Using assert statements.
