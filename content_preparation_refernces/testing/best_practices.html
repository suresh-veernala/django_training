    # Characteristics of a good unit test - FIRST

    • Fast: It is not uncommon for mature projects to have thousands of unit tests. Unit tests should take very little time to run. It should be in the order of Milliseconds.

    • Isolated: Unit tests are standalone. We should be able to run in isolation, and have no dependencies on any outside factors such as a file system or a database.

    • Repeatable: Running a unit test should be consistent with its results. That is, it should always return the same result if you do not change anything in between different runs.

    • Self-Checking: The test should be able to automatically detect if it is passed or failed without any human interaction.

    • Timely: A unit test should NOT take a disproportionately long time to write compared to the code being tested. If you find that writing the test code is taking a large amount of time compared to writing the application code, you should change your application design such that it is more testable.

    # Best Practices To Follow

    # Naming your tests

    The name of your test should consist of three parts:

    • The name of the method being tested.
    • The scenario under which it's being tested.
    • The expected behavior when the scenario is invoked.

    # Why?

    • Naming standards are very important because they explicitly express the intent of the test.

    Tests are more than just making sure your code works, they also provide documentation. Just by looking at the suite of unit tests, you should be able to infer the behavior of your code without even looking at the code itself. Additionally, when tests fail, you should be able to see exactly which scenarios do not meet your expectations.

    Bad:

    
def test_single():
    string_calculator = StringCalculator()
    actual = string_calculator.add('0')

    assert 0 == actual

    Better:

    
def add_with_single_number_returns_same_number():
    string_calculator = StringCalculator()
    actual = string_calculator.add("0")

    assert 0 == actual

    # Arranging your tests

    Arrange, Act, Assert is a common pattern used in unit testing. As the name implies, it consists of three main actions:

    • Arrange your objects, creating and setting them up as necessary.
    • Act on an object.
    • Assert that something is as expected.

    # Why?

    • Clearly separates what is being tested from the arrange and assert steps.
    • Less chance to intermix assertions with "Act" code.

    Readability is one of the most important aspects when writing a test. Separating each of these actions within the test clearly highlights the dependencies required to call your code, how your code is being called, and what you are trying to assert. While it may be possible to combine some steps and reduce the size of your test, the primary goal is to make the test as readable as possible.

    Bad:

    
def add_empty_string_returns_zero():
    # Arrange
    string_calculator = StringCalculator()

    # Assert
    assert 0 == string_calculator.add("")

    Better:

    
def add_empty_string_returns_zero():
    # Arrange
    string_calculator = StringCalculator()

    # Act
    actual = string_calculator.add("")

    # Assert
    assert 0 == actual

    # Write minimally passing tests

    The input to be used in a unit test should be the simplest possible in order to verify the behavior that you are currently testing.

    # Why?

    • Tests become more resilient to future changes in the codebase.
    • Testing code will be closer to 'testing the behavior' over 'testing the implementation'.

    Tests that include more information than that is required to pass the test have a higher chance of introducing errors into the test. They will make the intent of the test less clear.

    When writing tests, you want to focus on the behavior. Setting extra properties on models or using non-zero values when not required, only distracts from what you are trying to test.

    Bad:

    
def add_single_number_returns_same_number():
    string_calculator = StringCalculator()
    
    actual = string_calculator.add("42")

    assert 42 == actual

    As our focus is to test the add function with a single parameter, using "42" as an input without any significance for it, only distracts us from what we are trying to test. We could instead use the simplest possible input. i.e, "0".

    Better:

    
def add_single_number_returns_same_number():
    string_calculator = StringCalculator()
    
    actual = string_calculator.add("0")

    assert 0 == actual

    # Avoid magic strings

    Naming variables in unit tests is as important (if not more) as naming variables in production code. Unit tests should not contain magic strings. Magic strings are the strings which do not explicitly convey their significace in the context of the test case. (look at the example for a better understanding)

    Why?

    • Prevents the need for the reader of the test to inspect the production code in order to figure out what makes the value special.
    • Explicitly shows what you're trying to prove rather than trying to accomplish.

    Magic strings can cause confusion to the reader of your tests. If a string looks out of the ordinary, they may wonder why a certain value was chosen for a parameter or return value. This may lead them to take a closer look at the implementation details, rather than focus on the test.

    TIP

    When writing tests, you should aim to express as much intent as possible. In the case of magic strings, a good approach is to assign these values to constants.

    Bad:

    
def add_big_number_throws_exception():
    string_calculator = StringCalculator()
        
    with pytest.raises(OverflowException):
        string_calculator.add("1001")

    Here, just by looking at the test case, a developer can not understand the significance of the string "1001". So, we can assign it to a variable with proper name which reveals the intent of it.

    Better:

    def add_maximum_sum_result_throws_overflow_exception():
    string_calculator = StringCalculator()
    MAXIMUM_RESULT = "1001"
    
    with pytest.raises(OverflowException):
        string_calculator.add(MAXIMUM_RESULT)

    This is not only applicable for strings, the same concept applies for any data type.

    # Avoid logic in tests

    When writing your unit tests, avoid manual string concatenation and logical conditions such as if, while, for, switch, etc.

    Why?

    • Less chance to introduce a bug inside of your tests.
    • Focus on the end result, rather than implementation details.

    When you introduce logic into your test suite, the chance of introducing a bug into it increases dramatically. The last place that you want to find a bug is within your test suite. You should have a high level of confidence that your tests work, otherwise, you will not trust them. Tests that you do not trust, do not provide any value. When a test fails, you want to have a sense that something is actually wrong with your code and that it cannot be ignored.

    TIP

    If logic in your test seems unavoidable, consider splitting the test into two or more different tests.

    Bad:

    
def add_multiple_numbers_returns_correct_results():
    string_calculator = StringCalculator()
    expected = 0
    test_cases = [
        "0,0,0",
        "0,1,2",
        "1,2,3"
    ]

    for test in test_cases:
        assert expected == string_calculator.add(test)
        expected += 3

    Better:

    
@pytest.mark.parametrize("input, expected", [("0,0,0", 0), ("0,1,2", 3), ("1,2,3", 6)])
def add_multiple_numbers_returns_sum_of_numbers(input: str, expected: int):
    string_calculator = StringCalculator()
    
    actual = string_calculator.add(input)

    assert actual == expected

    # Avoid multiple asserts

    When writing your tests, try to include only one Assert per test. Common approaches which can be used to reduce assert statements in tests include:

    • Creating a separate test for each assert.
    • Using parameterized tests.

    Why?

    • If one Assert fails, the subsequent Asserts will not be evaluated.
    • Ensures that you are not handling multiple cases in a single test.
    • Gives you the entire picture as to why your tests are failing.

    When introducing multiple asserts into a test case, it is not guaranteed that all of the asserts will be executed. In most unit testing frameworks, once an assertion fails in a unit test, the proceeding tests are automatically considered to be failing. This can be confusing as the functionality that is actually correct, could be shown as failing.

    Note

    A common exception to this rule is when asserting against an object. In this case, it is generally acceptable to have multiple asserts against each property to ensure that the object is in the state that you expect it to be in.

    Bad:

    def add_edge_cases_throws_argument_exceptions():
    string_calculator = StringCalculator()
        
    with pytest.raises(ArgumentException):
        string_calculator.add(None)
    with pytest.raises(ArgumentException):
        string_calculator.add("a")

    Better:

    @pytest.mark.parametrize("input", [None, "a"])
def add_input_null_or_alphabetic_throws_argument_exception(input: str):
    string_calculator = StringCalculator()
    
    with pytest.raises(ArgumentException):
        string_calculator.add(input)

    # Validate private methods by unit testing the public methods which use them

    In most cases, there should be no need to test a protected or private method. Protected(or Private) methods are an implementation detail. You can think of it this way: protected(or private) methods never exist in isolation. At some point, there is going to be a public facing method that calls the protected(or private) method as a part of its implementation. What you should care about is the end result of the public method that calls the protected(or private) method.

    Consider the following case

    
class Parer:
    def parse_log_line(input: str)->str:
        sanitized_input = self._trim_input(input)
        return sanitized_input

    def _trim_input(input: str)->str:
        return input.strip()

    Your first reaction may be to start writing a test for _trim_input, because you want to make sure that the method is working as expected. However, it is entirely possible that parse_log_line manipulates sanitized_input in such a way that you do not expect that running a test against _trim_input is useless.

    So, the real test should be done against the public facing method parse_log_line, because that is what you should ultimately care about.

    def parse_log_line_by_default_returns_trimmed_result():
    parser = Parser()

    result = parser.parse_log_line(" a ")

    assert "a" == result

    With this viewpoint, if you see a private method, find the public method and write your tests against that method. Just because a private method returns the expected result, it does not mean that the system which calls the private method behaves correctly.

    # Stub static references

    One of the principles of a unit test is that it must have full control of the system under test. This can be problematic when production code includes calls to static references (e.g. datetime.weekday). Consider the following code:

    def get_discounted_price(price: int)->int:
    import datetime
    now = datetime.datetime.now()

    if now.weekday() == 1: # Checking for tuesday
        return price / 2
    else:
        return price

    How can this code be unit tested? You may try an approach such as

    def getdiscountedprice_bydefault_returns_full_price():
    price_calculator = PriceCalculator()

    actual = price_calculator.get_discounted_price(2)

    assert 2 == actual

def get_discounted_price_on_tuesday_returns_half_price():
    price_calculator = PriceCalculator()

    actual = price_calculator.get_discounted_price(2)

    assert 1 == actual

    Unfortunately, you will quickly realize that there are a couple of problems with your tests.

    • If the test suite is run on a Tuesday, the second test will pass, but the first test will fail.
    • If the test suite is run on any other day, the first test will pass, but the second test will fail.

    To solve these problems, you'll need to introduce a seam

    def get_discounted_price(price: int, date_time_object)->float:
    if date_time_object.weekday() == 1:
        return price / 2
    else:
        return price

    Your test suite now becomes

    def getdiscountedprice_bydefault_returns_full_price():
    price_calculator = PriceCalculator()
    mock_date_time_object = Mock()
    mock_date_time_object.weekday.return_value = 4

    actual = price_calculator.get_discounted_price(2, mock_date_time_object)

    assert 2 == actual

def get_discounted_price_ontuesday_returns_half_price():
    price_calculator = PriceCalculator()
    mock_date_time_object = Mock()
    mock_date_time_object.weekday.return_value = 1

    actual = price_calculator.get_discounted_price(2, mock_date_time_object)

    assert 1 == actual

    Now the test suite has full control over weekday method and can stub any value when calling into the method.
