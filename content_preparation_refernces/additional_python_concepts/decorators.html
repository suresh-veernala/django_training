    # Background: Functions are first-class objects

    # Assigning a function to a variable

    In Python, functions are first-class objects. That means, functions can be assigned to a variable, just like we assign numbers, strings, lists, etc.,

    def square(number):
    return number * number

    Let's assign the square function to a variable and then call the function using the variable it is assigned to.

    >>> x = 10
>>> y = x
>>> f = square
>>> f(x)
100

    # Passing a function as an argument

    We can also pass a function as an argument to any other function.

    # Code

    >>> def call_function(func):
...    value = 3
...    return func(value)

>>> def square(number):
...    return number ** 2
>>> def cube(number):
...    return number ** 3
>>> call_function(square)
9
>>> call_function(cube)
27

    # Returning a function

    A function can also return another function, just like returning any other python object i.e, integer, string, list etc.

    
>>> def square(number):
...    return number ** 2

>>> def get_function_for_operation(operation):
...    def cube(number):    # A function inside another function
...         return number ** 3
...    if operation == "SQUARE":
...         return square
...    elif operation == "CUBE":
...         return cube
>>> func = get_function_for_operation("SQUARE")
>>> func
<function __main__.square(number)>
>>> func(3)
9
>>> func = get_function_for_operation("CUBE")
>>> func
<function __main__.get_function_for_operation.<locals>.cube(number)>
>>> func(3)
27

    # Hard to Reuse

    In a web application, when interacting with another service over HTTP, the APIs might work correctly 99.9% of the time. But, a small fraction of the requests will cause the server to return an internal error, and you have to retry the request. In that case, you might implement some retry logic as shown in the below snippet.

    resp = None
while True:
    resp = make_api_call()  # makes an api call to the external service.
    if resp.status_code == 500 and tries < MAX_TRIES:
        tries += 1
        continue
    break
process_response(resp)

    Now imagine that you have dozens of functions like make_api_call(), and they are called all over the codebase.

    • Are you going to implement that while loop everywhere?
    • Are you going to do it again and again, every time you add a new API-calling function?

    This kind of pattern creates a need for some kind of boilerplate code.

    # Analytics and Logging

    Especially with large applications, we often need to specifically measure what’s going on, and record metrics that quantify different activities. So, we might have to write logic at every function to measure the time and other required metrics.

    # Validation and run time checks

    You might want to enforce sophisticated custom checks on data going in or out. For example, imagine that you have a set of functions, each returning a dictionary, which (among other fields) includes a field called “summary.” The value of this summary must not be more than 80 characters long; if violated, that’s an error.

    In python, 'decorators' can easily handle all of the above problems while making the code more readable.

    # Decorator

    A decorators in Python allows the developers to wrap new functionality around an existing function/method/class. A decorator takes a function as an argument, performs certain tasks before and after calling it and returns the results.

    # Introduction

    Consider the sum function mentioned below.

    def sum(number_one, number_two):
    return number_one + number_two

    Suppose, for logging purposes, we need to log the function name, inputs and outputs of each of the function call. We could do something like this

    def sum(number_one, number_two):
    print("Call to {}".format(sum))
    print("Args: {}", (number_one, number_two))
    
    result = number_one + number_two

    print("Return: {}".format(result))
    return result

    Some of the problems with this solution are:

    • It make your code ugly, filled with lot of print statements.
    • It introduces lot of duplicate code as you need to write the same print statements in all the functions.
    • The above example is a simplistic scenario, but to do the same with functions involving complex logic, it'll become even harder.

    A better approach could be to use a function to remove the duplication.

    
def log_function_call(func, *args, **kwargs):
    print("Call to {}".format(func))
    print("Args: {}", args)
    print("Kwargs: {}", kwargs)
    
    result = func(*args, **kwargs)

    print("Return: {}".format(result))
    return result

def sum(number_one, number_two):
    return number_one + number_two

    >>> log_function_call(sum, 1, 2)
3

    This looks like a cleaner approach than the previous one, as we need not add print statements in every function that we have to log.

    The problems with this solution are: <!-- Why should we do this?

    • For every new wrapping logic we introduce, we have to change all the references (all the places where this function is being called to enable this wrapping) -->
    • With such a solution, we might find calls to the log_function_call function everywhere in the code, which makes the code very hard to read.
    • Any developer who would like to use the function sum should explicitly check if there are any implementations (like log_function_call) that wrap the sum function. This has to be done to make sure that the changes are consistent with rest of the codebase.
    • Applying more than one such wrapping functions is clumsy and hard to maintain. If some_wrapping_logic_1, some_wrapping_logic_2, etc., are some functions just like log_function_call. Then applying multiple wrappers would look like below.
    >>> log_function_call(sum, 1, 2)
>>> some_wrapping_logic_1(log_function_call, sum, 1, 2)
>>> some_wrapping_logic_2(some_wrapping_logic_1, log_function_call, sum, 1, 2)
...

    This makes the code even harder to read and understand.

    In the above implementation of log_function_call, apart from addding logging functionality, it is calling the input function and returning the result. But, what if log_function_call just wraps the logging functionality around the input function and returns another function?

    def log_function_call(func):
    def wrapper(*args, **kwargs)
        print("Call to {}".format(func))
        print("Args: {}", args)
        print("Kwargs: {}", kwargs)
        
        result = func(*args, **kwargs)
    
        print("Return: {}".format(result))
        return result
    return wrapper

def sum(number_one, number_two):
    return number_one + number_two

    >>> log_function_call(sum)(1, 2)

    This can also be written as

    @log_function_call
def sum(number_one, number_two):
    return number_one + number_two

    >>> sum(1, 2)

    Decorator is just a syntactic sugar. The @decorator syntax is a shorthand representation. So, with this approach we need not make changes to the rest of the code and can easily wrap existing functions with new behavior.

    # Solution to the above problems

    # Hard to Reuse

    Recall the retry problem mentioned in hard to reuse section

    resp = None
while True:
    resp = make_api_call()  # makes an api call to the external service.
    if resp.status_code == 500 and tries < MAX_TRIES:
        tries += 1
        continue
    break
process_response(resp)

    Using decorators, the solution to this problem would be as simple as writing a retry decorator as mentioned below.

    # The decorated function returns a Response object, which has a status_code attribute.
# 200 means success; 500 indicates a server-side error.
def retry(func):
    def retried_func(*args, **kwargs):
        MAX_TRIES = 3
        tries = 0
        while True:
            resp = func(*args, **kwargs)
            if resp.status_code == 500 and tries < MAX_TRIES:
                tries += 1
                continue
            break
        return resp
    return retried_func

# This gives you an easy-to-use @retry decorator:
@retry
def make_api_call():
    # ....

    # Multiple Decorators

    We can use multiple decorators for a function

    # Example

    # Decorator Implementation
    def print_function(func):
    def wrapper(*args, **kwargs):
        print("Call to decorator print_function")
        result = func(*args, **kwargs)
        return result
    return wrapper

def print_time(func):
    def wrapper(*args, **kwargs):
        import datetime
        print(datetime.datetime.now())
        result = func(*args, **kwargs)
        return result
    return wrapper

@print_function
@print_time
def print_hello_world():
    print("Hello World")

print_hello_world()

    The decorators are executed in the order they’re listed. In this case, @print_function calls @print_time, which calls print_hello_world().

    Call to decorator print_function
2020-05-10 12:59:58.086679

    # Decorating Methods in Class

    # Built-in class decorators

    Some commonly used decorators that are available in Python by default are:

    • @classmethod,
    • @staticmethod,
    • @property.

    Decorating a method in a class is same as decorating a function.

    # Example

    Let's apply @print_time decorator to the print_hello_world method in MyClass given below.

    # Code
    def print_time(func):
    def wrapper(*args, **kwargs):
        import datetime
        print("Executed at : ", datetime.datetime.now())
        result = func(*args, **kwargs)
        return result 
    return wrapper
    

class MyClass:
    
    @print_time
    def print_hello_world(self):
        print("Hello World")


my_class_object = MyClass()
my_class_object.print_hello_world()

    # Result

    Executed at :  2020-05-10 13:09:24.929451
Hello World

    # Checkpoints while writing decorators

    • Ensure that you are writing custom logic in appropriate location. i.e., before or after the function call
    def decorator(func):
    def wrapper(*args, **kwargs):
        # Logic before the actual function call

        result = func(*args, **kwargs)

        # Logic after the actual function call

        return result
    return wrapper
    • Ensure that your decorator is accepting arguments
    def decorator(func):
    def wrapper(*args, **kwargs): <- Should write *args and **kwargs here so that any input to the original function can be accepted

        result = func(*args, **kwargs)  

        return result                   
    return wrapper
    • Ensure that you are passing the arguments to the func function
    def decorator(func):
    def wrapper(*args, **kwargs): 

        result = func(*args, **kwargs)  <- Should pass the arguments to the func function

        return result                   
    return wrapper
    • Ensure that you are returning the outcome of the func function call.
    def decorator(func):
    def wrapper(*args, **kwargs):

        result = func(*args, **kwargs)

        return result           <- Should return the func function call outcome.        
    return wrapper
