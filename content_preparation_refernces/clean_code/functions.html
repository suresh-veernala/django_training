      # Writing Clean Code Part II - Functions

      A professional software developer should provide a rational justification for every line of code they write. These set of rules, which are suggested by many senior software professionals, provide you with concrete activities, patterns, and techniques to keep your code clean.

      # Small!

      "The first rule of functions is that they should be small. The second rule of functions is that they should be smaller than that."

      Lengthy functions are tough to understand. They are also more prone to bugs which makes them very difficult to work with.

      Code Guideline

      "Functions should be small: each function shouldn't cross 20 lines (with a hard limit of 30 lines for exceptional cases). Each line shouldn't cross 80 characters"

      # Blocks and Indenting

      Code Guideline

      The indent level of a function should not be greater than three. (it can be four only in some exceptional cases)

      # Flag Arguments

      Each function should do 'only one thing' which is clearly explained by the name of the function.

      # Example 1

      # Ugly Code

      
def register_user():
    
    user = create_user()
    
    should_send_email = user.email is not None 
    should_send_sms = user.phone_number is not None

    email_template=None
    sms_template=None
    
    if should_send_email:
        email_template = "...."
    
    if should_send_sms:
        sms_template = "...."

    send(user, email_template=email_template, sms_template=sms_template, email=should_send_email,sms=should_send_sms) 

def send(user, email_template=None, sms_template=None, email=False, sms=False):
    if email:
        # send email logic
    if sms:
        # send sms logic

      Here, send function does multiple things. It sends email and also sends sms. All the logic is written within a single function, which is a very bad practice.

      The logic is handled by passing a boolean argument into the function. Passing boolean arguments is a truly terrible practice. It immediately complicates the signature of the method, loudly proclaiming that this function does more than one thing. It does one thing if the flag is true and another if the flag is false!

      Code Guideline

      "Don't use flag (boolean, enums, integers) arguments for functions. Flag arguments loudly declare that the function does more than one thing. It is better to have many functions than to pass a variable into a function to select the behavior."

      # Command-Query Separation

      Either your function should change the state of an object, or it should return some information about that object.Doing both often leads to confusion

      # Example1

      # Ugly Code

      def set(attribute: str, value: str):
    pass

      set function sets the value of a named attribute and returns true if it is successful and false if no such attribute exists. This leads to odd statements like this:

      
if (set(”username”, ”admin”)):
    ...

      Imagine this from the point of view of the reader. What does it mean? Is it asking whether the “username” attribute was previously set to “admin”? Or is it asking whether the “username” attribute was successfully set to “admin”? It’s hard to infer the meaning from the call because it’s not clear whether the word “set” is a verb or an adjective.

      We could try to resolve this by renaming the set function to setAndCheckIfExists, but that doesn’t much help the readability of the if statement

      # Clean Code

      if (attribute_exists(”username”)) {
     set_attribute(”username”, ”admin”);
   }

      Code Guideline

      Separate the command from the query so that ambiguity doesn't occur
