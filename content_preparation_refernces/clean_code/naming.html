    # Writing Clean Code Part I - Naming

    A professional software developer should provide a rational justification for every line of code they write. These set of rules, which are suggested by many senior software professionals, provide you with concrete activities, patterns, and techniques to keep your code clean.

    # Using Intention-Revealing Names

    Name of a variable/function/class should convey the intent of it i.e., When reading the name of a variable/function/class, everyone should understand the same intent for which it is written.

    # Example 1:

    # Ugly Code:

    d = 10

    The above code snippet is from an application, where developer is supposed to store the number of students registered in a variable. Clearly, this intent is not being conveyed in the name d.

    # Clean Code:

    number_of_students_registered = 10

    In clean code, you are able to understand the intent of the variable even without any comments.

    Tip

    If a name requires a comment, then the name does not reveal its intent.

    # Example 2:

    # Ugly Code:

    file_age = 10

    Here file_age is trying to convey that it contains the age of a file. For a new developer, it's impossible to figure out if this number corresponds to seconds or days or years without any other help.

    # Clean Code:

    file_age_in_days = 10

    Code Guideline

    Choose a name that specifies what is being measured and the unit of that measurement.

    # Example 3

    # Ugly Code:

    def f(l):
    b = []
    for a in l:
        if x(a) is True:
            b.append(a)
    return b

def x(k):
    if k % 2 != 0:
        return False
    else:
        return True

    After looking at the above code snippet, try to answer the following questions?

    • What does f do?
    • What is l? a list, a dictionary or a string?
    • what does f return?

    You would have taken some time to figure out answers to those questions. But, with clean code, you don't even have to think about it. You can just answer them in a snap.

    # Clean Code:

    def filter_odd_numbers_from_list(numbers):
    odd_numbers = []
    for number in numbers:
        if is_odd_number(number):
            odd_numbers.append(number)
    return odd_numbers

def is_odd_number(number):
    if number % 2 != 0:
        return False
    return True

    Code Guideline

    The name of a variable, function, or class, should answer all the big questions. It should tell you

    • Why it exists
    • What it does
    • How it is used

    # Example 4

    # Ugly Code:
    class c:
    def __init__(self, name, age, gender):
        self.name = name
        self.age = age
        self.gender = gender

    # Clean Code:

    class Voter:
    def __init__(self, name, age, gender):
        self.name = name
        self.age = age
        self.gender = gender

    Note:

    Choosing good names takes time, but saves more than what it takes.

    # Avoid Disinformation

    Programmers must avoid leaving false clues that hide the actual meaning of the code.

    # Example 1:

    # Ugly Code

    accounts_list = {"account_id": 1, "balance": 1000 ...}
user = User.objects.all()
fruit = ["apple", "banana" ...]

    Here, the names user and fruit suggest that they are a single object. They are hiding the actual facts about the code.

    # Clean Code

    account_dict = {"account_id": 1, "balance": 1000 ...}
users = User.objects.all()
fruits = ["apple", "banana" ...]

    Code Guideline

    Use appropriate names. Use plural in case of group of objects. Use suffixes like _dict, _list, etc. to properly convey the intent.

    # Example 2

    # Ugly Code

    XYZControllerForEfficientHandlingOfStrings
XYZControllerForEfficientStorageOfStrings

    If two variables are named as shown above, during the development, there is a high chance that you could use one variable instead of the other.

    # Clean Code

    XYZControllerForEfficientStringsHandling  <- variation at the ending  
XYZControllerForEfficientStringsStorage

    Having a variation at the end makes it easy to check if you are using the correct variable or not.

    Code Guideline

    For names which vary in small ways - Put the variation at the beginning or ending to differentiate easily. This will help in avoiding common mistakes like referring to wrong variables.

    # Make Meaningful Distinctions

    Sometimes, when you can’t use the same name to refer to two different things in the same scope, you might be tempted to change one name in an arbitrary way.

    # Example 1

    # Ugly Code

    quiz = Quiz.objects.get(id=1)
kwiz = Quiz.objects.get(id=2)

    Here, a reader can not make a difference between quiz and kwiz. These names are not revealing the intent of the developer.

    # Clean Code

    django_quiz = Quiz.objects.get(id=1)
python_quiz = Quiz.objects.get(id=2)

    Rule of Thumb

    You shouldn't create a variable named kar just because the name car was used for something else.

    # Example 2

    # Ugly Code

    def sum(number_one, number_two):
    return number_one + number_two

def is_subset(list_one, list_two): 
    is_subset_flag = True
    for item in list_one:
        if item not in list_two:
          is_subset_flag = False
    return is_subset_flag

    Having number_one and number_two might be ok in case of sum as there is nothing more to convey.

    But in case of is_subset, list_one and list_two don't convey the complete meaning i.e., a reader doesn't understand which one is subset and which one is superset without analyzing the code by himself.

    # Clean Code

    
def sum(number_one, number_two):  # intended 
    return number_one + number_two

def is_subset(subset_list, superset_list): 
    is_subset_flag = True
    for item in subset_list:
        if item not in superset_list:
            is_subset_flag = False
        elif subset_list.count(item) != superset_list.count(item):
            is_subset_flag = False
    return is_subset_flag

    Rule of Thumb

    Don't use number-series naming (a1, a2, .. aN), unless they are intended.

    # Example 3

    # Ugly Code

    product
product_details
product_info

    Here the variable product is indistinguishable from product_details and product_info i.e., a reader won't be able to distinguish between them. A new developer can not know what each of the variables represent without additional help.

    # Clean Code

    product
product_image_details
product_complete_details

    Rule of Thumb

    Don't use noise words (ex: [data, info]) for distinction. Ex: Don't use StudentInfo and StudentData as different class/variable names.

    # Use Pronounceable Names

    Using pronounceable names makes the code more readable.

    # Example 1

    # Ugly Code

    
genymdhms = ....
modymdhms = ....

    Here genymdhms is used to represent (generation date, year, month, day, hour, minute, and second) and similarly modymdhms (modified date, year, month, day, hour, minute, and second)

    New developers need to have the variables explained to them, and even then they will be speaking in silly made-up words instead of using proper English terms.

    # Clean Code

    generation_timestamp = ...
modified_timestamp = ...

    Code Guideline

    Make your variable names pronounceable Ex: There should not be variable name like genymdhms (generation date, year, month, day, hour, minute, and second)
