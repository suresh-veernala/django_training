      # Git Commands Cheatsheet

      # Basic Commands

      # init

      Creates an empty Git repo.

      $ git init
Initialized empty Git repository in ...

      # clone

      git clone <repo> - Clones repo located at onto your local machine.

      $ git clone https://github.com/...
Cloning into 'xxxxx-project'...
remote: Enumerating objects: 47, done.
remote: Counting objects: 100% (47/47), done.
remote: Compressing objects: 100% (29/29), done.
remote: Total 47 (delta 10), reused 45 (delta 8), pack-reused 0
Unpacking objects: 100% (47/47), done.

      # status

      Lists all the files which are staged, unstaged, and untracked.

      $ git status
Changes to be committed:
  (use "git reset HEAD <file>..." to unstage)

        new file:   file_one.txt

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in working directory)

        modified:   README.md

Untracked files:
  (use "git add <file>..." to include in what will be committed)

        venv
        some_file.txt

      # add

      Adds changes to the staging area for next commit.

      • You can add multiple folders/files to the staging area at once.
      $ git add <path1> <path2> ...
      • git add -u can be used to stage changes in already tracked files.
      • git add -p can be used to stage chunks of code.

      # commit

      Creates a commit for staged changes. When creating a commit, we should add a message which describes the changes in the commit.

      $ git commit -m "<message>"

      Commit message helps you to get an overview of the changes in the commit without even looking into the details of the changes in the commit.

      # pull

      Fetches commits of the current branch from the remote repository which aren't in your local repository.

      $ git pull

      # push

      Pushes all the commits and objects to remote repository

      $ git push

      # Branches

      # branch

      Lists all the branches in your repository.

      $ git branch
feature/feature-one
feature/feature-two
feature/oauth
feature/rest
* master

      • Current branch is indicated with "*".

      • git branch <branch> - argument can be used to create a new branch with the name as .

      $ git branch feature/feature-three

      $ git branch
feature/feature-one
feature/feature-two
feature/feature-three
feature/oauth
feature/rest
* master
      • A new branch feature/feature-three is created, but your are still on master branch.
      • It is suggested to use "-" instead of "_" in the branch name to avoid conflict with file name during checkout.

      # checkout

      • git checkout - Switch to an an existing branch.
      $ git checkout feature/feature-one
Switched to branch 'feature/feature-one'

      $ git branch
* feature/feature-one
feature/feature-two
feature/feature-three
feature/oauth
feature/rest
master
      • git checkout -b <branch> - Create a new branch and switch to the newly created branch.
      $ git checkout -b feature-four
Switched to branch 'feature/feature-four'

      $ git branch
feature/feature-one
feature/feature-two
feature/feature-three
* feature/feature-four
feature/oauth
feature/rest
master

      # push

      Pushes the newly created branch to (remote repository), along with any necessary commits and objects. Creates named branch in the remote repo if it doesn’t exist.

      $ git push -u <remote> <branch>

      # merge

      git merge <branch> - Merges <branch> into the current branch.

      $ git merge feature/feature-one

      # stash

      git stash command takes your uncommitted changes (both staged and unstaged), saves them away for later use, and then reverts them from your working copy

      $ git stash
Saved working directory and index state WIP on master: 5002d47 our new homepage
HEAD is now at 5002d47 our new homepage

      git stash apply applies the latest stashed changes to your working directory.

      $ git stash apply

      # Other Concepts

      # gitignore

      Git sees every file in your working copy as one of three things:

      • tracked - a file which has been previously staged or committed;
      • untracked - a file which has not been staged or committed; or
      • ignored - a file which Git has been explicitly told to ignore.

      Ignored files are tracked in a special file named .gitignore that is checked in at the root of your repository. There is no explicit git ignore command: instead the .gitignore file must be edited and committed by hand when you have new files that you wish to ignore

      # remote

      remote is used to manage set of remote repositories

      • git remote add origin url - to add a remote repository to your local repository
      • git remote remove <name> - unlink a remote repository

      # Best Practices

      Below are some suggested practices to help avoid most of the common mistakes during development.

      # Create new branch from right source branch

      Before starting a new feature check with your RP from which branch you need to create a new branch.

      # Code on the right branch

      Before writing any code, check if you are on the right branch.

      # Code on latest changes only

      Start coding if your local repository is up to date with remote, i.e. if your working branch in the local repository has all the commits from the remote repository.

      # Proper branch naming

      Ensure you are following proper branch naming conventions.

      # Review your changes

      Before adding changes the staging area/making a commit, use your IDE/Editor VCS feature to ensure if you are adding only required changes. It is a must for a developer to cross-check every line of change before making a commit.

      # Appropriate commit message

      Before creating a commit, ensure that your commit message is conveying the changes.

      # Commit frequently

      Commit your changes frequently.

      # Stash/Commit Before Pull

      Before pulling remote changes, ensure that your local changes are either committed or stashed.

      # Pull Before Push

      Before pushing any changes to remote, pull commits from remote repository.

      # Push Frequently

      Push your local commits to remote repository often.

      # Stash/Commit Before Checkout

      Before switching to a different branch ensure that you either commit or stash your local changes.

      # Best Practices for Better Commit Message

      Good commit messages serve at least three important purposes:

      • To speed up the reviewing process.
      • To help us write a good release note.
      • To help the future developers of the project, say five years into the future, to find out why a particular change was made to the code or why a specific feature was added.

      # Multi-line commit messages:

      • Short (50 chars or less) summary of changes. Don't end the summary line with a period - it's a title and titles don't end with a period.
      • If needed, you can add more information as Bullet points. Typically a hyphen or asterisk(preferably hyphen) is used for the bullet, preceded by a single space, but conventions may vary.
      • Write what you have done in the imperative mood that is as if you were commanding someone. Start the line with "Fix", "Add", "Change" instead of "Fixed", "Added", "Changed".
      • Always leave the second line blank in case of multi-line messages.

      Example:

      Add changes story EP001
 
- Add interactor, presenter, storages & testcases for get_user_posts API.
- Fix incorrect reactions count bug.
- Remove replies from get_post interactor.
