    # Basic DB Performance Optimization in Django

    You will be learning the basics of performance optimization related to databases in this tutorial.

    WARNING

    Results will vary depending on your hardware, database type, models, and the nature of your data.

    # What happens when you evaluate a queryset?

    Below are the broad steps involved

    1. Django Application sends the SQL query to the database (typically in other server located in some other location)
    2. Database executes the query
    3. Data is sent back to the Django Application
    4. Data is converted to Model instance(s) by Django Application (depending on the query)

    Let T1, T2, T3, & T4 be the time taken at each step respectively. So roughly, the total time required for a queryset to evaluate = T1 + T2 + T3 + T4

    Note

    T1 & T3 depend on the location of the database, T3 also depends on the amount of data that is returned from the database (this is similar to downloading a file on the internet, larger the file more the time)

    We will use the below models for the examples that follow.

    from django.db import models

class Library(models.Model):
    name = models.CharField(max_length=200, default='')
    address = models.CharField(max_length=200, default='')

    name = models.CharField(max_length=200, default='')

class Book(models.Model):
    library = models.ForeignKey(
        Library,
        on_delete=models.CASCADE,
        related_name='books',
    )
    author = models.ForeignKey(
        Author,
        on_delete=models.CASCADE,
        related_name='books'

    def get_page_count(self):
        return self.pages.count()

    book = models.ForeignKey(
        Book,
        on_delete=models.CASCADE,
        related_name='pages',
    )
    text = models.TextField(null=True, blank=True)
    page_number = models.IntegerField()

    def profile():
    def decorator(func):
        def handler(*args, **kwargs):
            import line_profiler
            profiler = line_profiler.LineProfiler()
            profiler.enable_by_count()
            profiler.add_function(func)
            result = func(*args, **kwargs)
            profiler.print_stats()
            return result

        handler.__doc__ = func.__doc__
        return handler

    return decorator

    # Optimization Techniques

    # Avoid queries in loops

    @profile()
def get_books_by_library_id(book_ids):
    from library_app.models import Book
    from collections import defaultdict
    result = defaultdict(list)
   
    for book_id in book_ids:
        book = Book.objects.get(id=book_id)
        result[book.library_id].append(book)
    return result

    Consider the get_books_by_library_id given above. The problem with this is we’re making a round trip to the database to fetch each book i.e., we are going to database to get book details for each book. As you already know that you need all the books, an optimal way would be request database once to get all the books As it saves time to time for round trip to the database.

    @profile()
def get_books_by_library_id_one_query(book_ids):
    from library_app.models import Book
    from collections import defaultdict
    books = Book.objects.filter(id__in=book_ids)
    result = defaultdict(list)
    
    for book in books:
        result[book.library_id].append(book)
    return result

    Boom. Now you’re only making one SQL query regardless of how many libraries exist in your database. So how do we know if this is any faster? Use the profile decorate given

    # Fetch all necessary data at once - select_related

    Imagine you wanted a list of strings in the format of Harry Potter and the Sorcerer's Stone by J.K. Rowling corresponding to each library.

    You might write something like this:

    def get_books_by_author():
    from library_app.models import Book         
    from collections import defaultdict    
    books = Book.objects.all()
    result = defaultdict(list)
    for book in books:
        author = book.author
        title_and_author = '{} by {}'.format(
            book.title,
            author.name
        )
        result[book.library_id].append(title_and_author)
    return result

    Here’s the problem: each time you access book.author you’re making a query equivalent to Author.objects.get(id=book.author_id)(Check SQL queries generated using connection.queries). If you’re looping over thousands of books you’re also making thousands of completely unnecessary queries. Use select_related to avoid this:

    def get_books_by_author_select_related():
    from library_app.models import Book
    from collections import defaultdict
    books = Book.objects.all().select_related('author')
    result = defaultdict(list)
    for book in books:
        author = book.author
        title_and_author = '{} by {}'.format(
            book.title,
            author.name
        )
        result[book.library_id].append(title_and_author)
    return result

    Note: You will learn select_related in detail in the next reference.

    select_related works by making a single more complex SQL query, which also returns the fields of the related object. So instead of just getting back data about all of the books, you’re grabbing data for each book’s author as well. Now when you access book.author you’re accessing a cached version of the author rather than making a separate database query. The function using select_related was 21x faster:

    # Fetch all necessary data at once - prefetch_related

    prefetch_related is similar to select_related but used for ManyToMany & reverse ForeignKey relationships. prefetch_related makes separate queries for each relationship and "joins" the results together in python. The downside of this approach is it requires multiple round trips to the database (so if something can be done by select_related, its better to use select_related).

    from library_app.models import Author
Author.objects.filter(name__startswith='R').prefetch('books')

    How it works:

    • first a request is fired off which runs the primary query Author.objects.filter(name__startswith=letter).
    • Then a second query equivalent to: Book.objects.filter(author_id__in=PKS_OF_AUTHORS_FROM_FIRST_REQUEST) is executed

    both of the responses are merged together into an Author queryset that has each author’s books cached in memory.

    Use prefetch_related to fetch reverse ForeignKey or ManyToMany relationship Note: You will learn prefetch_related in detail in the upcoming references.

    # Fetch only necessary fields

    The time an SQL query takes depends on the amount of data you are fetching and

    In the function below, we’re fetching all of the fields defined on the book and author even though we only need the author’s name and the book’s library_id and title.

    def get_books_by_author_select_related():
    from library_app.models import Book
    from collections import defaultdict
    books = Book.objects.all().select_related('author')
    result = defaultdict(list)
    for book in books:
        author = book.author
        title_and_author = '{} by {}'.format(
            book.title,
            author.name
        )
        result[book.library_id].append(title_and_author)
    return result

    Because of this, we incur pretty significant overhead that can be eliminated by calling .values() or .values_list() on the queryset to ask only for the fields we need:

    
def get_books_by_author_select_related_values():
    from library_app.models import Book
    from collections import defaultdict
    books = (
        Book.objects
         .all()
         .values('title', 'library_id', 'author__name')
    )
    result = defaultdict(list)
    for book in books.iterator():
        title_and_author = '{} by {}'.format(
            book['title'],
            book['author__name']
        )
        result[book['library_id']].append(title_and_author)
    
    return result

    .values() returns a list of dictionary representations of the model instance: [{'title': 'Snow Crash', 'library_id': 9, 'author__name': 'Neil'}, ...], the functions using.values() runs ~6.2x faster than the original function.

    def get_books_by_author_select_related_values_list():
    from library_app.models import Book
    from collections import defaultdict
    books = (
        Book.objects
         .all()
         .select_related('author')
         .values_list('title', 'library_id', 'author__name')
    )
    result = defaultdict(list)
    for book in books.iterator():
        title_and_author = '{} by {}'.format(
            book[0],
            book[2]
        )
        result[book[1]].append(title_and_author)
    
    return result

    .values_list() returns a list of tuples representing the model instance [('Snow Crash', 9, 'Neil'), ...].

    # Prepare model instance only if it is needed

    Even if you need all of the fields on the object, .values() and .values_list() will still provide a substantial performance boost by skipping model initialization. You can get all of the fields back by not passing in any fields as arguments.

    # returns list of model instances
def get_book_instances():
    return list(
        Book.objects
        .all()
        .select_related('author')
    )
# returns list of dictionaries representing model instances
def get_book_dictionaries():
    return list(
        Book.objects
        .all()
        .select_related('author')
        .values()
    )
# returns a list of dictionaries with the name of each book
def get_book_dictionaries_title_only():
    return list(
        Book.objects
        .all()
        .select_related('author')
        .values('title')
    )

    Getting the dictionary representation of the books is 9x faster, and asking for a particular field on the book is 15.8x faster than requesting the model instances.

    # Prefer Bulk Actions

    This one is pretty simple. If we’re going to create more than one object at a time, use bulk_create instead of creating the objects in a loop. As the name suggests, bulk_create will insert a list of objects into the database using one query, regardless of how many objects we’re inserting.

    # SQL is (generally) faster than Python

    Let’s imagine you wanted a function that returns the total number of pages for each library. Using what we’ve learned above you might end up with something like this:

    def get_page_count_by_library_id():
    from collections import defaultdict
    from library_app.models import Book        
    result = defaultdict(int)
    books = Book.objects.all().prefetch_related('pages')
    for book in books:
        result[book.library_id] += book.get_page_count()
    return result

    Even though this will only fire off two queries, we still have to pull all of the books into memory and loop over each one of them. This sort of thing can easily be turned into an annotation.

    
def get_page_count_by_library_id_using_annotation():                                                    
    from django.db.models import Sum
    from library_app.models import Library
    result = {}
    libraries = (
        Library.objects
        .all()
        .annotate(page_count=Sum('books__pages'))
        .values_list('id', 'page_count')
    )
    for library_id, page_count in libraries:
        result[library_id] = page_count
    return result

Our new function runs about 115x faster than the original. This example is pretty simple, but there’s a lot you can do with annotations. If you’re running into performance issues while doing math on a large queryset, consider writing an annotation to offload that work to the database.

    • Calculating count
    count = Book.objects.filter(author_id=5).count()

    is faster than

    count = len(Book.objects.filter(author_id=5))
    • To check if item exists
    Book.objects.filter(id=5).exists()

    is better than

    Book.objects.get(id=5)
    • Bulk Delete
    queryset.delete()

    is better than

    for item in queryset:
    item.delete()

    # Summary

    • Reduce the number of queries i.e

      • Your number of SQL queries shouldn't increase with the input/data in the database
      • Fetch all necessary data in a single query instead of multiple queries i.e use select_related & prefetch_related
      • Use bulk operations, i.e., bulk_create instead of create, update instead of save in case of operations on more than one object

    • Fetch data only that you need, use values & values_list.

    • Don't fetch Model object when you don't need the object i.e., use values & values_list.

    • Prefer operations at SQL level instead of Python level, i.e.,

      • use filter & exclude to select data you want instead of fetching all the data from database and filtering at python level.
      • use aggregate/annotate to compute data instead of fetching all the data from the database and computing aggregates at python level.
